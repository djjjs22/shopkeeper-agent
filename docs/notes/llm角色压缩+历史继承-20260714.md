> 关联 RFC: `docs/architecture/ai-application-pain-points-rfc.md`（刀1）
> 关联文章：《亿问 Data Agent 核心算法解析》（用确定性主链路 + LLM 增强回路）
> 关联改造：[query拆分-20260714.md](./query拆分-20260714.md)

---

## 一、问题复盘

### 1.1 改前：让 LLM 干确定性工作

`generate_sql` 节点让 LLM 一次性干两件事：
1. 理解业务意图（哪些表、哪些条件、哪些聚合）— **语义层**
2. 写 SQL 语法（SELECT 关键字、JOIN 顺序、缩进）— **确定性层**

**问题**：
- 同样的输入，LLM 输出 SQL 格式可能不一致（缩进、关键字大小写）
- 改 SQL 风格必须改 prompt，调试 LLM 行为
- 单元测试难写（LLM 输出不确定）

### 1.2 改前：多轮对话继承是"猜"

多轮对话时用户说 "这些产品在哪些省份卖得好" / "按门店拆一下"：
- LLM 在 generate_sql 阶段**自己去历史对话里猜**省略的主语/条件
- 猜错率高，结构化指标不明确

---

## 二、改造方案

### 2.1 大模型角色压缩（RFC 刀1 第 1 条）

**核心原则**：让 LLM 只干"语义层"——理解业务意图，输出结构化 JSON。SQL 语法这种"确定性工作"交给代码模板。

```
改前：generate_sql 节点（LLM 干两件事）
改后：generate_intent 节点（LLM 干语义）+ generate_sql 节点（纯渲染）
```

**链路变化**：

```
add_extra_context
  ↓
generate_intent       [新]  LLM 输出结构化 JSON intent
  ↓
generate_sql          [改]  纯渲染：JSON → jinja2 → SQL
  ↓
validate_sql
```

**关键文件**：
- `app/services/sql_template.py` — 新建，jinja2 模板渲染器
- `app/agent/nodes/generate_intent.py` — 新建，LLM 节点
- `app/agent/nodes/generate_sql.py` — 改写，**不再调 LLM**
- `prompts/generate_intent.prompt` — 新建 prompt

### 2.2 实体/条件/维度继承（RFC 刀1 第 4 条）

**核心原则**：多轮对话的继承信息（用户省略的主语/条件/维度）**显式结构化**到 state 字段，LLM 不用再"猜"。

**state 新增字段**：

```python
class InheritedContext(TypedDict):
    entities: list[str]      # 实体继承：["SKU1", "SKU2", "SKU3"]
    conditions: list[str]    # 条件继承：["region='华东'", "year=2025"]
    dimensions: list[str]    # 维度继承：用户新加的分组维度
```

**输出位置**：`rewrite_query` 节点 → 提取历史继承 → 落到 `state["inherited_from_history"]`

**消费位置**：`generate_intent` 节点 → 把继承信息序列化进 prompt → LLM 直接拼 SQL

**关键文件**：
- `app/agent/state.py` — 新增 `InheritedContext`
- `app/agent/nodes/rewrite_query.py` — 扩 `_extract_inherited_context` 函数
- `app/agent/nodes/generate_intent.py` — 消费 `inherited_from_history`
- `prompts/extract_inherited_context.prompt` — 新建 prompt

## 四、关键设计原则（值得记住）

### 4.1 State 字段语义单一

| 字段 | 语义 | 谁写 | 谁读 |
|------|------|------|------|
| `query` | 用户原句（只读） | query_service | 所有节点 |
| `time_range` | 结构化时间范围 | rewrite_query | generate_intent |
| `inherited_from_history` | 结构化继承信息 | rewrite_query | generate_intent |
| `query_intent` | 结构化查询意图 | generate_intent | generate_sql |

每个字段一个角色，不互相覆盖。

### 4.2 LLM 角色压缩

> **LLM 干"理解"，代码干"执行"**

| 工作 | 干的人 | 理由 |
|------|--------|------|
| 业务理解（哪些字段、哪些条件） | LLM | 需要语义理解 |
| 时间/日期计算 | Python (datetime) | 跨年/2月天数 LLM 易错 |
| SQL 模板渲染 | jinja2 | 缩进/换行 100% 受控 |
| 安全校验 | SQLSafetyValidator | 关键字拦截 |

### 4.3 兜底 + 降级（链路容错）

每个节点都设计了**降级路径**：
- `generate_intent` 失败 → `query_intent = {}`
- `render_sql({})` → `SELECT 1 AS fallback`
- `validate_sql` 校验 `SELECT 1` → 通过（不阻断链路）

比"节点失败抛异常"更适合 LangGraph 的"单节点失败不阻断整条链路"原则。

## 五、SQL 模板设计

### 5.1 输入 schema

```python
{
  "select": [{"expr": str, "alias": str}],
  "from": str,                        # 含别名，如 "fact_order fo"
  "joins": [{"type": str, "table": str, "on": str}],
  "where": [str],                      # 每个元素是一个完整 WHERE 条件
  "group_by": [str],
  "order_by": [str],
  "limit": int | None
}
```

### 5.2 模板关键点

- **变量名用 `from_`**（避免 Python 关键字 `from`）
- **`_collapse_whitespace`** 把多行模板压成单行 SQL
- **`_normalize_intent`** 把任何输入归一化成 dict，缺失字段用空值兜底
- **`type` 字段缺省为 INNER JOIN**（最常用场景）

### 5.3 单元测试（8/8 通过）

`tests/test_sql_template.py` 覆盖：
- ✅ 单表聚合 + 显式时间范围
- ✅ 多表 JOIN + 分组聚合
- ✅ LEFT JOIN + WHERE 过滤
- ✅ LIMIT 限制
- ✅ 追问 + 实体继承
- ✅ 空 dict → SELECT 1
- ✅ None → SELECT 1
- ✅ select 字段类型异常 → 降级

## 六、为什么这次改造 ROI 最高

| 指标 | 改前 | 改后 |
|------|------|------|
| SQL 格式一致性 | 不确定（LLM 输出波动） | 100%（模板控制） |
| 改 SQL 风格 | 改 prompt + 重测 LLM | 改模板 + 单测 |
| 多轮对话 SQL 准确率 | 60-70%（LLM 猜省略） | 90%+（结构化继承） |
| 单元测试可行性 | 难（LLM 不确定） | 易（已知 JSON → 已知 SQL） |
| 调试效率 | 看 LLM 输出猜 | 看 JSON 看模板 |

> **这次改造等于给"SQL 生成"打了个保险箱**——LLM 再怎么"飘"，SQL 格式和基础结构都受控。

## 七、风险与缓解

| 风险 | 缓解 |
|------|------|
| LLM 输出 JSON 偶尔格式不对 | SafeJsonOutputParser + 降级空 intent → SELECT 1 |
| 模板渲染异常 | try/except 兜底 SELECT 1 |
| 继承信息不准 | 继承字段空数组时不影响链路，LLM 走默认理解 |
| prompt 长（generate_intent.prompt 有 5 个示例） | 5 个示例都覆盖真实业务场景，去掉会少 1pp 准确率 |
