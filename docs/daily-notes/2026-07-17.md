## 部署踩坑 + 修复（同日 2026-07-17 15:08-15:18）
第一次"重启"时（用户 Ctrl+C）实际**没杀干净**：
- `netstat -ano | grep :8000` 看不出问题

启动后发现方向 1 仍**没生效**，进一步排查发现 **2 个 bug**：

### Bug 1：loguru JSON formatter 错误
**症状**：设 `LOG_FORMAT=json` 后所有日志报 `KeyError: '"ts"'`，连控制台 log 都丢了
**原因**：手写的 `_json_formatter(record)` 函数被 loguru 当成 str.format 模板用
**修复**：改用 loguru 内置 `serialize=True` 参数（一行搞定）：
```python
logger.add(sink=..., serialize=_USE_JSON)
```

### Bug 2：timed_node try/except/else 不会执行 else 分支
**症状**：所有节点都打 `failed` 日志，没有 `ok` 日志
**原因**：Python 文档原话 "no return statement was executed" → try 块里 `return` 时 **else 不执行**
**修复**：改用 `try/except + finally`：
```python
error = None
try:
    return await func(...)
except BaseException as exc:
    error = exc
    raise
finally:
    duration_ms = ...
    if error:
        log_fn.error("failed")
    else:
        log_fn.info("ok")  # ✅ 现在能执行了
```

### Bug 3（运行时发现）：rewrite_query 内部辅助函数 llm 没传参
**症状**：`name 'llm' is not defined` 错误，rewrite_query 节点内部失败但 timed_node 仍然 ok（被 except 兜底吞了）
**原因**：批量改 8 个节点时只改了顶层 `async def rewrite_query` 里的 `llm = get_llm(...)`，但内部辅助函数 `_extract_inherited_context` 也引用 `llm`，没参数传入
**修复**：辅助函数加 `llm` 参数，调用处传 `llm`：
```python
async def _extract_inherited_context(llm, query, history): ...
inherited = await _extract_inherited_context(llm, query, history)
```

**审计方法**：批量改节点后必须审计所有 `llm` 引用是否在赋值之后：
```bash
for f in app/agent/nodes/*.py; do
  grep -nE "(^|[^a-zA-Z_])llm" "$f" | grep -v "get_llm\|from app.agent.llm"
done
```

## 性能分析误判（同日 2026-07-17 16:04）
用户质疑"recall_metric/column/value 不就是并行的吗"——我去看 `app/agent/graph.py` 核实后承认错误：

**graph.py 的实际并行结构**（不是我之前以为的全串行）：
```python
# 三路召回并行
graph_builder.add_edge("extract_keywords", "recall_column")
graph_builder.add_edge("extract_keywords", "recall_value")
graph_builder.add_edge("extract_keywords", "recall_metric")
add_edge("recall_*", "merge_retrieved_info")  # 三个都汇入，等最慢的完成后才触发

# 两个 filter 并行
add_edge("merge_retrieved_info", "filter_table")
add_edge("merge_retrieved_info", "filter_metric")
add_edge("filter_*", "add_extra_context")
```

**时间戳证据**（req-2613af1c）：
- extract_keywords ok: 15:32:14.448（基线）
- recall_metric ok: 15:32:17.440 (+2992ms)
- recall_column ok: 15:32:17.824 (+3376ms)
- recall_value ok: 15:32:20.589 (+6141ms = max)
- merge_retrieved_info ok: 15:32:20.645 (~ +6197ms = max+尾巴)

**整个召回阶段时间 = max(三个) = 6141ms，不是 12501ms（求和）**。

**之前错误的优化方案**：
- 改动 ① "并行化 recall_* 节省 8 秒" ❌ 已经并行了
- 改动 ④ "recall 并行 + batch" ❌ 只是 batch_size 单查询内部优化，不改并行

**重做真实瓶颈分析**（基于已并行结构）：
- 串行关键路径：classify_intent(5273) → rewrite_query(8205) → [3 路召回 max=6138] → merge(54) → [2 filter max=2715] → ... ≈ 22.4 秒最优
- 实测 36 秒 - 22.4 秒理论值 = ~13 秒 Python+LangGraph 调度开销

**真正可省的优化**：
1. P0: classify_intent 用 weak 模型（5 选 1 cheap 够用）省 ~2.5s — 需用户拍板是否违反"只有 chat 用 weak"策略
2. P1: rewrite_query 两阶段合 1 次 LLM 省 ~3s
3. P2: 排查后端启动 warmup（jieba/模型加载），之前看到首次 cache miss 要 11s

**教训（重要）**：
- **永远先看实际 graph 结构再分析性能**，不要凭直觉说"串行→并行"！
- 时间戳分析必须看**start_time**或前一个节点 ok 时间，不是 duration 自己
- duration_ms 只是单节点耗时，不能反映依赖图拓扑
- 类似 fan-in 边（3 个 recall → merge）的图形，等待时间是 max，不是 sum

  {
    "sub_queries": [
      {"id": 0, "query": "本月销售额", "depends_on": []},
      {"id": 1, "query": "上月销售额", "depends_on": []},

- sub 0 + sub 1 并行跑（无依赖），sub 2 等 0 + 1 完成后跑
- 每个 sub_query 走完整 SQL subgraph（classify→rewrite→recall→...→sql→run）

**STAR 话术**：
- S：环比增长率类 query 准确率 78% + 延迟 18s + 漏字段率 22%
