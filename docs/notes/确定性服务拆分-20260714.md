## 2. 之前 vs 现在的对比

### 2.1 之前（已废弃）：bind_tools 方案

```
classify_intent → rewrite_query → ... → generate_sql（LLM + bind_tools）→ validate_sql
```

`generate_sql` 节点：
- 给 LLM `bind_tools([get_current_date, list_table_columns, lookup_business_metric])`
- LLM 自己选调哪个 tool
- LLM 自己写 SQL 文本
- 问题：让 LLM 干确定性工作（算日期 / 拼 SQL 关键字），违反"LLM 角色压缩"

### 2.2 现在：service 拆分

```
classify_intent → rewrite_query → ... → generate_intent（LLM 输出 JSON）→ generate_sql（jinja2 渲染）
                                                                       ↑
                                                  Python services 注入: date_resolver / schema_resolver / metric_resolver
                                                                       +
                                                  远程已有: synonyms_service / rule_service / sql_template
```

3 个 service 模块各自负责**确定性工作**：
- `date_resolver` — 算今天日期 + 把"本月/上个月/过去30天"转成 SQL BETWEEN
- `schema_resolver` — 查表字段 + examples（缓存 + 容错）
- `metric_resolver` — 查业务指标口径（精确 / 归一化 / alias 三级匹配）

LLM 在 `generate_intent` 节点只输出结构化 JSON 业务意图：
```json
{
  "select": [{"expr": "SUM(fo.order_amount)", "alias": "销售额"}],
  "from": "fact_order fo",
  "where": ["fo.date_id BETWEEN 20260701 AND 20260731"]
}
```

`sql_template.render_sql(intent)` 用 jinja2 模板把 JSON 渲染成 SQL。

## 4. service 设计要点

### 4.1 date_resolver

```python
from app.services.date_resolver import resolve_date, resolve_time_range

# 单点：拿今天 + 边界
dates = resolve_date(today=date(2026, 7, 14))
# → {"today": "2026-07-14", "last_month_start": "2026-06-01", ...}

# 整段：把 query 里的相对时间转成 SQL 片段
sql_fragments = resolve_time_range("本月各天销售额")
# → {"matched_tags": ["this_month"],
#    "where_clauses": ["BETWEEN 20260701 AND 20260731"],
#    "date_col": "fact_order.date_id"}
```

边界处理：
- 跨年 1 月 → `last_month_start` 是上一年 12 月
- 2 月天数 → `last_month_end` 用 `this_month_start - timedelta(days=1)`
- 跨月边界 → 5 种粒度都支持

### 4.2 schema_resolver

```python
from app.services.schema_resolver import SchemaResolver, list_table_columns

resolver = SchemaResolver(meta_mysql_repository=meta_repo)
cols = await list_table_columns("fact_order", resolver=resolver)
# → [{"name": "order_id", "examples": [...], ...}, ...]
```

设计要点：
- **缓存**：同一次问数重复调用只查一次仓库
- **降级**：repository 缺失 / 仓库异常 → 返回空 list，不抛错
- **examples 容错**：JSON 字符串自动转 list（防御）

### 4.3 metric_resolver

```python
from app.services.metric_resolver import MetricResolver, lookup_business_metric

resolver = MetricResolver(meta_mysql_repository=meta_repo)
metric = await lookup_business_metric("支付成功率", resolver=resolver)
# → {"name": "支付成功率",
#    "sql_expression": "COUNT(CASE WHEN payment_status='成功' THEN 1 END) * 1.0 / COUNT(*)",
#    "relevant_columns": ["fact_order.payment_status"],
#    "alias": ["成功率", "付款成功率"]}
```

匹配策略（按优先级）：
1. 精确匹配 `metric.name`
2. 归一化匹配（去空白 + 小写）
3. 匹配 `metric.alias`
4. 归一化匹配 alias
