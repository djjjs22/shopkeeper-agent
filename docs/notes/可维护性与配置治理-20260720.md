### #9 ALLOWED_TABLES 动态从 meta_db 加载

#### 为什么（硬编码白名单与"元数据驱动"冲突）

项目整体走的是"元数据驱动"路线——`meta_config.yaml` + `build_meta_knowledge` 流程的设计目的就是：**业务方加一张新表，改 yaml 配置 + 跑一次构建脚本，不用动 Python 代码**。所有表 / 字段 / 指标都落在 meta_db 里，运行时靠 repository 查出来用。

但是 `SQLSafetyValidator.ALLOWED_TABLES`（`app/core/sql_safety.py:74-80`）却是个反例：

```python
ALLOWED_TABLES: List[str] = [
    "dim_region",    # 地区维度表（省份、大区、国家）
    "dim_customer",  # 客户维度表（会员等级、性别等）
    "dim_product",   # 商品维度表（品类、品牌）
    "dim_date",      # 时间维度表（年、季度、月、日）
    "fact_order",    # 订单事实表（核心数据表）
]
```

它把 5 张表名**硬编码在 Python 源码里**，导致：

1. **加表流程被打断**：业务方按"元数据驱动"流程加了表 + 跑了 `build_meta_knowledge`，但 LLM 生成的 SQL 一旦 FROM 新表，会被这个白名单拦下来报"非白名单表"。要解掉必须改 Python + 发版。
2. **现状自相矛盾**：`meta_mysql_repository.get_all_table_infos()`（`app/repositories/mysql/meta/meta_mysql_repository.py:157`）本来就能拿到全量表清单，结果这份清单根本没人喂给校验器——meta_db 知道的"全量表"和校验器知道的"白名单表"是两套数据，长期会漂移。
3. **双写维护成本**：每加一张表要在两个地方登一遍（meta_config.yaml + ALLOWED_TABLES），违反"single source of truth"。

改动的核心思路：**让 SQL 校验器的白名单跟着 meta_db 走**，meta_db 加了什么表，校验器下次启动自动认得。

#### 原来是怎么样的

改动前 `sql_safety.py` 的白名单就是一个类变量，写死 5 张表：

```python
class SQLSafetyValidator:
    ALLOWED_TABLES: List[str] = [
        "dim_region",    # 地区维度表（省份、大区、国家）
        "dim_customer",  # 客户维度表（会员等级、性别等）
        "dim_product",   # 商品维度表（品类、品牌）
        "dim_date",      # 时间维度表（年、季度、月、日）
        "fact_order",    # 订单事实表（核心数据表）
    ]
    # ⚠️ 硬编码，加表要改代码
```

`validate()` 校验表名时只看 `cls.ALLOWED_TABLES` 这一个来源——加表必须改 Python 源码 + 走发版流程。

#### 怎么做（动态白名单 + 三级优先级）

设计分三块：

**1. 新增"动态白名单"存储 + 注入入口**

在 `SQLSafetyValidator` 上加一个类变量 `_DYNAMIC_ALLOWED_TABLES`（注意是下划线开头表示"内部使用"），默认空 set。再加两个 `@classmethod`：

- `set_dynamic_allowed_tables(table_names)` —— 给外部（lifespan）调的注入入口
- `get_effective_allowed_tables()` —— 查询当前生效的白名单（调试 / 日志用）

**2. `validate()` 内部按三级优先级解析最终白名单**

```
1. 调用方显式传 allowed_tables       → 测试 / 临时场景
2. _DYNAMIC_ALLOWED_TABLES（非空时）  → 生产路径（启动时从 meta_db 加载）
3. 类变量 ALLOWED_TABLES              → 兜底（meta_db 连不上时）
```

为什么是这三层而不是直接覆盖？因为它们解决的是不同场景的问题：

- 第 1 层给单测用：`SQLSafetyValidator.validate("SELECT * FROM users", allowed_tables=["users"])`，单测想测啥表就传啥表，不用动全局状态。
- 第 2 层给生产用：lifespan 启动时从 meta_db 加载，让加表零代码改动。
- 第 3 层给兜底用：万一 meta_db 暂时连不上（启动顺序问题、网络抖动），退回到硬编码 5 张表，服务还能跑、白名单不空。

**3. lifespan 启动时主动加载**

在 `app/api/lifespan.py` 的 lifespan 函数里加一段：meta_db 起来之后，调 `MetaMySQLRepository.get_all_table_infos()` 取全量表，注入到 `SQLSafetyValidator`。整个加载包了 `try / except`，失败就 warning 一条然后跳过——服务照常启动，校验器自动落到第 3 级兜底。

**为什么用 set 不用 list？**

注入时做了一次 `{t.lower() for t in table_names if t and isinstance(t, str)}`：

- `t.lower()` —— meta_db 里表名可能大小写不一（`Dim_Region` vs `dim_region`），统一转小写避免后续比对漏判。
- `if t and isinstance(t, str)` —— 防御性：万一某行 `name` 是 None 或非字符串（脏数据），跳过不报错。
- 用 set —— 自动去重，且后续 `set.difference` 求差集比 list 快。

##### sql_safety.py：新增字段 + 注入方法

`app/core/sql_safety.py:82-103`（紧跟在原 `ALLOWED_TABLES` 之后）：

```python
# 2026-07-20 (#9)：启动时从 meta_db 动态加载的表名白名单
# 为空时退回类变量 ALLOWED_TABLES（兜底）。
# 通过 set_dynamic_allowed_tables() 在 lifespan 启动钩子里设置。
_DYNAMIC_ALLOWED_TABLES: set[str] = set()
# ↑ 类变量：直接定义在 class 体内、所有方法之外，属于类本身。
#   所有实例共享同一份（这里类只用类方法，没有实例也照样能用）。
#   set[str] 是类型注解，表示"字符串集合"。
#   初始化为空 set()——表示"还没被 lifespan 注入过"。

@classmethod
# ↑ @classmethod 装饰器：让方法可以直接 SQLSafetyValidator.set_dynamic_allowed_tables(...)
#   调用，不需要先 validator = SQLSafetyValidator() 创建实例。
#   第一个参数 cls 代表类本身（类似 self 但指向类）。
def set_dynamic_allowed_tables(cls, table_names: list[str]) -> None:
    """启动时从 meta_db 加载表名 → 注入到 SQL 校验器

    让"加表"流程零代码改动：在 meta_config.yaml 加表 + 跑 build_meta_knowledge，
    下次启动后 SQL 校验器自动识别新表，不用改 ALLOWED_TABLES 常量。
    """
    # set comprehension：{表达式 for 变量 in 可迭代对象 if 条件}
    # 等价于：
    #   result = set()
    #   for t in table_names:
    #       if t and isinstance(t, str):
    #           result.add(t.lower())
    cls._DYNAMIC_ALLOWED_TABLES = {
        t.lower() for t in table_names if t and isinstance(t, str)
    }

@classmethod
def get_effective_allowed_tables(cls) -> list[str]:
    """返回当前生效的表名白名单（动态优先，否则兜底）

    供 lifespan 日志 / 调试时确认"实际生效的白名单是哪些"。
    """
    if cls._DYNAMIC_ALLOWED_TABLES:
        # 动态已加载 → 返回排序后的 list（排序是为了日志可读、可对比）
        return sorted(cls._DYNAMIC_ALLOWED_TABLES)
    # 动态没加载（meta_db 没起来 / 注入失败）→ 退回硬编码兜底
    return list(cls.ALLOWED_TABLES)
```

##### sql_safety.py：`validate()` 改三级优先级逻辑

`app/core/sql_safety.py:331-363`（在原"第三层：SQL 注入特征检测"之后）：

```python
# ═══════════════════════════════════════════════════════════════
# 第三层半：表名白名单校验（2026-07-20 修复：之前形同虚设）
# ═══════════════════════════════════════════════════════════════
# 从 FROM / JOIN 后提取表名（已移除字符串字面量，防止 'FROM' 之类字面量误命中），
# 大小写不敏感比对白名单；任何一个不在白名单即拦截。
#
# CTE 名（WITH temp AS (...)）作为临时表被 FROM 引用是合法的，
# 抽取后动态并入白名单，避免误杀。
#
# 优先级（2026-07-20 #9 动态加载）：
#   1. 调用方显式传 allowed_tables
#   2. 启动时从 meta_db 加载的 _DYNAMIC_ALLOWED_TABLES（如有）
#   3. 类变量 ALLOWED_TABLES 硬编码兜底
# 传空列表 [] 则跳过校验（用于单元测试或非生产环境临时关闭白名单）。
if allowed_tables is not None:
    # 第 1 级：调用方显式传了（即使是空列表），按调用方的来
    # 注意：传 [] 表示"主动关闭白名单"，下面 if effective_whitelist 会判空跳过
    base_whitelist = [t.upper() for t in allowed_tables]
elif cls._DYNAMIC_ALLOWED_TABLES:
    # 第 2 级：没传 allowed_tables，但 lifespan 已经从 meta_db 注入了动态白名单
    # 这里 _DYNAMIC_ALLOWED_TABLES 是小写 set，转大写 list 跟提取出来的表名对齐
    base_whitelist = list(cls._DYNAMIC_ALLOWED_TABLES)
else:
    # 第 3 级：都没用上 → 兜底硬编码（meta_db 没起来时）
    base_whitelist = [t.upper() for t in cls.ALLOWED_TABLES]
effective_whitelist = base_whitelist
if effective_whitelist:
    # 提取 CTE 名（WITH temp AS (...)）→ 当前 SQL 自己定义的临时表
    cte_names = cls._extract_cte_names(sql_no_comments)
    # 提取 FROM / JOIN 后面的真实表名
    referenced_tables = cls._extract_tables(sql_no_comments)
    # CTE 名视作"当前 SQL 内合法引用"，并入白名单
    # set | set 是集合并集运算，等价于 union
    allowed_set = set(effective_whitelist) | cte_names
    # set.difference：referenced_tables - allowed_set 保留"引用了但不在白名单"的表名
    # sorted(...) 让错误信息里的表名有稳定顺序（便于测试 match）
    disallowed = sorted(referenced_tables - allowed_set)
    if disallowed:
        raise ValueError(
            f"SQL 安全拦截：检测到非白名单表 {disallowed}。"
            f"只允许查询 {cls.ALLOWED_TABLES}。"
        )
```

> 补充：上面这段的"为什么大小写不敏感"——`_extract_tables` 内部把提取出来的表名 `.upper()`（见 `app/core/sql_safety.py:399`），白名单这一侧也 `.upper()`（见第 1 / 第 3 级的列表推导），所以最终两边都是大写比对。第 2 级动态白名单注入时是小写 set，比对时也是先转大写 list 再用，路径一致。

##### lifespan.py：启动加载

`app/api/lifespan.py:122-143`（在 Redis 探活协程启动之后、定时调度器启动之前）：

```python
# 2026-07-20 (#9)：从 meta_db 加载表名 → 注入 SQL 安全校验器
# 让加表流程零代码改动：在 meta_config.yaml 加表 + 跑 build_meta_knowledge 后，
# 下次启动自动识别新表，无需改 ALLOWED_TABLES 常量。
try:
    # 函数内 import：避免模块加载时强依赖（如果 meta_mysql_repository 出问题，
    # 不会拖垮整个 lifespan 模块的 import）。
    from app.core.sql_safety import SQLSafetyValidator
    from app.repositories.mysql.meta.meta_mysql_repository import (
        MetaMySQLRepository,
    )

    # async context manager：session_factory() 返回一个 async context manager，
    # 进入时 __aenter__ 创建 session，离开时 __aexit__ 关闭 session（即使中间抛错也关）
    # 这里只 borrow 一个 session 用完就归还，符合"短生命周期 session"最佳实践
    async with meta_mysql_client_manager.session_factory() as session:
        meta_repo = MetaMySQLRepository(session)
        # 查 meta_db.table_info 全量 → list[TableInfo]
        tables = await meta_repo.get_all_table_infos()
        # getattr(t, "name", "")：取 TableInfo 的 name 字段，不存在就返回 ""
        # 防御性：万一某行缺 name，不会 AttributeError 把整个启动崩了
        names = [getattr(t, "name", "") for t in tables]
        # 注入到 SQLSafetyValidator 类变量
        SQLSafetyValidator.set_dynamic_allowed_tables(names)
        _log.info(
            "[OK] 动态表名白名单已加载: %s",
            SQLSafetyValidator.get_effective_allowed_tables(),
        )
except Exception as e:
    # 兜底：meta_db 连不上 / SQL 查失败 / TableInfo 解析失败 都跳过
    # 服务照常启动，校验器自动落到第 3 级硬编码 ALLOWED_TABLES 兜底
    _log.warning(
        "[SKIP] 动态表名白名单加载失败，退回硬编码 ALLOWED_TABLES: %s", e
    )
```

为什么这段放在 Redis 探活之后、scheduler 启动之前？

- 放在所有 `_safe_init` 之后——meta_mysql_client_manager 必须先 `init()` 完，`session_factory` 才存在。
- 放在 `yield` 之前——所有启动逻辑都在 yield 前。
- 不放进 `_safe_init` / `_safe_async_init` 包装——那两个是给"manager.init / manager.start"这种纯 init 用的，这里的逻辑是"用 manager 取数据再注入到另一个类"，流程不一样，独立 try / except 更清晰。

#### 加表流程对比

| 场景 | 改前 | 改后 |
|---|---|---|
| 加一张新表 | 1. 改 `meta_config.yaml` 加表配置<br>2. 跑 `build_meta_knowledge` 重建 meta_db<br>3. **改 `sql_safety.py` 的 ALLOWED_TABLES**<br>4. 跑测试<br>5. **发版** | 1. 改 `meta_config.yaml` 加表配置<br>2. 跑 `build_meta_knowledge` 重建 meta_db<br>3. **重启服务**（lifespan 自动加载新表名） |
| 改动文件数 | yaml + py = 2 个 | yaml = 1 个 |
| 是否需要发版 | 是（改了 .py） | 否（只改配置 + 重启） |

加表这件事从"代码改动"降级成了"配置 + 重启"。

#### 验证方式

启动时观察 lifespan 日志，应该看到：

```
[OK] 动态表名白名单已加载: ['dim_customer', 'dim_date', 'dim_product', 'dim_region', 'fact_order']
```

如果 meta_db 没起来：

```
[SKIP] 动态表名白名单加载失败，退回硬编码 ALLOWED_TABLES: ...
```

### #21 配置三套统一（清理 LLM 配置认知负担）

#### 为什么（三套配置并存让人困惑）

打开 `conf/app_config.yaml`，LLM 相关配置块有**三套**：

| 配置块 | 位置 | 作用 |
|---|---|---|
| `llm:` | `conf/app_config.yaml:46-54` | 老的 single 配置（一个 model_name / api_key / base_url） |
| `llm_profiles:` | `conf/app_config.yaml:81-95` | 新的双 profile（cheap + strong，每个独立配置） |
| `node_profiles:` | `conf/app_config.yaml:101-123` | 节点名 → profile 名的映射（哪个节点用哪个 profile） |

再加上 `app/agent/llm.py:213-217` 还保留了一个模块级全局变量：

```python
try:
    llm: BaseChatModel = _registry.get("strong")
```

#### 怎么做（保留兼容 + 清理认知）

权衡过 3 个选项：

| 选项 | 做法 | 收益 | 代价 |
|---|---|---|---|
| 选项 A | 彻底删老 `llm` 配置 + 模块级 `llm` 全局 | 配置认知最干净 | 破坏"简单部署只配 yaml.llm 没配 llm_profiles"的兜底；`_init_from_config` 的 fallback 分支要重写 |

1. **风险最低**——这次提交主题是"可维护性治理"，不是"重构老配置"。任何行为变化都可能引入新 bug，跟主题不匹配。

3. **测试不动**——`test_llm_registry.py:138-147` 那条"老 import 仍可用"测试继续跑通，回归风险为零。

### #22 LLM metrics endpoint

#### 为什么（无运行时指标）

改造前的现状（grep `app/agent/llm_callbacks.py` 可以确认）：

- `LLMTimingCallback` 已经在每次 LLM 调用结束（`on_llm_end`）时记录了 `model_name / duration_ms / prompt_tokens / completion_tokens / total_tokens`，绑定到 logger 输出（`app/agent/llm_callbacks.py:139-149`）。
- 配合 `LOG_FORMAT=json`（项目用 loguru / structlog 风格的结构化日志），这些字段能被 ELK / Loki 消费，做事后查询。
- `on_llm_error` 也记了 `error_type` + `duration_ms`（`app/agent/llm_callbacks.py:181-189`）。

但**没有进程内聚合**。运营 / 开发想看这些数据时做不到：

- 总调用数（多少次 LLM 调用？）
- 总 token（账单对账——LLM 厂商账单 vs 我们自己的统计）
- 错误率（多少次失败了？某个 profile 是不是不稳？）
- 各 profile 的平均耗时（cheap 是不是真的比 strong 快？慢请求出现在哪个 profile？）

`admin_router` 在改动前只有两个端点（`app/api/routers/admin_router.py:63-168`）：

- `GET /api/admin/llm-profile` —— 看 profile 配置
- `POST /api/admin/llm-profile` —— 热切换节点到指定 profile

都只能看 / 改**配置**，看不到**运行时数据**。

实际后果：账单来了对不上号，只能去 ELK 翻日志手算 token 总数；线上错误率波动，也只能 grep 日志数行数。这些都是"明明数据已经在内存里，只差一个 endpoint"的问题。

#### 原来是怎么样的

`LLMTimingCallback` 已经做了什么（保留原样，这次没改它的核心逻辑）：

- `on_llm_start` —— 用 `time.perf_counter()` 记开始时间戳，从 `serialized["kwargs"]` 提取 model_name（`app/agent/llm_callbacks.py:75-96`）
- `on_llm_end` —— 算 `duration_ms`，从 `response.llm_output.token_usage` 或 `response.generations[0][0].message.usage_metadata` 提取 token，绑定到 logger 输出（`app/agent/llm_callbacks.py:98-149`）
- `on_llm_error` —— 同样算 `duration_ms`，记 error_type，绑定到 logger 输出（`app/agent/llm_callbacks.py:168-189`）

简言之：**单次调用粒度的数据齐全，但没"加总"**。

#### 怎么做（进程级 collector + admin endpoint）

整体设计三块：

**1. 新建 `app/agent/llm_metrics.py`：`LLMMetricsCollector` 单例**

按 profile 维度聚合，每个 profile 维护一个 `ProfileStats`：

- `calls` —— 总调用数（含错误）
- `errors` —— 错误次数
- `prompt_tokens` / `completion_tokens` —— 累计 token
- `total_duration_ms` —— 累计耗时（用来算 avg）

**2. `LLMTimingCallback` 在 hook 里调 collector**

- `on_llm_end` —— 调 `record_call(profile, prompt_tokens, completion_tokens, duration_ms)`
- `on_llm_error` —— 调 `record_error(profile, duration_ms)`
- profile 从哪来？langchain 的 `BaseCallbackHandler.on_llm_start` 接受一个 `metadata` 参数，而 `llm.py` 的 `_build_model` 在 `with_config` 时已经挂了 `metadata={"profile": profile_name}`（`app/agent/llm.py:97-99`）。callback 从这里取。

**3. `admin_router` 加两个端点**

- `GET /api/admin/metrics` —— 返回所有 profile 的 `snapshot()`
- `POST /api/admin/metrics/reset` —— 清零（调试用，比如"我想看接下来 1 小时的指标"就先 reset 再看）

**为什么用进程级内存，不接 Prometheus / Redis？**

- 单进程内存：零外部依赖，零网络开销，写代码 50 行搞定。
- 当前部署是单进程（uvicorn 单 worker），进程内聚合 = 全局聚合。
- 数据精度够用：账单对账只需要"总 token"，错误率只需要"错误数 / 总数"，都不需要时序。
- 不存历史时序：只存聚合值。要看历史趋势，去 ELK 翻日志（数据本来就在那里）。

**为什么不需要锁？**

`llm_metrics.py` 的注释里写了一句"asyncio 单线程，无需锁"。展开说：

- Python 的 asyncio 是单线程事件循环，所有协程都在同一个线程里跑。
- `LLMTimingCallback.on_llm_end` 是被 langchain 在 await 链路里同步调用的——它在协程上下文里执行，不会被打断（除非遇到 await，但 `record_call` 里没有 await）。
- `record_call` 内部就是几个 `+=` 和 `setdefault`，纯 Python 字节码，GIL + 单线程保证不会数据竞争。
- 如果以后改成多线程 / 多进程部署，这里要加 `threading.Lock` 或者换成 `collections.Counter` 之类的线程安全结构。

#### 代码改动（含完整注释）

##### 新增 `app/agent/llm_metrics.py`（完整文件）

```python
# -*- coding: utf-8 -*-
"""
LLM 调用指标聚合器（2026-07-20 #22）

LLMTimingCallback 已经把每次调用的 token/duration 写到日志，但没进程内聚合。
本模块维护一个 dict[profile, Stats]，供 admin /api/admin/metrics 查询：
- 总调用次数
- 总 token（prompt + completion）
- 错误次数
- 累计耗时（用于算 avg）

设计：
- 单进程内存计数（asyncio 单线程，无需锁）
- 不存历史时序数据，只存聚合值（粒度够用，零外部依赖）
- profile 维度聚合（cheap/strong 分别统计）
"""

from __future__ import annotations
# ↑ 启用"延迟类型注解求值"：让类型注解里的 str / dict 等可以按字符串方式求值，
#   避免循环 import 或前向引用报错。Python 3.10+ 写法。

from dataclasses import dataclass
# ↑ dataclass 装饰器：自动给类生成 __init__ / __repr__ / __eq__ 等方法。
#   对"纯数据容器"（只有字段、没什么业务逻辑的类）特别合适。


@dataclass
class ProfileStats:
    """单个 profile 的累计指标

    用 dataclass 而不是普通 class，因为：
    - 字段全是数值，没复杂行为
    - dataclass 自动生成的 __init__ 让构造写起来短：
        ProfileStats()  ← 所有字段都用默认值 0
        ProfileStats(calls=5, errors=1)  ← 只覆盖想改的字段
    - 自动生成的 __repr__ 方便调试（print 出来就是 ProfileStats(calls=5, ...)）
    """

    calls: int = 0
    # ↑ 字段类型注解 + 默认值。calls 是 int，默认 0。
    errors: int = 0
    prompt_tokens: int = 0
    completion_tokens: int = 0
    total_duration_ms: float = 0.0
    # ↑ 用 float 不用 int：duration_ms 可能是浮点（time.perf_counter 算出来的），
    #   累加多次后浮点精度比 int 更准。展示时再 round 成 1 位小数。

    def to_dict(self) -> dict:
        """转成可直接 JSON 序列化的 dict（给 admin endpoint 用）

        这里**算两个派生指标**：
        - error_rate：错误率 = errors / calls（calls=0 时返回 0.0，避免 ZeroDivisionError）
        - avg_duration_ms：平均耗时 = total_duration_ms / calls（同上）
        - total_tokens：prompt + completion，方便"总 token 数"对账

        不存历史时序 → 算不了 p95/p99（要算 p95 必须存所有数据点）。
        avg 够用：粗略判断"哪个 profile 慢"。
        """
        avg_ms = self.total_duration_ms / self.calls if self.calls else 0
        # ↑ if self.calls else 0 是 Python 三元表达式：
        #   等价于 0 if self.calls == 0 else self.total_duration_ms / self.calls
        #   防止 calls=0 时除以 0 报 ZeroDivisionError
        return {
            "calls": self.calls,
            "errors": self.errors,
            "error_rate": (self.errors / self.calls) if self.calls else 0.0,
            "prompt_tokens": self.prompt_tokens,
            "completion_tokens": self.completion_tokens,
            "total_tokens": self.prompt_tokens + self.completion_tokens,
            # ↑ total_tokens 不存字段，每次算出来——避免"prompt/completion 改了
            #   total 忘记同步"的脏数据风险。
            "avg_duration_ms": round(avg_ms, 1),
            # ↑ round(x, 1)：保留 1 位小数。avg 算出来可能是 1234.5678ms，
            #   展示给 admin 时 1234.6 够了，没必要那么精确。
        }


class LLMMetricsCollector:
    """进程级 LLM 指标收集器（单例）

    设计要点：
    - 单例：模块底部 _metrics = LLMMetricsCollector() 创建一次，全模块共享
    - 按 profile 聚合：dict[profile_name, ProfileStats]，cheap/strong 分别统计
    - 无锁：asyncio 单线程，所有 on_llm_end 回调串行调用，不会并发改 _stats
    """

    def __init__(self) -> None:
        self._stats: dict[str, ProfileStats] = {}
        # ↑ dict[str, ProfileStats]：key 是 profile 名（"cheap"/"strong"/...），
        #   value 是该 profile 的累计指标。第一次见到某 profile 时通过 setdefault
        #   自动塞一个空 ProfileStats()。

    def record_call(
        self,
        profile: str,
        prompt_tokens: int,
        completion_tokens: int,
        duration_ms: float,
    ) -> None:
        """记录一次成功的 LLM 调用

        被 LLMTimingCallback.on_llm_end 调用。每次调用累加到对应 profile 的 stats 上。
        """
        # setdefault(key, default)：key 存在就返回旧值，不存在就先塞 default 再返回。
        # 等价于：
        #   if profile not in self._stats:
        #       self._stats[profile] = ProfileStats()
        #   stats = self._stats[profile]
        # 但一行写完，更清晰。
        stats = self._stats.setdefault(profile, ProfileStats())
        stats.calls += 1
        stats.prompt_tokens += prompt_tokens
        stats.completion_tokens += completion_tokens
        stats.total_duration_ms += duration_ms

    def record_error(self, profile: str, duration_ms: float) -> None:
        """记录一次失败的 LLM 调用（也计 calls，方便算 error_rate = errors / calls）

        被 LLMTimingCallback.on_llm_error 调用。
        注意 token 不计：失败时拿不到 token 数，硬塞 0 会污染总 token。
        """
        stats = self._stats.setdefault(profile, ProfileStats())
        stats.calls += 1
        # ↑ 错误也算一次 calls，这样 error_rate = errors / calls 才有意义。
        #   如果只加 errors 不加 calls，error_rate 永远是 0/0 或 1/0。
        stats.errors += 1
        stats.total_duration_ms += duration_ms
        # ↑ 错误的耗时也记：失败前等待的时间也是真实的延迟，有助于看"超时类错误的耗时分布"

    def snapshot(self) -> dict:
        """返回所有 profile 的指标快照（深拷贝意义不大，dict + to_dict 已经够隔离）

        被 admin GET /api/admin/metrics 调用。
        """
        # 字典推导式：{key_expr: value_expr for ...}
        # 这里遍历 _stats，每个 ProfileStats 调 to_dict() 转成纯 dict。
        # 转换后返回的是新的 dict + 新的子 dict，跟 collector 内部状态解耦——
        # admin 拿到后即使改了也不影响内部统计。
        return {p: s.to_dict() for p, s in self._stats.items()}

    def reset(self) -> None:
        """清零（admin API 可选支持）

        被 admin POST /api/admin/metrics/reset 调用。
        场景：调试时"我想看接下来 1 小时的指标"——先 reset 再观察。
        注意：clear() 只清空 dict，不替换 dict 对象本身。其他持有 _stats 引用的
        地方（理论上没有，但万一）依然指向同一个 dict。
        """
        self._stats.clear()


# 模块级单例
# ↑ 模块在 Python 里只被 import 一次（首次 import 时执行所有顶层语句），
#   所以这行只执行一次，创建一个全局共享的 collector。
#   全项目通过 get_metrics_collector() 拿到的都是同一个实例。
_metrics = LLMMetricsCollector()


def get_metrics_collector() -> LLMMetricsCollector:
    """统一的访问入口（其他模块都走这里，不直接 import _metrics）

    好处：
    - 函数封装比"直接暴露变量"更灵活——以后想换实现（比如改成线程安全版本）
      只改这个函数即可，调用方无感。
    - 测试时 monkeypatch 这个函数比 patch 模块变量更稳。
    """
    return _metrics
```

##### `llm_callbacks.py` 改造

3 处改动：

**改动 1：`__init__` 加 `_profile` 字段**（`app/agent/llm_callbacks.py:52`）

```python

    # 不用 super().__init__() —— BaseCallbackHandler.__init__ 是空操作
    self._start_ts: float | None = None
    self._model_name: str = "unknown"
    self._profile: str = "unknown"  # 2026-07-20 (#22): 用于 metrics 归属
    # ↑ 新增：每个 callback 实例记住"我这次调用属于哪个 profile"。
    #   实例变量（不是类变量）——每个 LLMTimingCallback 实例绑定一次 LLM 调用，
```

实际用途：

- **账单对账**：每月初拿 LLM 厂商账单的总 token，跟 `/api/admin/metrics` 返回的 `total_tokens` 对比，看是否漂移。
- **错误率监控**：某个 profile `error_rate` 突然飙高 → 配合 `on_llm_error` 写的日志定位错误类型。
- **慢查询排查**：`avg_duration_ms` 突然涨 → 配合 ELK 按 duration_ms 倒序找具体慢调用。

#### 已知限制

这套方案的边界，写出来避免被误用：

- **单进程内存**：`_metrics` 是模块级变量，只在当前进程内有效。多 worker 部署（uvicorn --workers 4）每个进程独立计数，**聚合值不共享**。要看全局得自己加总各 worker 的 snapshot。
- **不存历史时序**：只存聚合值（calls 总数 / total_tokens 总数），不存"每分钟的指标曲线"。要看趋势必须配合 ELK / Prometheus。
- **只算 avg 不算 p95**：`avg_duration_ms` 是平均值，会被个别极端值（10s 的超时调用）拉高。算 p95 要存所有数据点 + 排序，内存开销大，当前没做。如果未来需要，建议直接接 Prometheus，而不是在内存里维护。
- **重启清零**：`_metrics` 是内存数据，进程重启后归零。如果要跨重启保留，得自己写到 Redis / DB。
- **token=0 的调用**：`on_llm_end` 提取 token 失败时会塞 0（见 `prompt_tokens or 0`），导致总 token 偏低。如果某次调用明明消耗了 token 但没被记上，对账时会少算。

## 改动之间的关联

这三条改动看似分散，其实都在回答同一个问题：**怎么让项目不再被"现状"绑住？**

- **#9（动态白名单）** 让"加表"流程不再需要改 Python 代码——把硬编码常量变成元数据驱动的运行时配置。
- **#21（配置注释清理）** 让"理解 LLM 配置"不再需要追问老员工——把"两个都能用"的歧义变成"这个是 DEPRECATED"的明确信号。
- **#22（metrics endpoint）** 让"运营决策"不再凭感觉——把"日志里能看到但没法聚合"的数据变成"一个 API 就能拿到的指标"。

三个维度——**演进成本、认知负担、运营数据**——共同提升项目的"可演进性"（evolvability）。

更深一层的共同模式是：**都把"原本藏起来的信息"显式化**。

- #9 把"5 张表硬编码在 Python 里"显式化成"白名单来自 meta_db，跟配置走"。
- #21 把"老 llm 入口是不是还活着"显式化成注释里的 DEPRECATED 标签。
- #22 把"每次调用消耗了多少 token"显式化成可查询的聚合指标。

信息藏起来 → 维护成本指数级增长（每个人都得自己重新发现一遍）；信息显式化 → 维护成本线性（读注释 / 调 API 就知道）。

另一个共同点是**都遵守"渐进式改造"**：

- #9 没删硬编码 `ALLOWED_TABLES`，保留作为兜底（meta_db 连不上时不至于服务起不来）。
- #21 没删老 `llm` 配置，只加 DEPRECATED 注释（零行为变化）。
- #22 没接 Prometheus，先用进程内内存满足 80% 场景（账单对账、错误率监控）。

每一步都把"破坏面"控制到最小——能加注释就不改代码，能加新文件就不动旧文件，能在 lifespan 加 try/except 就不重构整个启动流程。这种克制让改造可以小步快推，而不会一口气吞下一个大重构回头出 bug。
