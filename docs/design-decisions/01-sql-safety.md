# 决策 01：SQL 安全设计

> 目的：把"为什么这么设计"写进文档，面试时开卷答题。
> 范围：LLM 生成的 SQL 怎么防注入、防破坏。

---

## 背景

LLM 生成的 SQL 不可信。可能写出 `DROP TABLE`、可能拼出注入语句、可能用 `;` 堆叠多语句。
传统 ORM 拦不住——LLM 直接拼 SQL 字符串绕过了所有 ORM 防护。

这是 LLM Agent **最敏感的攻击面**——一旦让一个有写权限的 SQL 跑出去，要么炸库、要么脱库。

---

## 备选方案

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| A. 关键字黑名单 | DROP/DELETE 等关键字正则匹配 | 简单、快 | 易绕过（大小写、注释、Unicode 变体） |
| B. 解析器层校验 | 用 sqlglot / sqlparse 看 AST | 严谨、不易绕过 | 慢（50-200ms）、要学方言、调试复杂 |
| C. 白名单 + 注入特征检测 | 只允许 SELECT/WITH + 检测注入指纹 | 平衡、覆盖率高 | 维护成本 |
| D. 隔离环境 | 写操作走独立 read-only 账号 | 彻底 | 需要 DB 层配合 |

---

## 我的选择：C（白名单 + 注入特征检测）

**三层校验**（按顺序执行，任一不过就拒）：

1. **第一层：危险关键字拦截（13 个）**  
   关键字：`DROP` / `DELETE` / `UPDATE` / `ALTER` / `TRUNCATE` / `INSERT` / `CREATE` / `REPLACE` / `GRANT` / `REVOKE` / `RENAME` / `LOAD` / `IMPORT`  
   命中立即拒，快速失败。

2. **第二层：白名单入口**  
   SQL 必须以 `SELECT` 或 `WITH` 开头，否则拒。

3. **第三层：SQL 注入特征检测**  
   检测：`UNION SELECT`、OR 永真条件（`'1'='1'`）、`--` 注释绕过、块注释绕过、多语句堆叠。

**证据**：`app/core/sql_safety.py` 实现 + `tests/test_sql_safety.py` 17 个 pytest 全绿。

---

## 代价

- **性能**：每条 SQL 多走 1-2 次正则（毫秒级，可接受）。
- **误杀风险**：注释里的关键字、字段名里的关键字会被误判。17 个测试里专门有 case 验证："注释里写 DELETE"、"字段名叫 `update_time`" 必须放过。
- **维护成本**：加新业务时检查关键字是否会被误杀。
- **没法防的**：语义上错但语法上安全的 SQL（`WHERE` 条件错算错答，但不会炸库）。这一层只能靠 SQL 校验 + 业务规则校验，不归 SQL 安全管。

---

## 如果上线

按"防御深度"原则，应用层是快速失败，DB 层是兜底。两层都要有。

1. **接入 read-only DB 账号（最底层防御）**  
   - 就算应用层所有校验都被绕过，DB 账号本身写不进  
   - 这是**最后一道防线**，必须做

2. **限流 + 监控 SQL 异常模式**  
   - 同一用户高频触发黑名单 = 攻击行为  
   - 不同 query 触发的关键字集合 = 可以反推 prompt 注入尝试

3. **季度审计黑名单关键字是否够**  
   - 业务演进可能引入新风险面（新的 SQL 模式）  
   - 审计 + 更新 17 个测试用 case

4. **考虑用 sqlglot 做 AST 校验替代正则**  
   - 防绕过的终极方案（注释、大小写、Unicode 变体全都能识别）  
   - 代价：每条 SQL 多 50-200ms，要学方言版本

---

## 面试可能问

### Q1: 为什么不用 SQL 解析器（AST 校验）？

A: 当前规模用正则够了。

- **性能**：正则毫秒级，AST 解析要 50-200ms，每次都要走
- **复杂度**：sqlglot 要装、要学方言版本，调试要新工具
- **风险面**：现在 17 个 pytest 覆盖了已知攻击面，再加 AST 是双保险

生产规模上去（QPS 1000+、SQL 多样性上来）我会上 sqlglot 做 AST 校验作为第二层，正则保留作为快速失败。

### Q2: LLM 万一绕过怎么办？

A: 兜底是 DB 层，不是应用层。

- 应用层是**快速失败**（错误 SQL 立即拒，节省 LLM token 和用户等待时间）
- DB 层是**兜底**（read-only 账号，就算 SQL 写错了也写不进去）

两层都要有——应用层防 99%，DB 层防剩下 1%。

### Q3: 黑名单关键字会不会误杀？

A: 会，注释里的关键字、字段名里的关键字会被误判。

17 个测试里专门有 case 验证：

- 注释里写 `DELETE FROM ...` 必须放过（这是文档注释不是真操作）
- 字段名叫 `update_time` 必须放过（关键字在标识符里）
- 字符串字面量里的关键字必须放过（`WHERE name = 'delete me'`）

维护成本是有的——加新业务时检查关键字是否会被误杀。

### Q4: 你没生产用过，怎么知道这个设计是好的？

A: 老实说我没生产用过。

我能讲的是：

1. 这个设计覆盖了 OWASP 推荐的 SQL 注入三类防护（黑名单、白名单、注入特征）
2. 17 个 pytest 是按 SQL 注入分类写的（黑名单 8 个、白名单 1 个、注入特征 3 个、边界 5 个）
3. 三层是递减优先级：先黑名单快速失败，再白名单严格收紧，最后特征检测兜底

没做过的地方我会标出来——比如 sqlglot 我只在 demo 里用过，没在生产替换过正则。
