# Query 拆分改造：让 jieba 切原句、SQL 用结构化时间

> 日期: 2026-07-14
> 关联 RFC: `docs/architecture/ai-application-pain-points-rfc.md` (刀1 意图分类 + 刀5 上下文污染)
> 触发问题: 全链路演示时发现"抽取关键词看的是改写后的 query，含日期字符串"

---

## 一、问题复盘

### 1.1 现象

链路演示时，藤子发现：

> "抽取关键词跟上一个查询改写节点是割裂的啊，那输出"2025-12-01至2025-12-31华北销售额是多少？"有什么意义吗，抽取关键词还是看之前的原句子。"

具体链路：

```
用户输入: "上个月华北销售额是多少？"
         ↓
[rewrite_query] 覆盖 state["query"] = "2025-12-01至2025-12-31华北销售额是多少？"
         ↓
[extract_keywords] 切的是 state["query"] → 切到 "2025-12-01"、"至"、"2025-12-31"
```

### 1.2 三个具体伤害

1. **jieba 切到日期字符串**：jieba 是 TF-IDF + 词性，对 `2025-12-01` 无能为力，会切成 3 段无效 token
2. **Embedding 召回污染**：整句 `2025-12-01至2025-12-31华北销售额是多少？` 作为兜底关键词进入 Qdrant/ES，向量也认不出语义
3. **"原句兜底"反而帮倒忙**：`keywords = list(set(keywords + [query]))` 这行让原句被改写后又作为整句去检索，**整句检索的语义是混乱的**

### 1.3 根因

`rewrite_query` 节点把改写结果**字符串覆盖**到 `state["query"]`，破坏了 `query` 字段作为"用户原句"的语义。

设计原则：

> **`state["query"]` 应该是只读的输入**——它代表"用户当前的问题"，是个不变量。任何改写结果应该落到**独立字段**（如 `time_range`）。

---

## 二、改造方案

### 2.1 设计原则

- **`query` 永远是用户原句**——不被任何节点覆盖
- **改写结果结构化**——`time_range: TimeRangeState` 独立字段
- **jieba 切原句**——拿到的是"上个月华北销售额是多少？"这种自然语言
- **SQL 生成消费 time_range**——不用从 query 字符串里再解析

### 2.2 state 加 TimeRangeState

```python
class TimeRangeState(TypedDict):
    start_date: str    # "2025-12-01"，无显式时间为 ""
    end_date: str      # "2025-12-31"
    raw_expression: str  # "上一个自然月"，无显式时间为 ""
```

### 2.3 rewrite_query 输出改造

```python
# 改前
def _resolve_relative_time(text: str) -> str:
    # 把"上一个自然月"字符串替换成"2025-12-01至2025-12-31"
    text = text.replace("上一个自然月", f"{start}至{end}")
    return text

async def rewrite_query(state, runtime):
    rewritten = _resolve_relative_time(rewritten)
    return {"query": rewritten}  # ← 覆盖了原句！

# 改后
def _resolve_relative_time(text: str) -> tuple[str, TimeRangeState]:
    # 1. 算出起止日期
    # 2. 从 text 里删掉"上一个自然月"这种标准表达（避免 jieba 切到）
    # 3. 返回 (清理后文本, TimeRangeState)
    return text, TimeRangeState(start_date, end_date, raw)

async def rewrite_query(state, runtime):
    rewritten, time_range = _resolve_relative_time(rewritten)
    return {"time_range": time_range}  # ← 不再覆盖 query
```

### 2.4 extract_keywords 去掉"原句兜底"

```python
# 改前
keywords = jieba.analyse.extract_tags(query, allowPOS=allow_pos)
keywords = list(set(keywords + [query]))  # ← 整句兜底，污染召回

# 改后
keywords = jieba.analyse.extract_tags(query, allowPOS=allow_pos)
# 既然 query 永远是原句，整句兜底就是 jieba 切出来 + 原句，重复了，去掉
```

### 2.5 generate_sql 消费 time_range

新加一个 prompt 变量 `{time_range}`，序列化结构化时间：

```
查询改写结果（结构化时间范围）：
原始表达: 上一个自然月
起止日期: 2025-12-01 至 2025-12-31
```

LLM 看到这条后**直接用 BETWEEN 写死日期**，不再用 `DATE_SUB(CURRENT_DATE, ...)` 算。

`generate_sql.prompt` 新增规则：

```
9. 日期处理（重要）：
   - 如果【查询改写结果】给出了具体起止日期，必须用 BETWEEN 直接写死日期
   - 只有【查询改写结果】是"无"（用户没显式时间）时，才用 CURRENT_DATE / DATE_SUB 计算
```

## 四、影响面分析

### 4.1 不需要改的地方（state["query"] 语义保持）

所有读 `state["query"]` 的节点继续工作（拿到的就是原句）：

- `classify_intent` ✅ 原句更适合分类
- `respond_chitchat` ✅ 原句更适合闲聊
- `respond_metadata` ✅ 原句更适合元数据查询
- `recall_column` / `recall_value` / `recall_metric` ✅ 原句 Embedding 召回更准
- `filter_table` / `filter_metric` ✅ 原句过滤更准
- `correct_sql` ✅ 错误信息时打印原句更清晰
- `run_sql` ✅ 空结果警告打印原句

### 4.2 需要下游消费 time_range 的节点

- `generate_sql` ✅（本笔记改造）
- 未来：如果是 `add_extra_context` 节点想把 time_range 也注入到 context，可以扩展

### 4.3 兼容性

- `state["time_range"]` 是个**新增字段**——老调用方不传也兼容（`state.get("time_range", ...)` 兜底）
- 不删任何已有字段

## 五、预期收益

| 指标 | 改前 | 改后 |
|------|------|------|
| jieba 切到日期字符串 | 3 段无效 token | 0（query 里没日期） |
| Embedding 整句检索噪声 | 1 次噪声请求 | 0（去掉整句兜底） |
| SQL 时间条件正确性 | LLM 算易出错 | BETWEEN 写死 100% 对 |
| 跨年/2月天数 | LLM 算易错 | Python datetime 准 |

## 六、关键设计原则（值得记住）

> **State 字段应该按"语义类型"分，而不是按"修改时机"分。**
>
> 改前：`query` 字段既当"用户输入"又当"改写结果"，一个字段承担两个角色
> 改后：`query` 是只读原句，`time_range` 是改写结果，**每个字段职责单一**

这条原则适用于所有 LangGraph State 设计：**别让一个字段既表示"输入"又表示"处理后的中间结果"**。
