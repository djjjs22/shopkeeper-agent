### #7 date_resolver 跨日 bug 修复

#### 为什么（问题）

`app/services/date_resolver.py` 是把"今天 / 本月 / 上个月 / 最近 7 天"这种相对时间翻译成具体日期边界的纯 Python 服务。它内部有一个 `_today_cached()` 函数，专门返回"今天的日期"作为所有相对时间计算的锚点。

老实现给这个函数加了 `@lru_cache(maxsize=1)`。`lru_cache` 的语义是"**第一次调用后，结果永久缓存，再也不会重新执行函数体**"。

这导致一个隐蔽的生产 bug：

- 服务在 7 月 20 日 23:50 启动 → `_today_cached()` 返回 `2026-07-20`，被缓存
- 30 分钟后跨过 0 点，进入 7 月 21 日
- 用户问"今天的订单" → `_today_cached()` 仍然返回缓存的 `2026-07-20`（昨天）
- SQL 里写的是 `date_id = '20260720'` —— 用户拿到的是昨天的数据，且**完全无任何报错**

原 docstring 里写了"但本项目主要用于批处理 + 评测，跨日不重启的场景几乎不存在"，但实际上服务跑起来就是长进程，跨日是常态。

#### 原来是怎么样的

```python
# app/services/date_resolver.py（改动前）
from functools import lru_cache

@lru_cache(maxsize=1)
def _today_cached() -> date:
    """进程启动时拿一次今天日期，作为相对时间锚点

    为什么用缓存：
    - 同一次问数链路里，所有相对时间解析都基于同一锚点
    - 跨节点不会因为时分秒漂移导致"上个月"边界不一致
    - 测试场景可以 monkey-patch `_today_cached` 来固定时间

    注意：
    - 如果系统时钟变化，这个值不会自动刷新
    - 长跑的服务（N 天不重启）可能在月底跨日时仍然把"今天"算成旧值
    - 但本项目主要用于批处理 + 评测，跨日不重启的场景几乎不存在
    """
    return date.today()


def clear_cache() -> None:
    """清缓存（测试用）"""
    _today_cached.cache_clear()
```

问题在三处：
1. `@lru_cache(maxsize=1)` 让 `date.today()` 只执行一次
2. docstring 写"跨日不重启的场景几乎不存在"，**这是错误的假设**
3. 测试本来用 `clear_cache()` 刷新，但生产没人会去调

#### 怎么做（新方案）

**核心决策：直接删掉缓存，每次调 `date.today()`。**

理由：
1. `date.today()` 是 C 层实现，单次开销远小于 0.1ms，相比下游 LLM 调用（秒级）完全可以忽略
2. "同一次问数链路里基于同一锚点"这个担忧不存在 —— 同一天内 `date.today()` 返回值恒等，本来就是同一锚点
3. 测试不需要 monkey-patch 模块，改成通过 `resolve_date(today=fake_date)` 注入参数

`clear_cache` 函数保留为空操作，避免外部调用方（测试或老代码）报 `AttributeError`。这是"删除带副作用的 API"时的标准做法：**留空壳 + docstring 说明**。

#### 代码改动（含完整注释）

```python
# app/services/date_resolver.py:52

    """返回今天的日期，作为相对时间锚点

    永不刷新，服务跨过 0 点后"今天/本月/上个月"全部错位（用户问"今天的订单"，
    SQL 里写的是昨天的日期）。

    新实现：去掉 lru_cache，每次直接调 `date.today()`。
    - `date.today()` 本身是微秒级的 C 调用，单次开销远小于 0.1ms
    - 同一次问数链路里多次调用拿到的是同一锚点（同一日内 date.today() 恒等）
    - 测试仍可通过 `resolve_date(today=fake_date)` 注入假时钟，无需 patch 模块
    """

```

```python

    # today 为 None 时走 _today_cached() 拿真实今天；
```

- 跨日 bug 的回归用例：在测试里 `monkeypatch` `_today_cached` 两次返回不同值，断言两次 `resolve_date()` 结果不同（改造前会被缓存吞掉，第二次返回旧值）

### #14 prompt_loader 加 lru_cache

#### 为什么（问题）

`app/prompt/prompt_loader.py` 负责按名字读取 `prompts/*.prompt` 文件内容。一次完整的问数链路至少触发 8 次加载：

```
rewrite_query.prompt
extract_inherited_context.prompt
classify_intent.prompt
extend_keywords_unified.prompt     （#16 改造后合并版）
filter_table.prompt
filter_metric.prompt
generate_intent.prompt
generate_sql.prompt
```

每次 `Path(...).read_text(encoding="utf-8")` 都是一次磁盘 IO。单次 IO 不慢（SSD 上 ~1ms），但累积起来就是 8ms+ 的纯等待，且完全没有必要 —— **`.prompt` 文件运行期不变**，重复读同一文件纯属浪费。

#### 原来是怎么样的

```python
# app/prompt/prompt_loader.py（改动前）
from pathlib import Path


def load_prompt(name: str) -> str:
    """读取指定名称的 prompt 模板内容"""

    # app/prompt/prompt_loader.py 向上两级回到项目根目录，再进入 prompts 目录
    prompt_path = Path(__file__).parents[2] / "prompts" / f"{name}.prompt"
    return prompt_path.read_text(encoding="utf-8")
```

没有任何缓存，每次调用都走磁盘。

#### 怎么做（新方案）

**核心决策：加 `@lru_cache(maxsize=128)`。**

为什么用 `lru_cache` 而不是手动 `dict`：
1. `lru_cache` 是标准库，零依赖
2. 自带线程安全（CPython GIL + 内部锁）
3. `maxsize=128` 远超 prompt 总数（目前 10+ 个），LRU 淘汰永远不会触发，等于"永久缓存"
4. 装饰器写法对调用方零侵入，业务代码一行不动

同时暴露 `clear_prompt_cache()`：未来如果要做"admin API 改完 `.prompt` 文件立即生效"（热切换），调用这个函数清缓存即可。**目前不暴露 admin 端点**，因为暂无热切换需求；先留好接口是面向未来。

#### 代码改动（含完整注释）

```python
# app/prompt/prompt_loader.py
"""
Prompt 模板加载工具

按名称从项目根目录的 prompts 目录读取 .prompt 文件
业务节点只需要传入逻辑名称，不需要关心提示词文件的具体路径

2026-07-20 优化：加 lru_cache
================================
原实现每次调用都 `read_text()` 磁盘 IO，一次完整 data_query 至少触发 8 次
prompt 加载（rewrite_query、classify_intent、3 个 extend_keywords、filter_table、
filter_metric、generate_intent...）。这些 .prompt 文件运行期不变，缓存零风险。

如未来需要热切换 prompt（admin API 改文件立即生效），暴露 clear_prompt_cache()。
"""

from functools import lru_cache
from pathlib import Path


@lru_cache(maxsize=128)
def load_prompt(name: str) -> str:
    """读取指定名称的 prompt 模板内容

    结果按 name 缓存（.prompt 文件运行期不可变）。第一次调用读盘，
    后续调用直接返回缓存。
    """

    # app/prompt/prompt_loader.py 向上两级回到项目根目录，再进入 prompts 目录
    prompt_path = Path(__file__).parents[2] / "prompts" / f"{name}.prompt"
    return prompt_path.read_text(encoding="utf-8")


def clear_prompt_cache() -> None:
    """清空 prompt 缓存（admin 改完 .prompt 文件后调用，让下次 load 重新读盘）"""
    load_prompt.cache_clear()
```

#### 验证方式

- `tests/test_prompt_loader.py`：调两次同名 `load_prompt`，断言第二次返回值与第一次相同且 `cache_info().hits == 1`
- 改 `.prompt` 文件后调 `clear_prompt_cache()`，断言下次 `load_prompt` 返回新内容

### #15 session_store 并行写 Redis

#### 为什么（问题）

`app/services/query_service.py` 在一次问数结束后，要把本轮对话写回 `session_store`（L1 存，Redis 主、内存兜底）。写入是**两条消息**：用户原始 query + 助手回复。

老实现是两次串行 `await add_message(...)`：

```python
await add_message(session_id, "user", query)
await add_message(session_id, "assistant", result_text)
```

两次 await 意味着两次 Redis 往返（每次 `RPUSH + LTRIM + EXPIRE` 三个命令打包成 pipeline，算 1 个 RTT）。第二次写完全不依赖第一次的返回值，串行等是白白多走一个 RTT。

本项目的 SSE 接口对尾延迟敏感：用户已经看到结果了，再卡 1 个 RTT 才返回响应关闭，体感很差。

#### 原来是怎么样的

```python
# app/services/query_service.py（query 路径，改动前）
if last_result is not None:
    await add_message(session_id, "user", query)
    result_text = str(last_result)[:200]
    await add_message(session_id, "assistant", result_text)
```

```python
# query_multi_agent 路径（改动前）
if last_result is not None:
    await add_message(session_id, "user", query)
    result_text = str(last_result)[:200]
    await add_message(session_id, "assistant", result_text)
```

两个路径各有一份同样的串行写。

#### 怎么做（新方案）

**核心决策：用 `asyncio.gather` 并行跑两次 `add_message`。**

为什么安全：
1. `add_message` 内部对同一 session_id 是 `RPUSH key msg`（追加到 list 尾部），Redis 单线程模型下两条 RPUSH 是**原子且互不冲突**的
2. 不同 session_id 写不同 key，更没有冲突
3. 内存兜底路径用 `asyncio.Lock` 保护同一 session，gather 内部会顺序拿到锁，结果一致（只是不并行，但语义正确）

唯一需要注意的：**RPUSH 顺序**。`asyncio.gather` 按传入顺序返回结果，但实际执行顺序由事件循环调度决定。如果依赖"user 在 assistant 之前"的顺序，需要小心。这里**不依赖顺序**（历史读取时按 LINDEX 取最近 N 条，order by 是 Redis list 本身的写入顺序，业务上 user/assistant 谁先谁后都能正确还原对话）。如果未来要严格保序，可以在 `add_message` 内部用 pipeline 把两条命令一起发，但当前没必要。

顺带改造：`query()` 和 `query_multi_agent()` 两个路径都改，保持一致。

#### 代码改动（含完整注释）

```python
# app/services/query_service.py:131（query 路径）
async def query(self, query: str, session_id: str = "default"):
    ...
    if last_result is not None:
        # ⭐ 2026-07-20 优化：两次 add_message 并行（不同 key 写 Redis 原子，
        # 同 key 也是 RPUSH 互不冲突），消除 1 个 Redis RTT 的尾延迟
        result_text = str(last_result)[:200]  # 截断防止存太大
        # 注意：这里存的是用户原始 query，不是 enhanced_query
        # 否则历史里全是"【对话历史】...【任务类型】..."这种元数据
        await asyncio.gather(
            add_message(session_id, "user", query),
            add_message(session_id, "assistant", result_text),
        )
```

```python

    # ⭐ 2026-07-20 优化：两次 add_message 并行（同 query() 路径）

        add_message(session_id, "user", query),
```

### #12 rewrite_query 两次 LLM 并行（asyncio.gather）

#### 为什么（问题）

`app/agent/nodes/rewrite_query.py` 节点要做两件独立的事：

1. **省略补全 + 时间标准化**（主链路）：把"换成华北"补全成完整的查询，把"上个月"标准化成"上一个自然月"
2. **提取历史继承**：从历史对话里抽出用户省略的实体/条件/维度（entities / conditions / dimensions）

两者**输入完全独立**（都是 `query + history`），**输出也互不依赖**（分别写 `state['time_range']` 和 `state['inherited_from_history']`）。

老实现是串行的：先跑主链路，再跑历史继承。第二次 LLM 调用白白等了第一次的 RTT。

LLM 单次往返通常 500ms-2s，串行就意味着用户每次问数都要多等这段时间。

#### 原来是怎么样的

```python
# app/agent/nodes/rewrite_query.py（改动前）

    template=load_prompt("rewrite_query"),
    template_format="jinja2",
    input_variables=["query", "history"],
)
chain = prompt | llm | _build_strip_parser_runnable()

# LLM 负责语义层面的补全和时间标准化
rewritten = await chain.ainvoke({
    "query": query,
    "history": _format_history_for_prompt(history),
})

# 程序确定性解析：把标准化的时间表达替换成具体日期范围
rewritten, time_range = _resolve_relative_time(rewritten)

# ── 第二步：提取历史继承（2026-07-14 新增）──
inherited = await _extract_inherited_context(llm, query, history)
```

注意第二步 `_extract_inherited_context` 完全不依赖第一步的 `rewritten` —— 它的输入是原始 `query + history`，跟主链路一样。

#### 怎么做（新方案）

**核心决策：把两个 await 改成 `asyncio.gather` 并发。**

关键决策点：

1. **先创建协程，再 gather**：注意不能写成 `await chain.ainvoke(...)` 之后再 gather，那样第一个调用已经立即执行了。必须**先把两个 coroutine 对象创建出来**（创建协程对象不会执行），再统一交给 gather。
2. **用 gather 默认 fail-fast**：不传 `return_exceptions=True`。任一抛错立即向上抛，进入外层统一的 `except Exception` 兜底（改写失败不阻断链路，返回空 time_range + 空 inherited）。
3. **`_resolve_relative_time` 不能并行**：它依赖 `rewritten` 的输出，必须在 gather 之后串行执行。

### #17 metric/schema resolver 缓存改 TTLCache

#### 为什么（问题）

`app/services/metric_resolver.py` 和 `app/services/schema_resolver.py` 各有一个模块级缓存：

- `_METRIC_CACHE`: dict[str, dict | None]
- `_TABLE_COLUMNS_CACHE`: dict[str, list[dict]]

老实现用普通 `dict`，有两个问题：

1. **永不过期 + 无上限 → 内存泄漏**。`_METRIC_CACHE` 的 key 是用户输入的指标名，用户输入各种同义词、错别字（`gmv` / `GMV` / `G M V` / `成交总额` / "成交总"），每个新 key 都会写一条缓存条目，永远不释放。长期跑会慢慢吃光内存。
2. **元数据更新后不刷新**。运营在 meta_db 里改了某个指标的 `sql_expression`，缓存里仍然是旧值，要重启服务才生效。

另外 `_METRIC_CACHE` 还要区分"未命中"和"已知不存在"：
- 用户第一次问 "ROI" → 查库 → 没找到 → 返回 None
- 用户第二次问 "ROI" → 应该直接走缓存（知道没有），不再打 DB

老实现用 `if metric_name in _METRIC_CACHE` 判断，**None 也作为值写进 dict**，所以"已知不存在"也能被缓存。但这跟"未命中"难以区分（dict 里 None 跟"没有这个 key"语义上要分开），代码读起来不清晰。

为什么不用 `cachetools`：
1. 项目依赖里没有 cachetools，引入要联网装 + 改 `uv.lock`
2. 实际只需要"按 TTL 过期 + 容量上限 LRU 淘汰"这一个数据结构，30 行就够
3. 自己实现方便加项目特定的注释和约定（比如哨兵区分"未命中"）

新 `TTLCache` 的设计要点（见 `app/core/ttl_cache.py`）：

1. **基于 `OrderedDict`**：3.7+ 普通 dict 虽然有序，但**没有 `move_to_end`**，做不了 LRU。`OrderedDict.move_to_end(key)` 是 LRU 的关键操作。
2. **`time.monotonic()` 而不是 `time.time()`**：`monotonic` 单调递增，不受系统时间回拨影响（NTP 校时、夏令时切换不会让缓存全部"瞬间过期"或"永不过期"）。
3. **懒清理**：过期条目在 `get` 时才删除，不维护后台清理任务。简化实现，代价是过期条目最多多占一个 TTL 周期的内存（可接受）。
4. **`__contains__` 不刷 LRU 顺序**：`in` 操作只是判断存在性，不应该影响 LRU 顺序。
5. **哨兵 `default` 区分"未命中"和"None 值"**：调用方传 `_MISSING = object()`，判断 `cached is _MISSING` 即可。

metric / schema resolver 用哨兵改写后，"None 也算命中"这个语义就清晰了：

- `_TABLE_COLUMNS_CACHE`: `maxsize=64, ttl=3600`。表数量有限（5 张），64 富余；TTL 同 1 小时。

#### 代码改动（含完整注释）

**新增 `app/core/ttl_cache.py`**：

```python
# -*- coding: utf-8 -*-
"""
超轻量 TTL + 容量上限缓存

为什么不直接装 cachetools：
- cachetools 不在依赖里，引入要联网装 + 改 uv.lock
- 实际只需要"按 TTL 过期 + 容量上限 LRU 淘汰"这一个数据结构，30 行就够

设计：
- OrderedDict 维护访问顺序，淘汰时弹出最旧（LRU 近似）
- 每次写入记 timestamp，读取时检查是否过期
- asyncio 单线程下无需加锁
"""

from collections import OrderedDict
from time import monotonic
from typing import Any, Generic, TypeVar

K = TypeVar("K")
V = TypeVar("V")


class TTLCache(Generic[K, V]):
    """带 TTL 和最大容量的缓存

    Args:
        maxsize: 最大条目数，超出时按 LRU 淘汰最旧访问的 key
        ttl:     条目存活秒数；超过即视为未命中（懒清理）
    """

    def __init__(self, maxsize: int = 512, ttl: float = 3600.0) -> None:
        self.maxsize = maxsize
        self.ttl = ttl
        # value 存成 (实际值, 写入时的 monotonic 时间戳)，用来判断是否过期
        self._store: OrderedDict[K, tuple[V, float]] = OrderedDict()

    def get(self, key: K, default: Any = None) -> Any:
        """返回未过期的缓存值；不存在或已过期返回 default

        default 用于区分"未命中"：调用方可以传一个哨兵（如 object()），
        然后判断 `result is sentinel` 区分"未命中"和"命中了 None"。
        """
        entry = self._store.get(key)
        if entry is None:
            return default
        value, ts = entry
        # 用 monotonic 而不是 time.time：不受系统时钟回拨影响
        if (monotonic() - ts) > self.ttl:
            # 过期，懒删除（不在后台起清理任务，简化实现）
            self._store.pop(key, None)
            return default
        # LRU：访问后挪到末尾（标记为最近访问，淘汰时不会先被弹出）
        self._store.move_to_end(key)
        return value

    def set(self, key: K, value: V) -> None:
        """写入或更新条目；超出 maxsize 时弹出 LRU 最旧条目"""
        self._store[key] = (value, monotonic())
        # 新写入也挪到末尾（最近访问）
        self._store.move_to_end(key)
        # 容量上限保护：超出时弹出 last=False（最旧，即最久未访问）
        while len(self._store) > self.maxsize:
            self._store.popitem(last=False)  # 弹出最旧

    def clear(self) -> None:
        self._store.clear()

    def __len__(self) -> int:
        return len(self._store)

    def __contains__(self, key: object) -> None:
        # __contains__ 不该刷新 LRU 顺序；同时已过期的视为不在
        entry = self._store.get(key)  # type: ignore[arg-type]
        if entry is None:
            return False
        _, ts = entry
        if (monotonic() - ts) > self.ttl:
            return False
        return True


__all__ = ["TTLCache"]
```

### #18 MySQL pool 加 max_overflow + db_info 进程级缓存

#### 为什么（问题）

两个独立的性能点合并到这一条：

**1. MySQL 连接池打满导致请求卡 30s**

`app/clients/mysql_client_manager.py` 老配置：

```python
self.engine = create_async_engine(
    self._get_url(), pool_size=10, pool_pre_ping=True
)
```

`pool_size=10` 且没配 `max_overflow`，意味着连接池上限就是 10 个。当 11 个并发请求同时查库时，第 11 个请求会等 SQLAlchemy 默认的 `pool_timeout=30s`，期间整个请求卡死，前端体感是"请求超时"。

教学项目的并发不高，但一旦做并发评测（同时跑 20 个测试用例），立刻会卡住。

**2. `dw_mysql_repository.get_db_info()` 每次问数都查 `SELECT version()`**

```python

    result = await self.session.execute(text(sql))
    version = result.scalar()
    dialect = self.session.bind.dialect.name
```

`dialect` 和 `version` **启动后就不变**（除非中途换 DB，不可能），但每次问数都打一次 DB 拿同样的值。纯粹浪费一次 RTT。

#### 原来是怎么样的

#### 怎么做（新方案）

**连接池：加 `max_overflow=20` + `pool_timeout=10`。**

参数含义：
- `pool_size=10`：连接池常驻 10 个连接
- `max_overflow=20`：突发流量时允许再开 20 个，瞬时上限到 30 个。空闲后会回收回 pool_size
- `pool_timeout=10`：10 秒内拿不到连接就抛 `TimeoutError`，快速失败（不再傻等 30s）
- `pool_pre_ping=True`：拿连接前先发 ping，自动剔除 MySQL 重启后的死连接（保留原配置）

`max_overflow=20` 的权衡：MySQL `max_connections` 默认 151，这里到 30 是安全的。如果是多进程部署（比如 4 个 worker），就是 4 × 30 = 120，要注意别超过 MySQL 上限。

**db_info 缓存：模块级变量 + `global` 声明。**

第一次查询后把结果存到 `_DB_INFO_CACHE`，后续直接返回。`dialect + version` 是进程级不变量，缓存零风险。

为什么用模块级变量而不是 `lru_cache`：
1. 这是 async 函数，`lru_cache` 不能直接装饰 async 函数（会缓存 coroutine 对象，不是结果）
2. 模块级变量更直观，配合 `global` 声明修改

### #24 _recall_helpers gather 不吞 CancelledError

#### 为什么（问题）

`app/agent/nodes/_recall_helpers.py` 的 `parallel_recall_dedup` 用 `asyncio.gather(*tasks, return_exceptions=True)` 并行跑 N 个关键词的检索任务。`return_exceptions=True` 的设计意图是"单个关键词检索失败不影响其他关键词"，所以下面遍历结果时把 `Exception` 当作"跳过这个失败的关键词"。

但这个逻辑有个**严重的副作用**：`asyncio.CancelledError` 也是 `BaseException` 的子类（Python 3.8+），用 `isinstance(result_item, Exception)` 判断虽然不会匹配到它（CancelledError 继承 BaseException 不是 Exception），但循环里"任何不是 Exception 的就当成功结果处理"的逻辑会把 CancelledError 当成"成功结果"塞进 `seen` dict —— 这会让取消信号完全失效。

实际场景：
- 用户点了"停止"按钮 → 前端断开 SSE → 上游 `astream` 抛 `CancelledError`
- 本应立即停掉所有 N 个 recall 子任务，但 `parallel_recall_dedup` 把 CancelledError 吞掉当成普通成功结果
- 结果：用户已经走了，后端还在跑完所有 5 个关键词的 embedding + qdrant 检索，**白白消耗 LLM / Qdrant / ES 资源**

注意：在 Python 3.8+，`asyncio.CancelledError` 继承自 `BaseException` 而不是 `Exception`。所以 `isinstance(ce, Exception)` 实际上是 `False`。但 `gather(..., return_exceptions=True)` 会把 `CancelledError` 也作为结果项返回（不是抛出），需要在循环里显式处理。

#### 原来是怎么样的

```python
# app/agent/nodes/_recall_helpers.py（改动前）
async def parallel_recall_dedup(
    keywords: list[str],
    search_one: Callable[[str], Awaitable[list[T]]],
    dedup_key: Callable[[T], str],
    label: str,
) -> list[T]:
    ...
    tasks = [_safe_search(search_one, kw, label) for kw in keywords]
    all_results = await asyncio.gather(*tasks, return_exceptions=True)

    # 用去重 key 做唯一键，多关键词命中同一条实体只保留一份
    seen: dict[str, T] = {}
    for result_item in all_results:
        if isinstance(result_item, Exception):
            # 单个关键词检索失败不影响其他关键词的召回结果
            logger.warning(f"[{label}] 关键词检索失败（跳过）: {result_item}")
            continue
        for item in result_item:
            key = dedup_key(item)
            if key not in seen:
                seen[key] = item
    return list(seen.values())
```

`return_exceptions=True` 下，`CancelledError` 也会作为结果项出现，但它不是 `Exception` 的子类，所以走 `else` 分支被当成"成功结果"处理。下一步 `for item in result_item` 会试图迭代一个 `CancelledError` 实例，行为不可预测。

#### 怎么做（新方案）

**核心决策：在循环里显式判断 `CancelledError`，立即向上抛。**

这是 asyncio 生态的约定：**`CancelledError` 必须向上传播，不能被 `except Exception` 默默吞掉**。Python 3.8 把它从 `Exception` 改成继承 `BaseException`，就是为了避免 `try/except Exception` 误吞取消信号。

修改后逻辑：
1. 先检查 `CancelledError` → 立即 `raise`（取消信号向上传播，让 gather 知道要取消其他任务）
2. 再检查 `Exception` → 跳过这个失败的关键词（保持原有的容错语义）
3. 其余当作成功结果处理

#### 代码改动（含完整注释）

注意检查顺序很重要：`CancelledError` 检查必须在 `Exception` 之前（虽然 Python 3.8+ 它已经不是 Exception 子类，但写在前头更清晰、对未来 Python 版本变化也更鲁棒）。

### #13 Embedding 客户端 aiohttp session 复用

#### 为什么（问题）

`app/clients/embedding_client_manager.py` 的 `TEIEmbeddings` 类负责调自托管的 TEI（Text Embeddings Inference）服务做向量化。

老实现每次调用都新建一个 `aiohttp.ClientSession`：

```python
async def aembed_documents(self, texts: List[str]) -> List[List[float]]:
    async with aiohttp.ClientSession() as session:           # 每次都新建
        async with session.post(...) as resp:
            ...
```

一次问数链路里，召回阶段会调多少次 embedding？

- `extract_keywords` 不调（jieba 抽取）
- `recall_column`：jieba 关键词 + LLM 扩展词，假设 5 个关键词 → 5 次 `aembed_query`
- `recall_metric`：同样 5 次
- （value 走 ES 不用 embedding）

也就是一次问数 5-10 次 embedding 调用 = 5-10 次 `ClientSession()` 新建 + 关闭。每次都走完整的 TCP 三次握手 + TLS 握手（如果走 HTTPS），开销集中在网络 RTT 而不是 embedding 计算本身。

aiohttp 官方文档明确推荐：**不要为每次请求新建 ClientSession**，应该用一个长生命周期的 session 复用连接池。

#### 原来是怎么样的

```python
# app/clients/embedding_client_manager.py（改动前）
class TEIEmbeddings(Embeddings):
    """自定义 TEI (Text Embeddings Inference) 客户端，通过 HTTP 调用自托管 Embedding 服务"""

    def __init__(self, base_url: str):
        self.base_url = base_url.rstrip("/")
        self._embed_url = f"{self.base_url}/embed"

    async def aembed_documents(self, texts: List[str]) -> List[List[float]]:
        """异步批量向量化"""
        async with aiohttp.ClientSession() as session:      # ⚠️ 每次新建
            async with session.post(
                self._embed_url,
                json={"inputs": texts},
            ) as resp:
                resp.raise_for_status()
                return await resp.json()
```

#### 怎么做（新方案）

**核心决策：长生命周期 `aiohttp.ClientSession`，在 init 时创建，lifespan 关闭时释放。**

设计要点：

1. **惰性创建 session**：`_ensure_session()` 首次调用时才创建。这样模块导入（`embedding_client_manager = EmbeddingClientManager(...)`）不会立即建连，跟原有的"显式 `init()` 才建连"约定一致。
2. **连接池上限放宽**：`aiohttp.TCPConnector(limit=32)`，默认是 100。本项目并发不高，32 够用，避免连接数失控。
3. **`aclose()` 方法**：释放底层 session，由 `EmbeddingClientManager.close()` 转发，再由 FastAPI lifespan 在应用关闭时调用。
4. **session 关闭后自动重建**：`if self._session is None or self._session.closed` 兜底处理。aiohttp session 关闭后不能再用，这里自动重建（理论上 lifespan 不应该让它在关闭后还被调用，但写一层防御没坏处）。
5. **同步方法保留**：`embed_query` / `embed_documents` 仍然支持同步调用（封装 `asyncio.run`），兼容老代码。但注意：同步方法在 async 上下文里调用会报错（`asyncio.run` 不能嵌套），生产代码已经全部走 async 路径。

同时配套改 `lifespan.py`：应用关闭时调 `embedding_client_manager.close()`，避免 session 泄漏（"Unclosed client session" 警告）。

```python
# app/clients/embedding_client_manager.py:17
class TEIEmbeddings(Embeddings):
    """自定义 TEI (Text Embeddings Inference) 客户端，通过 HTTP 调用自托管 Embedding 服务

    2026-07-20 优化：长生命周期 aiohttp.ClientSession
    原实现每次调用都 `async with aiohttp.ClientSession()`，一次问数里 5 个关键词
    就是 10 次建连 + 10 次 TCP 握手。改后 session 在 init 时建一次，复用到 close。
    aiohttp 官方推荐：不要为每次请求新建 ClientSession。
    """

    def __init__(self, base_url: str):
        self.base_url = base_url.rstrip("/")
        self._embed_url = f"{self.base_url}/embed"
        # 长生命周期 session，由 EmbeddingClientManager 在 lifespan 里 init/close
        # Optional 是为了"模块导入时不建连"：init() 时才真正创建
        self._session: Optional[aiohttp.ClientSession] = None

    def _ensure_session(self) -> aiohttp.ClientSession:
        """惰性创建 session（首次调用时建，后续复用）

        为什么惰性而不是在 __init__ 里建：aiohttp.ClientSession 必须在
        有事件循环的上下文里创建（__init__ 在模块导入时跑，那时还没有循环）。
        """
        if self._session is None or self._session.closed:
            self._session = aiohttp.ClientSession(
                # 适度放宽连接池上限，避免高并发下排队
                # limit=32：同一时刻最多 32 个并发连接，超出会排队
                connector=aiohttp.TCPConnector(limit=32),
            )
        return self._session

    async def aclose(self):
        """释放底层 aiohttp session（lifespan 关闭时调用）

        不调会触发 "Unclosed client session" 警告，且连接池里的 socket 不会及时释放。
        """
        if self._session is not None and not self._session.closed:
            await self._session.close()
            self._session = None

    def embed_documents(self, texts: List[str]) -> List[List[float]]:
        """同步批量向量化（封装异步调用）"""
        return asyncio.run(self.aembed_documents(texts))

    def embed_query(self, text: str) -> List[List[float]]:
        """同步单条向量化（封装异步调用）"""
        return asyncio.run(self.aembed_query(text))

    async def aembed_documents(self, texts: List[str]) -> List[List[float]]:
        """异步批量向量化（一次请求处理多条文本，TEI 原生支持）"""
        # 复用长生命周期 session，不再 async with ClientSession()
        session = self._ensure_session()
        async with session.post(
            self._embed_url,
            json={"inputs": texts},
        ) as resp:
            resp.raise_for_status()
            return await resp.json()

    async def aembed_query(self, text: str) -> List[List[float]]:
        """异步单条向量化"""
        embeddings = await self.aembed_documents([text])
        return embeddings[0]
```

### #16 三路 recall 关键词扩展合并到 extract_keywords

#### 为什么（问题）

老的三路 recall 链路里，每个 recall 节点（`recall_column` / `recall_metric` / `recall_value`）都会在开头各自调一次 LLM 做关键词扩展：

```python
def _normalize_extended(raw) -> dict[str, list[str]]:
    """把 LLM 输出规范化为 {column/value/metric: list[str]}

    LLM 偶发输出非 dict / 字段不是 list / 元素非 str，统一兜底。
    这是消费 LLM JSON 输出的标准防御姿势：每一层都做类型检查。
    """
    if not isinstance(raw, dict):
        # 完全坏掉 → 三个维度都返回空
        return {"column": [], "value": [], "metric": []}
    result: dict[str, list[str]] = {}
    for key in ("column", "value", "metric"):
        v = raw.get(key, [])
        if not isinstance(v, list):
            v = []                             # 字段不是 list → 当空
        result[key] = [str(item) for item in v if item is not None]  # 元素强转 str
    return result


async def _extend_keywords_unified(query: str) -> dict[str, list[str]]:
    """一次 LLM 调用产出三维度扩展词（column / value / metric）

    LLM 失败时降级为全空 dict，下游 recall 节点仍能基于 jieba 关键词检索。
    """
    prompt = PromptTemplate(
        template=load_prompt("extend_keywords_unified"),  # 走 prompt_loader（带 lru_cache）
        template_format="jinja2",
        input_variables=["query"],
    )
    # SafeJsonOutputParser 兼容 M3/DeepSeek 的 <think>...</think> 块
    chain = prompt | get_llm("extract_keywords") | SafeJsonOutputParser()
    try:
        result = await chain.ainvoke({"query": query})
        return _normalize_extended(result)
    except Exception as exc:
        # 任何失败都降级为空 dict，不阻断链路（recall 仍能跑 jieba 关键词）
        logger.warning(f"[extend_keywords_unified] 扩展失败，降级为空: {exc}")
        return {"column": [], "value": [], "metric": []}
```

## 改动之间的关联

这 9 条改动不是孤立的，互相之间有直接影响或共同的设计主题。

### 缓存主题（#7 / #14 / #17 / #18）

四条改动都在处理"缓存"问题，但性质完全不同：

- **#7（date_resolver 删 lru_cache）**：删掉一个**有害**的缓存。`date.today()` 会变，缓存它就是埋 bug。教训：**只缓存真正不变的值**。
- **#14（prompt_loader 加 lru_cache）**：加一个**零风险**的缓存。`.prompt` 文件运行期不可变，缓存它毫无副作用。教训：**纯函数 + 不可变输入 → 放心缓存**。
- **#17（metric/schema 改 TTLCache）**：把**有上限但会过期**的缓存正规化。普通 dict 既不过期也无上限，长期跑会泄漏；TTLCache 同时解决"过期"和"上限"。教训：**用户输入做 key 的缓存必须有上限**。
- **#18（db_info 进程级变量）**：缓存**进程级不变量**。dialect + version 启动后绝对不变，连 TTL 都不需要。教训：**进程级常量用最简单的模块变量即可，不必上框架**。

判断该用哪种缓存的决策树：
```
值会变吗？
├─ 会（如 date.today()）        → 不缓存（#7）
└─ 不会
   ├─ 进程级不变量（如 db_info） → 模块变量（#18）
   ├─ 文件级不变量（如 prompt）  → lru_cache（#14）
   └─ 元数据（可能被运营改）     → TTLCache（#17）
```

### 并行化主题（#12 / #15 / #16）

三条都是"把串行 await 改成 gather 并行"，但触发点不同：

- **#12（rewrite_query 并行）**：两个**独立的 LLM 调用**并行。输入相同（query + history），输出独立。
- **#15（session_store 并行）**：两个**独立的 Redis 写**并行。写不同消息（user vs assistant）。
- **#16（关键词扩展合并）**：不是 gather，而是**结构性消除**——三个独立 LLM 调用合并成一个 LLM 调用。

#12 和 #15 是"在不改变业务语义的前提下用 gather 提速"，#16 是"重新设计调用结构减少调用次数"。后者收益更大（节省 2 次 LLM 调用 >> 节省 1 个 RTT），但也更难想到。

### #16 与 #13 的关联

#16 让 Embedding 的调用次数**减半**（关键词列表短了 → 召回的 `_search_one_keyword` 调用变少）。这让 #13（aiohttp session 复用）的相对收益**下降**——本来一次问数 10 次建连，#16 后可能变 6 次，复用的绝对节省变小。

但 #13 仍然值得做：aiohttp session 复用是**最佳实践**，零成本（除了 close 配套），跟具体业务调用量无关。

### #24 与 #12 / #15 的关联

#24 修的 `CancelledError` 吞噬 bug 在 #12 / #15 的 gather 上下文里也存在风险。区别：
- #12 / #15 用的是 `gather` **默认 fail-fast**（不传 `return_exceptions=True`），任一抛错（包括 CancelledError）会立即向上抛，不会被吞。所以这俩不需要额外处理。
- #24 用的是 `gather(..., return_exceptions=True)`（业务上要容错单个关键词失败），所以必须显式判断 CancelledError。

教训：**`return_exceptions=True` 是把双刃剑**——容错了单个失败，但也可能容错掉取消信号。用的时候必须显式处理 CancelledError。

### #7 与 #17 的对比

两个都是修"缓存相关的 bug"，但性质完全不同：

- **#7（date_resolver）**：缓存**正确性 bug**——服务跨日返回错误数据。**用户可见**，影响业务结果。优先级 P0。
- **#17（metric/schema）**：缓存**资源泄漏 bug**——内存慢慢增长，元数据更新不刷新。**用户不可见**（除非服务 OOM），影响运维。优先级 P1。

判断缓存 bug 优先级的标准：**用户能不能感知到错误结果**。

## 风险与权衡

每条改动可能的风险、为什么可接受、未来怎么演进。

### #7 date_resolver 删 lru_cache

- **风险**：每次调 `date.today()` 有微秒级开销。同一次链路里多次调用拿到不同锚点（如果跨过 0 点）。
- **为什么可接受**：微秒级开销相比下游 LLM 秒级调用可忽略。同一日内 `date.today()` 恒等，跨日本身就是 bug 修复的目标。
- **未来演进**：如果要做更复杂的时钟管理（如支持业务时区），可以改成 `resolve_date(today=Clock.now())`，Clock 是注入的依赖。

### #14 prompt_loader 加 lru_cache

- **风险**：加了缓存后，**改了 `.prompt` 文件，运行中的服务不会重新读**。需要重启或显式调 `clear_prompt_cache()`。
- **为什么可接受**：当前没有热切换 prompt 的需求，运营改 prompt 是低频操作（配合发版）。
- **未来演进**：如果要做"admin API 改 prompt 立即生效"，加一个端点：
  ```python
  @admin_router.post("/api/admin/prompts/reload")
  async def reload_prompts(...):
      clear_prompt_cache()
      return {"status": "ok"}
  ```

### #15 session_store 并行写

- **风险**：两次 RPUSH 实际执行顺序不保证（gather 并发）。极端情况下 `assistant` 消息可能比 `user` 先进 list。
- **为什么可接受**：业务读取历史时按 LINDEX 取最近 N 条，顺序由 Redis list 维护；只要两条都写进去，对话语义就正确。
- **未来演进**：如果未来要严格保序，可以改 `session_store` 暴露一个 `add_message_pair(user_msg, assistant_msg)` API，内部用 pipeline 一次发两条 RPUSH（保序 + 1 个 RTT）。

### #12 rewrite_query 并行

- **风险**：两个 LLM 调用共用同一个 `llm` 实例，如果 LLM 客户端内部状态不线程/协程安全会出问题。
- **为什么可接受**：langchain 的 `BaseChatModel` 内部用 httpx async client，原生支持并发。
- **未来演进**：如果出现限流（admin 切到弱 profile），并行会让单次问数同时占两个并发槽。可以在 gather 外面包 `asyncio.Semaphore` 控总并发，但当前不需要。

### #17 TTLCache

- **风险 1**：自研 TTLCache 不如 cachetools 成熟，可能有边界 bug。
- **风险 2**：TTL=1h 内元数据更新不生效。
- **为什么可接受**：30 行实现逻辑简单，单元测试能覆盖；元数据更新是低频运维操作，1 小时延迟可接受。
- **未来演进**：admin API 改完元数据后调 `clear_cache()` 立即失效。或者引入 cachetools（如果项目依赖管理迁移到 poetry/uv，引入成本降低）。

### #18 MySQL pool + db_info 缓存

- **风险 1**：`max_overflow=20` 在多进程部署下可能超过 MySQL `max_connections`。比如 4 worker × 30 = 120，接近默认 151 上限。
- **风险 2**：db_info 缓存了进程级变量，如果 MySQL 主备切换后 dialect 变了（极罕见），缓存不会刷新。
- **为什么可接受**：本项目单进程部署，30 远低于 151。dialect 在主备切换后不变（都是 mysql）。
- **未来演进**：多进程部署时改成按 worker 数算 `max_overflow`。db_info 缓存加一个手动刷新入口（admin API）。

### #24 不吞 CancelledError

- **风险**：之前依赖"吞掉错误继续跑"的下游代码会受影响。
- **为什么可接受**：取消信号本身就该向上传播，原来的"吞掉"是 bug 不是 feature。
- **未来演进**：在 `parallel_recall_dedup` 上游加 `asyncio.timeout(N)` 显式控制单次 recall 的总时长。

### #13 aiohttp session 复用

- **风险 1**：session 在 lifespan 关闭后被调用会报错（已用 `_ensure_session` 兜底自动重建）。
- **风险 2**：长生命周期 session 的连接池如果出现死连接（MySQL 类似的 `pool_pre_ping`），aiohttp 没有内置 ping 机制。
- **为什么可接受**：TEI 是内网服务，连接稳定性高；aiohttp 内部有重试和连接健康检查。
- **未来演进**：如果 TEI 出现间歇性死连接，可以在 `_ensure_session` 里加 `keepalive_timeout` 配置或定期重建 session。

### #16 关键词扩展合并

- **风险 1**：新 prompt `extend_keywords_unified` 要求 LLM 一次输出三个维度的 JSON，比原来三个独立 prompt 复杂，LLM 输出坏 JSON 的概率上升。一旦 LLM 输出坏 JSON，三路 recall **全部**受影响（都拿不到扩展词）。
- **风险 2**：三个维度耦合在一个 prompt 里，调一个维度的规则会影响其他维度（prompt engineering 复杂度上升）。
- **为什么可接受**：`_normalize_extended` 做了完整兜底，LLM 失败降级为空 dict，三路 recall 仍能基于 jieba 关键词跑。链路不阻断，只是召回质量下降。
- **未来演进**：如果发现 LLM 输出三个维度的稳定性不够，可以加 retry（`retry_once` 包一下 `_extend_keywords_unified`），或者退回到"两次调用：column+metric 合并、value 单独"的折中方案。
