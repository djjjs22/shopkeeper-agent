## 1. 第一刀：例化数据 JSON 标准化

本次先修复的是评测失败里反复出现的这个异常：

```text
Agent 异常: 'str' object has no attribute 'append'
```

它不是 SQL 语法本身的问题，而是 **字段元数据结构不稳定** 导致的链路崩溃。

---

## 2. 根因

### 现象

在 `merge_retrieved_info.py` 里有这段逻辑：

```python
if value not in retrieved_column_infos_map[column_id].examples:
    retrieved_column_infos_map[column_id].examples.append(value)
```

这段代码默认 `examples` 是 `list`。

但实际从 `meta.column_info` 表里查出来时，`examples` 有时候不是 list，而是 JSON 字符串：

```python
'["微信支付", "支付宝", "银行卡"]'
```

字符串没有 `.append()` 方法，所以链路直接崩：

```text
'str' object has no attribute 'append'
```

### 为什么会这样

`column_info.examples` 和 `column_info.alias` 在 MySQL 里是 JSON 类型字段。

但 SQLAlchemy / MySQL 驱动在不同查询路径下返回值不完全一致：

| 查询路径 | 可能返回 |
|---|---|
| ORM Mapper 查询 | Python list |
| text SQL + mappings 查询 | JSON 字符串 |

之前代码直接：

```python
ColumnInfo(**dict(row))
```

没有把 JSON 字符串转回 list。

## 4. 核心修复代码

新增了统一标准化函数：

```python
def _normalize_column_info_row(row: dict) -> ColumnInfo:
    """把 column_info 查询结果标准化为 ColumnInfo 实体。

    背景：
    - column_info.examples 和 column_info.alias 在 MySQL 里是 JSON 类型字段。
    - SQLAlchemy 从不同驱动/不同查询路径取出来时，可能已经是 list，
      也可能还是 JSON 字符串，例如 '["微信支付", "支付宝"]'。
    - 下游 merge_retrieved_info 节点会执行 examples.append(value)。
      如果 examples 还是 str，就会触发：'str' object has no attribute 'append'。

    处理策略：
    - 每次从 column_info 表批量查询字段元数据时，统一把 examples/alias 转回 list。
    - 如果 JSON 内容异常，降级为空 list，避免整条问数链路因为单个脏字段中断。
    - 只在 Repository 出口做一次标准化，保证后续 Agent 节点拿到的数据结构稳定。
    """

    # mappings() 返回的是 RowMapping，不是真正的 dict；先转成普通 dict，便于修改字段值。
    data = dict(row)

    # examples: 字段真实样例值；alias: 字段同义词/口语表达。
    # 这两个字段在 Python 业务实体里都应该是 list，而不是 JSON 字符串。
    for key in ("examples", "alias"):
        value = data.get(key)

        # 只有字符串才需要 json.loads；如果数据库驱动已经返回 list，就保持原样。
        if isinstance(value, str):
            try:
                data[key] = json.loads(value)
            except json.JSONDecodeError:
                # 防御脏数据：字段 JSON 异常时不要让 Agent 崩溃，给空列表继续走。
                data[key] = []

    # 统一返回 ColumnInfo 实体，屏蔽数据库存储格式差异。
    return ColumnInfo(**data)
```

并把所有从 `column_info` 表用 `text SQL + mappings()` 查出来的地方，从：

```python
return [ColumnInfo(**dict(row)) for row in result.mappings().fetchall()]
```

改成：

```python
return [_normalize_column_info_row(row) for row in result.mappings().fetchall()]
```

覆盖的方法包括：

| 方法 | 用途 |
|---|---|
| `get_key_columns_by_table_id` | 查询单表主外键字段 |
| `get_column_infos_by_ids` | 按字段 ID 批量补齐字段 |
| `get_key_columns_by_table_ids` | 多表批量补齐主外键字段 |
| `get_columns_by_table_id` | 元数据查询场景按表查字段 |

## 5. 第二刀：空 SQL / 非 SELECT 拦截 + 修正计数

### 5.1 为什么需要拦截

旧链路：

```text
generate_sql → validate_sql (拿空 SQL 调 EXPLAIN) → correct_sql (拿空 SQL 让 LLM 修) → run_sql → 报错
```

空 SQL / 模型拒答文本一旦进入链路，反复烧 token 也没机会修复。

### 5.2 改动点 1：state 加计数器

`app/agent/state.py`：

```python
# SQL 修正次数，用来限制 validate_sql -> correct_sql -> validate_sql 的闭环次数。
# 为什么要放进 State：LangGraph 每个节点都是无状态函数，跨节点计数必须写回共享 state。
# 如果不限制次数，空 SQL / 非 SQL 文本 / 无法修复的 SQL 可能反复进入 correct_sql，造成死循环。
correction_attempts: int
```

### 5.3 改动点 2：validate_sql 前置拦截

`app/agent/nodes/validate_sql.py`：

```python
_EMPTY_SQL_ERROR = "EMPTY_SQL: LLM 未生成可校验的 SQL"
_NON_SELECT_SQL_ERROR = "NON_SELECT_SQL: LLM 输出不是 SELECT/WITH 查询"


def _clean_candidate_sql(raw_sql: str | None) -> str:
    """清理 LLM 生成的候选 SQL，返回可用于校验的纯 SQL 文本。

    为什么 validate_sql 也要清理：
    - generate_sql/correct_sql 已经用了 StripThinkStrParser，但链路稳定性不能只依赖上游。
    - 有些模型会返回 ```sql ... ``` 代码块，或者输出  THINK 块后残留空白。
    - validate_sql 是进入数据库前的统一入口，在这里再做一次轻量清理更稳。
    """
    sql = raw_sql or ""
    sql = strip_think_for_str(sql)
    sql = re.sub(r"^\`\`\`\s*\n?", "", sql.strip(), flags=re.IGNORECASE)
    sql = re.sub(r"\n?\`\`\`\s*$", "", sql.strip())
    return sql.strip()


def _is_readonly_query(sql: str) -> bool:
    """判断候选 SQL 是否是只读查询。

    这里只做最小白名单判断，不在 validate_sql 重复完整 SQLSafetyValidator：
    - SELECT：普通查询。
    - WITH：CTE 查询，后面仍应是 SELECT。

    这个判断的目标不是替代安全校验，而是提前识别“解释文字/空话/拒答文本”，
    避免 correct_sql 和 run_sql 拿到完全不像 SQL 的内容。
    """
    return sql.upper().startswith(("SELECT", "WITH"))


# 节点内部：
raw_sql = state.get("sql", "")
sql = _clean_candidate_sql(raw_sql)

# 第一层防御：空 SQL
if not sql:
    return {"sql": "", "error": _EMPTY_SQL_ERROR}

# 第二层防御：非查询文本
if not _is_readonly_query(sql):
    return {"sql": sql, "error": _NON_SELECT_SQL_ERROR}
```

## 6. 第三刀：图闭环 + 最大次数保护

### 6.1 旧链路

```text
generate_sql → validate_sql
              ├ error is None → run_sql
              └ error not None → correct_sql → run_sql
```

问题：

- correct_sql 修正后不再复验，错 SQL 直接送到 run_sql。
- 空 SQL / 模型拒答会反复触发 correct_sql，造成死循环。

### 6.2 新链路

```text
generate_sql → validate_sql
              ├ error is None                       → run_sql
              └ attempts < MAX (默认 2)              → correct_sql → validate_sql (复验)
              └ attempts >= MAX                     → run_sql (强制终止)
```

`app/agent/graph.py`：

```python
GRAPH_MAX_CORRECTION_ATTEMPTS = 2


def _route_after_validate_sql(state):
    error = state.get("error")
    if error is None:
        return "run_sql"
    attempts = int(state.get("correction_attempts") or 0)
    if attempts >= GRAPH_MAX_CORRECTION_ATTEMPTS:
        return "run_sql"
    return "correct_sql"


graph_builder.add_conditional_edges(
    source="validate_sql",
    path=_route_after_validate_sql,
    path_map={"run_sql": "run_sql", "correct_sql": "correct_sql"},
)
graph_builder.add_edge("correct_sql", "validate_sql")
graph_builder.add_edge("run_sql", END)
```

### 6.3 为什么用 MAX=2 而不是单次

- LLM 第一次修正就命中的概率不低，但不能假设每次都能命中。
- 给 MAX=2 ≈ 2 轮 attempt，绝大多数可修 SQL 在 2 轮内收敛。
- 同时保证最坏情况下 validate_sql 也不会被无限调用。

## 9. 第一刀为什么是这个修复点

有两个修法选择：

| 修法 | 问题 |
|---|---|
| 在 `merge_retrieved_info.py` 里判断 `examples` 是不是 str | 属于下游补丁，每个使用方都要防一遍 |
| 在 Repository 查询出口统一转类型 | 数据源头治理，所有下游节点都拿稳定结构 |

本次采用第二种。

原因：

1. `examples/alias` 本来在业务实体里就应该是 list。
2. Repository 是数据库到业务实体的边界，最适合做类型标准化。
3. 下游 Agent 节点不应该关心数据库 JSON 字段到底怎么返回。

## 11. 当前结论

本次三刀修复后，整条问数链路的关键稳定性问题都已经补齐：

1. **元数据 JSON 不稳定** → 用 `_normalize_column_info_row` 在 Repository 出口统一转 list。
2. **空 SQL / 模型拒答文本** → `validate_sql` 前置 sentinel（EMPTY_SQL / NON_SELECT_SQL）。
3. **修正死循环** → `correct_sql` 自增 attempts + `graph.py` 加 MAX_CORRECTION_ATTEMPTS=2。
4. **修正后不再复验** → 拓扑改为 `correct_sql → validate_sql` 闭环。

未跑 50 条端到端评测，按要求只跑了 4 个原图集成测试 + 4 个小样本冒烟（< 30 秒，覆盖了所有改动点的失败路径）。下一步如果要继续推进准确率，瓶颈已经不是稳定性，而是 prompt/参数质量，需要完整评测驱动对比。
