### 几个易混点辨析

**`state.get(k)` vs `state[k]` 的真实区别**

```python
state = {"query": "销售额"}  # 假装 state 里没有 intent 字段

# state["intent"]    # → 抛 KeyError: 'intent'，节点直接崩
# state.get("intent")           # → 返回 None，节点能继续跑（但下游可能拿到 None 出问题）
# state.get("intent", "data_query")  # → 返回 "data_query"，节点能继续跑且语义合理
```

策略文档里约定下游用 `state.get(k, default)`，因为 A 类节点（如 `classify_intent`）失败时会返回降级默认值（如 `"data_query"`），下游用 `.get` 拿到的是降级值而不是 KeyError。B 类节点失败时 raise，下游根本不会执行，所以对 B 类字段的 `.get` 是冗余但无害的防御。

**`raise` vs `return {"error": ...}`**

```python
# 风格 1：raise（B 类节点）
except Exception as e:
    logger.error(...)
    writer({"type": "error", ...})
    raise  # 让 LangGraph 框架捕获，上层 query_service 包装成 SSE error

# 风格 2：return error patch（A 类或混合类的可降级路径）
except ValueError as safety_err:
    writer({"type": "error", ...})
    return  # 不 raise，流程正常结束（state 不变）
```

区别：

- `raise` → 框架捕获 → **整条 graph 停止**，后续节点不跑，`astream` 流被框架终止
- `return` → 节点正常返回 → **graph 继续走下一步**（按边的定义），state 里没有新数据

`run_sql` 同时用两种：安全拦截用 `return`（链路应该正常结束，不抛异常），DB 执行失败用 `raise`（链路必须停，因为后续没有合理动作）。

**`try / except / finally` 三件套**

本次 #6 用了 `try / except / else`，没用到 `finally`。补一句 `finally` 的语义供对照：

```python
try:
    # 跑业务
except SomeError:
    # 只在抛 SomeError 时跑
else:
    # 只在没抛异常时跑
finally:
    # 不管抛没抛都跑（常用于资源释放：关连接、释放锁）
```

`finally` 在本次改动里没用到，因为 LangGraph 节点通常是无状态的（资源通过 `runtime.context` 注入，节点不负责释放）。

### #6 generate_intent 失败 warning 提示（消除沉默失败）

#### 为什么（用户感知问题）

`generate_intent` 是把用户问题转成结构化 JSON intent 的节点。它的输出会被下游 `generate_sql` 直接拿来渲染 SQL。

链路里有这么一条**沉默的失败路径**：

1. `generate_intent` 调 LLM 产出 JSON，再过 Pydantic 强校验
2. 如果 LLM 输出格式坏 / Pydantic 校验不过，retry 1 次仍失败
3. 节点降级为空 intent `{}`
4. 下游 `generate_sql` 看到空 intent，渲染出兜底 SQL `SELECT 1 AS fallback`（见 `app/agent/nodes/generate_sql.py:70`）
5. `run_sql` 执行 `SELECT 1 AS fallback`，返回 `[{"fallback": 1}]`
6. 前端 `summarizeResult(data)`（`frontend/src/lib/format.ts:16`）看到数组长度 1，显示：

   > 查询完成，共 1 行结果。

**用户的感知**：进度条全绿，结果"完成"，但拿到的根本不是他问的数据——他根本不知道 agent 其实没听懂他的问题。

这是典型的"错误降级被吞掉"。降级本身没错（链路不能因为一个节点挂了就崩），但**降级时一定要告诉用户发生了降级**，否则用户会按"查询完成"去解读一个无意义的结果，损害对系统的信任。

修复思路：在降级发生的那一刻，立刻往 `writer` 推一个 `type=warning` 事件，让前端把它渲染成黄色提示条，明确告诉用户"未能理解查询意图"。

#### 原来是怎么样的

改动前 `generate_intent.py` 的 except 块（`app/agent/nodes/generate_intent.py:181-184`）：

```python
        except Exception as e:
            # 解析失败（含 retry 后仍失败）：降级空 dict，让下游 generate_sql 用 SELECT 1 兜底
            logger.error(f"{step}: 解析失败（已 retry 1 次），降级为空 intent: {e}")
            intent = {}
            # ⚠️ 改动前：这里没有任何 writer 调用
            #    后端只往 logger 写了一行 error（运维看得到，用户看不到）
            #    前端拿到的依然是 progress=success + result=[{"fallback": 1}]
            #    所以 UI 上一片绿，仿佛查询成功完成
```

问题就在 `logger.error` 和 `writer` 的受众不同：

- `logger.error` 写到后端日志文件，运维 / 开发看得到
- `writer({...})` 推到 SSE 流，**用户**才看得到

只 `logger.error` 不 `writer`，等于把失败对用户彻底隐藏。

#### 怎么做（显式推 warning）

三处配合：

1. **后端节点**：在降级点（except 块内）立刻推 `{"type": "warning", "step": step, "message": "..."}`
2. **前端类型**：`agent.ts` 扩一个 `WarningEvent` 类型，把它加入 `AgentEvent` 联合；`ChatMessage` 加一个 `warnings?: string[]` 字段（累积多条）
3. **前端 App.tsx**：`onEvent` 里加 `if (event.type === "warning")` 分支，把消息推到 `message.warnings`
4. **前端 MessageBubble**：渲染 `message.warnings` 为黄色提示条

#### 代码改动（含完整注释）

##### 后端 generate_intent.py

改动后的 except 块（`app/agent/nodes/generate_intent.py:181-194`）：

```python
        except Exception as e:
            # 解析失败（含 retry 后仍失败）：降级空 dict，让下游 generate_sql 用 SELECT 1 兜底
            logger.error(f"{step}: 解析失败（已 retry 1 次），降级为空 intent: {e}")
            intent = {}
            # 2026-07-20 (#6)：失败时显式推 warning 给前端，避免用户看到 SELECT 1 兜底
            # 结果却误以为"查询完成"。前端把 warning 单独渲染（黄色提示条）。
            #
            # 为什么用 type=warning 而不是 type=error？
            #   - error 会让前端把消息状态切成 "error" 并终止流程（红色错误条）
            #   - warning 只是在结果旁边加一条提示，不影响主流程的"完成态"
            #   - 这里 intent 已经降级为 {}，链路能继续跑（generate_sql 会兜底 SELECT 1），
            #     只是结果对用户没意义——属于"提醒"而不是"报错"
            writer({
                "type": "warning",
                "step": step,  # "生成查询意图"，让前端能在 StepRail 上定位到这一步
                "message": (
                    "未能理解查询意图，将返回占位结果。请换一种更具体的问法，"
                    "例如明确时间范围、字段名或筛选条件。"
                ),
            })
        else:
            # try 块没有抛异常时执行的分支：Pydantic 强校验通过，
            # model_dump(by_alias=True) 把 Pydantic 模型转回 dict，
            # by_alias=True 是为了用 schema 里的 alias（from 字段名）做 key
            intent = intent_obj.model_dump(by_alias=True, exclude_none=False)
```

注意 `try / except / else` 结构的语义：

- `try` 块跑 LLM 解析（含 retry_once 包装）
- `except` 块只在抛异常时跑（降级 + 推 warning）
- `else` 块只在没抛异常时跑（正常路径，model_dump）

把"成功路径"和"失败兜底"用 `else` 在语法上分开，比把成功逻辑也塞在 try 里更清晰——读者一眼看出"这块代码只在失败时跑"。

##### 前端类型扩展（agent.ts）

改动在 `frontend/src/types/agent.ts:26-54`：

```typescript
// 2026-07-20 (#6)：warning 事件（如意图解析失败、结果截断）
// 与 error 区别：warning 不终止流程，只是提示用户结果可能不完整
export type WarningEvent = {
  type: "warning";
  message: string;   // 后端推过来的中文文案
  step?: string;     // 可选：标记是哪个节点发的（如"生成查询意图"）
};

// AgentEvent 是所有 SSE 事件的联合类型
// 加上 WarningEvent 后，onEvent 收到的事件可能是 4 种之一

  | ProgressEvent   // 进度（running/success/error）
  | ResultEvent     // 最终查询结果
  | ErrorEvent      // 硬错误（终止流程）
  | WarningEvent;   // 软提醒（不终止流程）

// ChatMessage 是 UI 上一条完整消息（用户提问 + 助手回答）的状态
export type ChatMessage = {
  // ... 其他字段省略 ...
  result?: unknown;
  truncated?: boolean;  // 2026-07-20 (#3)：结果集是否被截断
  warnings?: string[];  // 2026-07-20 (#6)：累积的 warning 提示
  error?: string;
};
```

**为什么 `warnings` 是数组而不是单个字符串？**

因为一次查询里可能有多个节点同时降级——比如 `generate_intent` 降级（推一条 warning）+ `run_sql` 检测到结果集超过 5000 行被截断（又推一条 warning）。如果只存单条，后面的会覆盖前面的，用户只看得到最后一条。用数组就能把链路里所有"软提示"都累积下来一起展示。

##### 前端 App.tsx 处理 warning

改动在 `frontend/src/App.tsx:172-182`：

```typescript
          if (event.type === "warning") {
            // 2026-07-20 (#6)：warning 累加到 message.warnings，不终止流程
            //
            // 为什么用累加而不是覆盖？
            //   一次查询可能有多条 warning（意图降级 + 结果截断），都要给用户看到
            //
            // 为什么去重？
            //   后端理论上不会推重复 message，但前端做一层 includes 判断兜底，
            //   防止极端情况下 UI 出现两条一模一样的黄色条
            const existing = message.warnings ?? [];
            return {
              ...message,
              warnings: existing.includes(event.message)
                ? existing
                : [...existing, event.message],
            };
          }
```

注意这里返回的是"新的 message 对象"（React 不可变更新模式）：用 `...message` 展开旧字段，再覆盖 `warnings`。**不能直接 `message.warnings.push(...)`**，否则 React 检测不到引用变化，UI 不会重渲染。

`??` 是空值合并运算符：`a ?? b` 等价于"`a` 是 null 或 undefined 时取 b，否则取 a"。这里 `message.warnings ?? []` 处理第一次收到 warning 时 `warnings` 字段还不存在的情况。

### #8 错误处理策略文档 + run_sql 重复代码合并

这条改动其实是两件事打包：**写一份约定文档** + **顺手修一处明显的重复**。一起做的原因是——定约定的目的就是让"约定之外的反例"越来越少，所以一边写文档一边清理违规代码。

#### 为什么（错误处理三套混用 + 重复代码）

**问题 A：节点错误处理三套风格混用**

改前，项目里节点的 `except` 块风格不统一，大致三种：

- **A 类（可降级的 LLM 输出节点）**：fallback 默认值 + 不 raise。例如 `generate_intent` 失败降级 `{}`、`classify_intent` 失败降级 `"data_query"`、`filter_table` 失败保留所有候选。
- **B 类（硬依赖 IO 的节点）**：raise 让框架处理。例如 `run_sql`、`recall_column` 等——失败就是失败，没有合理兜底。
- **混合类（`run_sql`）**：内层 `return`（安全拦截时）+ 外层 `raise`（DB 执行失败时），用嵌套 try 区分。

这三套本身**都合理**——不同节点对失败的容忍度不同，处理方式本就该不同。问题是没有写下来的约定：**新节点的作者不知道该选哪一类**，要么 copy 错样板、要么干脆每处都 `raise`（让本可降级的链路崩掉）、要么每处都 `return error`（让本该停的链路继续跑脏数据）。

**问题 B：run_sql 重复的 Step 5 块**

##### 写一份策略文档（`docs/notes/error-handling-policy.md`）

完整文档见 `D:\shopkeeper-agent\docs\notes\error-handling-policy.md`。核心内容概括为三块约定：

1. **节点分类**（按"是否可降级"）：

   - **A 类**：LLM 输出节点（建议性产出，有降级默认值）
     - 典型：`classify_intent` / `rewrite_query` / `filter_table` / `filter_metric` / `generate_intent` / `correct_sql` / `planner` / `aggregator` / `reviewer`
     - 策略：`logger.error` + 返回降级默认值 + **必须推 `writer({"type": "warning", ...})`**（这条就是 #6 在做的事）+ 不 raise

   - **B 类**：硬依赖 IO/数据节点（事实性产出，失败就是失败）
     - 典型：`run_sql` / `extract_keywords` / `merge_retrieved_info` / `recall_column` / `recall_metric` / `recall_value` / `validate_sql` / `add_extra_context` / `respond_chitchat` / `respond_metadata`
     - 策略：`logger.error` + `writer({"type": "progress", "status": "error"})` + **raise** 让框架统一处理 + message 走 `_safe_error_message` 脱敏

   - **混合类（`run_sql`）**：内层 try 用 A 类风格（return + 推 error event）、外层 try 用 B 类（raise）。两种失败用嵌套 try 分开。

2. **字段读取约定**：下游节点取上游字段必须用 `state.get(k, default)` 而不是 `state[k]`，因为 A 类节点失败时返回降级值，下游要能拿到合理默认；B 类节点失败时 raise，下游根本不会执行——但 `.get` 对两种情况都安全。

3. **前端展示约定**：

   | 事件类型 | UI 处理 | 触发场景 |
   |---|---|---|
   | `progress` (success/error/running) | StepRail 更新 | 每个节点的状态变化 |
   | `warning` | 黄色提示条 | A 类降级、空结果、结果截断 |
   | `result` | ResultTable 渲染 | `run_sql` 成功 |
   | `error` | 红色错误条 + toast | B 类 raise（脱敏后文案）|

### #10 DataAgentState 拆 MultiAgentState

#### 为什么（state 字段污染）

2026-07-17 multi-agent 改造（见 `docs/notes/multi-agent-改造-20260717.md`）时，为了少改文件，直接把 7 个 multi-agent 专属字段塞进了 `DataAgentState`：

- `plan: Optional[QueryPlan]` — Planner 输出的执行计划
- `sub_results: list` — 每个 sub_query 跑完的结果
- `final_response: dict` — Aggregator 合并后的最终回复
- `confidence: float` — Reviewer 打分 0-1
- `review_action: Optional[str]` — "retry" / None
- `review_loop_count: int` — 反思轮数
- `cached_pre_state: Optional[dict]` — 前置 subgraph 结果缓存

注释里写了"这些字段在现有 single-agent 链路里都是 None，不影响老代码"——**功能上**确实不影响。但带来三个隐性成本：

1. **认知负担**：single-agent 13 节点链路（`graph.py`）里这 7 个字段**永远是 None**。新人读 `graph.py` 时，看到 `DataAgentState` 里有 `plan` / `sub_results` / `final_response` 这些字段，会困惑"这些是哪一步写的？我读漏了哪个节点？"——其实根本没人写。

2. **误用风险**：所有节点函数签名都 import 同一个 `DataAgentState` 类型，每个节点都能在 IDE 自动补全里看到这些字段。理论上 single-agent 节点写 `state["plan"]` 也能编译过——只是永远拿到 None，相当于把 bug 埋到运行时才暴露。

3. **类型信息丢失**：multi-agent 节点（planner / aggregator / reviewer）和 single-agent 节点共用一个 state 类型，从函数签名上分不出"这个节点是不是 multi-agent 链路专属的"。

#### 怎么做（LangGraph state_schema 继承）

LangGraph 的 `state_schema` 参数接受任何 `TypedDict`，包括子类。`TypedDict` 支持标准 Python 继承——子类自动继承父类所有字段，可以追加新字段。利用这一点：

1. **新建** `MultiAgentState(DataAgentState)` 子类，把 7 个 multi-agent 字段挪进去
2. **`supervisor_graph.py`** 用 `state_schema=MultiAgentState` 编译
3. **`graph.py` / `data_subgraph.py`** 沿用 `state_schema=DataAgentState`（无需改动，反正 single-agent 链路本来就不读那 7 个字段）
4. **`query_service.py`** 的 multi-agent 路径改用 `MultiAgentState(...)` 初始化 state

因为 `MultiAgentState` 是 `DataAgentState` 的**子类**（多态），multi-agent 节点函数签名仍声明 `state: DataAgentState` 也能接受 `MultiAgentState` 实例——不需要改 planner / aggregator / reviewer 的签名。

### #11 cached_pre_state 加权衡注释（保留 cache，文档化风险）

这条改动**不改任何代码逻辑**，只是给 `cached_pre_state` 的使用点加一段详细的权衡注释。属于 #10 拆分时的关联考虑——拆 state 字段时必须想清楚每个字段"放哪里、为什么放那里、有什么风险"。

#### 为什么（隐性 bug 风险）

先理解 `cached_pre_state` 的设计意图（背景见 `docs/notes/multi-agent-改造-20260717.md`）：

- Multi-Agent 链路里，reviewer 给 confidence < 0.7 时会触发 retry（最多 2 次）
- retry 时整条 supervisor_graph 会重跑 `planner → data_agent → aggregator → reviewer`
- `data_agent` 节点（`_gather_sub_results`）里包含"共享前置 subgraph"——`classify_intent → rewrite_query → extract_keywords → 3 路召回`，跑一次约 16 秒（含多次 LLM 调用 + Qdrant/ES 检索）
- 16 秒 × retry 次数 = 用户多等 16-32 秒，体验差

为了省这 16 秒，做了个优化：**retry 时复用上次的前置结果**。具体做法：

1. 第一次跑 `_gather_sub_results` 时，把跑完的 `shared_pre_state`（含 `keywords` / `retrieved_*` 等字段）写回 state：`return {"sub_results": ordered, "cached_pre_state": shared_pre_state}`（见 `supervisor_graph.py:233-236`）
2. retry 时再进 `_gather_sub_results`，先 `state.get("cached_pre_state")`：如果不为 None，直接复用 cache 跳过 16s 的前置 subgraph

**潜在 bug**：

- reviewer 触发 retry 的反馈，理论上可能让 planner **重新拆出不同的 sub_query 拓扑**（比如第一次拆成 2 个 sub，retry 时因为 reviewer 提示改拆成 3 个）
- 但 `cached_pre_state` 里的 `retrieved_column_infos` / `retrieved_metric_infos` / `retrieved_value_infos` 是按**旧 sub_query 的语义**检索的（前置 subgraph 跑的是原始 query，不是某个具体 sub_query）
- 新拆出的 sub_query 如果语义和旧的差异较大，cache 里的召回结果对新 sub_query 就**不匹配**——但代码会无脑复用 cache，悄悄用错上下文

这是一个**概率性**问题：

- 如果 planner retry 时拆出的 sub_query 跟上次差不多（多数情况）→ cache 完全有效，省 16 秒
- 如果 planner retry 时拆出的 sub_query 跟上次差异很大（少数情况）→ cache 里的召回结果不准，生成的 SQL 可能偏差

#### 为什么不彻底修

要彻底消除这个风险，有 3 个选项：

**选项 A：把 cached_pre_state 迁到 `runtime.context`**

不可行。LangGraph 的 `runtime.context` 是**只读**的——节点函数内不能修改 context 的字段。`cached_pre_state` 必须在 retry 时被 `_gather_sub_results` 更新（第一次跑时写入，retry 时读出），所以它必须放在可写的 state 里，不能迁 context。

这就是 `state.py:143-145` docstring 里那句"cached_pre_state 留在本 schema（不迁 context）：LangGraph 的 runtime.context 是只读的"的由来。

**选项 B：retry 时清空 cache，强制重跑前置 subgraph**

可行但放弃优化。每次 retry 都重跑 16 秒前置 subgraph，reviewer retry 2 次意味着用户多等 32 秒——这是优化本身要消除的痛点。

**选项 C（当前选择）：保留 cache，加注释说明权衡**

务实选择。理由：

1. **实际场景下 planner retry 时很少重新拆分**：reviewer 的反馈主要影响 `generate_intent` / SQL 生成层（比如"时间范围错了"、"聚合维度漏了"），planner 收到反馈后绝大多数情况是**调整 sub_query 的 query 文本**，而不是改 sub_query 的拓扑结构。
2. **前置 subgraph 跑的是原始 query 而不是具体 sub_query**：所以即使 planner 拆出的 sub_query 文本变了，只要业务领域没变（比如还是问"销售额"相关），召回的 fields / metrics / values 大概率仍适用。
3. **省 16 秒的收益**明显大于"罕见情况下的语义偏差"风险。

权衡的结果：保留 cache，但**必须把风险写进注释**，让未来读代码的人知道这里是有意为之的权衡，不是疏忽。

#### 代码改动

改动在 `app/agent/supervisor_graph.py:165-180`，给 `_gather_sub_results` 读 cache 的地方加一段 7 行注释：

```python
async def _gather_sub_results(state: MultiAgentState) -> dict[str, Any]:
    """Multi-Agent 数据执行节点：
    1. 跑 1 次共享前置 subgraph（节省 classify/rewrite/recall 重复时间）
    2. 对每个 sub_query 跑后置 subgraph（按 depends_on 分层并行）
    """
    plan = state.get("plan")
    if not plan or not plan.sub_queries:
        return {"sub_results": []}

    context = state.get("context")
    base_state = {
        "history": state.get("history", []),
        # multi-agent 假定 intent == data_query，所以预处理强制 intent
        "intent": "data_query",
    }

    # ──── Step 0：跑 1 次共享前置（省 classify/rewrite/recall 重复）──
    pre_subgraph = get_preprocessing_subgraph()
    post_subgraph = get_postprocess_subgraph()

    # 优化：retry 时复用上次的前置结果（避免 reviewer 触发 retry 重复跑 16s）
    # ⚠️ 已知权衡（2026-07-20 #11）：
    #   planner retry 时若重新拆出不同 sub_query 拓扑，cached_pre_state 里的
    #   retrieved_* 仍按旧 sub_query 语义检索，可能跟新 sub_query 不匹配。
    #   实际场景下 reviewer 反馈主要影响 generate_intent/SQL 生成层，planner
    #   很少重新拆分，所以保留 cache 的收益（省 16s）大于风险。
    #   如果未来发现 reviewer retry 时结果异常，可在这里加 plan_signature 检查：
    #   比对当前 plan 与 cached 时的 plan 拓扑，不一致就清空 cache 强制重跑。
    cached_pre = state.get("cached_pre_state")
    if cached_pre is not None:

        )
```

注释要传达三件事，缺一不可：

1. **有什么风险**：planner retry 拓扑变了 → cache 里的 retrieved_* 跟新 sub_query 不匹配
2. **为什么仍然这么选**：实际场景下 planner 很少重新拆分，cache 收益（省 16s）大于风险
3. **未来怎么彻底修**：加 plan_signature 检查，比对拓扑不一致时清空 cache

写权衡注释比写普通注释更难——它不只是"这段代码做什么"，而是"作者知道这里有缺陷，但综合考量下选择接受，理由如下"。读代码的人最怕的不是"代码不完美"，而是"代码不完美但作者没说"——权衡注释就是把隐性决策显式化。

#### 未来怎么彻底修

注释里提到的 `plan_signature` 方案，展开描述：

**思路**：cache 时不只存 `shared_pre_state`，还存一个"当时的 plan 拓扑签名"。retry 时比对当前 plan 的签名和 cache 时的签名，不一致就清空 cache 强制重跑。

**签名怎么算**：用 sub_query 的数量 + 每个 sub_query 的 `query` 文本 + `depends_on` 拓扑结构做哈希。例如：

```python
def _plan_signature(plan: QueryPlan) -> str:
    """对 plan 拓扑做签名（不含 query 文本的语义，只看结构）"""
    parts = []
    for sq in plan.sub_queries:
        # 用 query 文本的 hash + depends_on 列表
        parts.append(f"{sq.id}:{hashlib.md5(sq.query.encode()).hexdigest()[:8]}:{sorted(sq.depends_on)}")
    return "|".join(parts)
```

cache 时存：`cached_pre_state = {"shared": shared_pre_state, "plan_sig": _plan_signature(plan)}`

retry 时：

```python
cached = state.get("cached_pre_state")
if cached is not None and cached.get("plan_sig") == _plan_signature(plan):
    # 签名一致 → 安全复用
    shared_pre_state = cached["shared"]
else:
    # 签名变了（或没 cache）→ 强制重跑
    shared_pre_state = await _run_preprocessing_once(...)
```

**为什么不现在做**：

1. 当前没观察到实际问题（reviewer retry 时 planner 重新拆分的概率极低）
2. 加签名检查会让 `cached_pre_state` 的数据结构变复杂（从 `dict` 变成 `{"shared": ..., "plan_sig": ...}`），需要改多处的读写代码
3. 先用注释文档化风险，等真正出 bug 时再修——避免过度设计

这是 YAGNI（You Aren't Gonna Need It）原则的应用：除非有真实数据证明这个 bug 影响 production，否则先不动。

## 改动之间的关联

这 4 条改动表面是零散的修复，本质是同一个治理主题——**把代码里"沉默的约定"变成"显式的契约"**：

```
┌──────────────────────────────────────────────────────────────┐
│  #8 错误处理策略文档（策略层 — 定约定）                       │
│  ┌────────────────────────────────────────────────────────┐  │
│  │ 约定 1：节点分 A 类/B 类/混合类                          │  │
│  │ 约定 2：A 类失败必须推 warning                           │  │
│  │ 约定 3：字段读取用 state.get                            │  │
│  │ 约定 4：前端 warning 走黄色条                            │  │
│  └────────────────────────────────────────────────────────┘  │
│                              ↓                                │
│       #6 generate_intent warning（约定的应用案例）            │
│       ── A 类节点的标准做法：fallback + warning，不 raise     │
│                              ↓                                │
│       #10 state 拆分（结构治理 — 让字段不污染）               │
│       ── multi-agent 字段挪到子类，single-agent 不再看到      │
│                              ↓                                │
│       #11 cached_pre_state 权衡注释（拆分时的关联考虑）       │
│       ── 拆字段时想清楚"它为什么放这里、有什么风险"           │
└──────────────────────────────────────────────────────────────┘
```

具体关系：

1. **#8 是策略层**：先写一份 `error-handling-policy.md`，定义"节点应该怎么处理错误"。这是契约本身。

2. **#6 是 #8 策略的应用案例**：`generate_intent` 在策略文档里被明确列为 A 类节点（可降级的 LLM 输出节点），按约定应该"fallback 默认值 + 推 warning + 不 raise"。改前 generate_intent 只做了 fallback 没推 warning，违反约定；改后补上 warning，符合 A 类节点的标准做法。

   策略文档的价值就在这里——没有它之前，"要不要推 warning"是个人判断；有了它之后，A 类节点必须推 warning 是**约定**，code review 时可以直接拿文档对照检查。

3. **#10 是 state 结构治理**：和错误处理无直接关系，但同样是"把隐性约定显式化"。改前 multi-agent 字段塞在 `DataAgentState` 里，读者要从注释里猜"哪些字段是 single-agent 用的、哪些是 multi-agent 用的"；改后字段位置直接说明归属，读者看类型定义就懂。

4. **#11 是 #10 拆分时的关联考虑**：拆 `cached_pre_state` 字段时必须想清楚"它为什么不能迁到 context"（因为 context 只读）、"它放 state 里有什么风险"（retry 时拓扑变化导致 cache 不匹配）。#11 把这些考虑写进注释，避免未来有人看到这个字段时重新踩一遍坑。

共同主题：

- **#6**：把"降级发生"这个事实从 logger 搬到 writer，让**用户**看到
- **#8**：把"节点该怎么处理错误"从口头约定写成**文档**，让新人有据可查
- **#10**：把"哪些字段属于哪条链路"从注释里搬到**类型定义**里，让 IDE 和类型检查器能识别
- **#11**：把"这个设计有什么权衡"从脑子里搬到**代码注释**里，让未来读者知道作者考虑过

这四件事的共同模式：**把信息从"只有作者知道"的地方（运行时行为、口头约定、大脑里的权衡）搬到"团队都能看到"的地方（UI、文档、类型系统、代码注释）**。这是工程化代码和个人项目代码的本质区别——前者要求"显式"，后者容忍"默契"。

## 附录：LangGraph state_schema vs context_schema 区别

本次 #10 / #11 改动反复涉及"字段放 state 还是 context"的决策，这里展开解释两者区别：

| 维度 | `state_schema` (StateGraph 第一个参数) | `context_schema` (StateGraph 第二个参数) |
|---|---|---|
| **可变性** | 节点可写回（return 的 dict 会被框架 merge 回 state） | **只读**（节点函数内不能修改 context 字段） |
| **生命周期** | 跟随一次 graph 执行（多次 retry 间持久化） | 跟随 graph 实例（多次调用间共享） |
| **典型内容** | 业务流转数据（query / keywords / sql / 结果等） | 依赖注入（仓储、客户端、配置等） |
| **本次例子** | `query` / `plan` / `cached_pre_state` / `sub_results` | `dw_mysql_repository` / `embedding_client` / `qdrant_client` 等 |

`cached_pre_state` 必须放 state 不放 context 的根本原因：它要被 `_gather_sub_results` **写回**（第一次跑完写入、retry 时读出），而 context 是只读的。

理解这个区别后，未来再遇到"这个字段放哪里"的问题，判断标准很简单：**节点需不需要写它？需要 → state，不需要 → context（或常量）**。

## 附录：常见问题 FAQ

### Q1：为什么 generate_intent 降级时不直接 raise，让上层报错给用户？

**A**：因为降级本身是设计行为，不是异常。链路设计上允许 generate_intent 失败时降级为 `{}`，让 generate_sql 渲染 `SELECT 1 AS fallback`——这条路径保证了"即使 LLM 输出坏 JSON，链路也不会崩"。如果改成 raise，会让本来能跑的链路挂掉，且用户看到一个红色错误条却不知道"其实系统还能给个占位结果"。

更细粒度地说：generate_intent 失败有两种后果——

- **完全无法理解用户意图**（LLM 输出乱码）→ 应该让用户重述问题，但仍然给个占位结果（当前做法）
- **意图部分理解但格式不对**（缺字段）→ 同上，降级 + 提示

两种情况都需要"占位结果 + 提示"，而不是"硬报错 + 终止"。所以推 warning 比 raise 更合适。

### Q2：MultiAgentState 是子类，那 multi-agent 节点能直接读 single-agent 字段吗？

**A**：可以。`MultiAgentState` 继承自 `DataAgentState`，子类实例包含父类所有字段。所以 `_gather_sub_results(state: MultiAgentState)` 里能直接 `state["query"]` / `state["keywords"]`——这些是父类字段，子类实例都有。

反过来不行：声明为 `DataAgentState` 的参数传 single-agent 实例时，**不能**读 `state["plan"]`（运行时拿不到，TypedDict 不强制校验，会返回 None 或 KeyError 取决于 dict 实际内容）。这就是为什么 `_gather_sub_results` 的签名从 `DataAgentState` 改成了 `MultiAgentState`——它要读 `plan` / `cached_pre_state` 等 multi-agent 专属字段。

### Q3：state.get 的默认值怎么选？随便给一个？

**A**：默认值要"对下游无害"。常见选择：

| 字段类型 | 合理默认 | 反例（有害默认） |
|---|---|---|
| 列表字段（如 `keywords`） | `[]` | `None`（下游 `for k in keywords` 会崩） |
| 字典字段（如 `time_range`） | `{}` 或全字段空 dict | `None`（下游 `time_range["start_date"]` 会崩） |
| 数值字段（如 `confidence`） | `1.0`（高分=通过） | `0.0`（会让 reviewer 误判要 retry） |
| 字符串字段（如 `intent`） | `"data_query"`（最常见分支） | `""`（路由分支可能匹配不上） |

核心原则：默认值要让下游**自然走到"最安全"的分支**，而不是触发异常或异常分支。比如 `confidence` 默认 `1.0` 是因为 reviewer 看到 1.0 会判通过、不 retry；如果默认 0.0 会无限 retry。

### Q4：run_sql 里的 _safe_error_message 会不会把有用的调试信息也藏掉？

**A**：会，但这是设计意图。原始异常（如 `Table 'dw.dim_product' doesn't exist`）含表名、列名、SQL 片段，对运维排查有价值但对**用户**无价值且敏感（可能暴露数仓结构）。所以策略是：

- **完整异常**写 `logger.error(f"... {e}")`（服务端日志保留，运维可查）
- **脱敏文案**写 `writer({"type": "error", "message": ...})`（前端展示，不含敏感信息）

调试时如果要看完整异常，去后端日志文件查（不是问用户截图）。这是生产级错误处理的常见做法——区分"内部诊断信息"和"用户可见信息"。

### Q5：cached_pre_state 的风险真有那么大吗？为什么之前没出问题？

**A**：风险**理论存在**，实际很低，原因有三：

1. **reviewer retry 频率本身就低**：reviewer 打分 `< 0.7` 才触发 retry，且 max_loop=2。线上多数查询不会进 retry。
2. **reviewer 反馈通常针对 SQL 层**：reviewer 看到"时间范围错了"、"WHERE 条件漏了"这种反馈，会让 planner 调整 sub_query 的 query 文本，而不是改 sub_query 拓扑（拆几个、依赖关系）。
3. **前置 subgraph 跑的是原始 query**：不是某个具体 sub_query。所以即使 sub_query 文本变了，前置召回的"业务领域相关字段/指标/取值"通常仍适用。

所以"之前没出问题"≠"没风险"，而是"风险触发条件很少满足"。注释的目的是让未来读者知道这个判断，如果哪天 reviewer 改造让 retry 频率变高 / 拓扑变化更频繁，就要重新评估这个权衡。

### Q6：拆 MultiAgentState 之后，老的测试需要改吗？

**A**：理论上不需要。改动只是把字段从父类挪到子类，对运行时行为零影响（TypedDict 是类型注解，运行时仍是普通 dict）。具体分两种情况：

- **multi-agent 测试**（如 `test_planner_node.py`）：构造 state 时用 `MultiAgentState(...)` 或直接用 dict 字面量都行，运行时等价。本次改动后 query_service 改用 `MultiAgentState(...)` 初始化，但测试里如果有 `state = {"query": "...", "plan": ...}` 这种字面量构造，仍然跑得通。
- **single-agent 测试**：完全不受影响，因为它从来就不读 multi-agent 字段。

唯一可能需要改的情况：测试里用了 `isinstance(state, MultiAgentState)` 这种类型判断——但 TypedDict 运行时是普通 dict，`isinstance` 检查本身就不靠谱，应该避免。

### Q7：error-handling-policy.md 里说"必须给前端一个 warning"，但有些节点忘了怎么办？

**A**：靠两道防线：

1. **Code review 时对照策略文档检查**：A 类节点 PR 必须看 except 块有没有 `writer({"type": "warning", ...})`。可以在 PR 模板里加一条 checklist。
2. **未来加 lint 检查**：写个简单的 AST 扫描脚本，找所有 `async def` 节点函数，如果 except 块里有 `return {...}` 降级但没有 `writer({"type": "warning"`，就报错提醒。这是后续可做的工程化治理。

短期靠人 review，长期靠工具。本次 #8 主要解决"约定不存在"的问题；"约定怎么强制"是下一步。
