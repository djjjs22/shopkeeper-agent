## 详解：每条改动的"为什么 / 怎么做 / 原来 / 代码改动"

### #6 前端：SSE warning 事件渲染

#### 为什么（用户看不到降级提示）

先把"warning 事件从哪来"交代清楚，这是理解前端改造的前提。

**后端链路（引用，不展开）**：

1. `app/agent/nodes/generate_intent.py:181-194`（后端 #6）：LLM 调用解析失败时，原本降级为空 `intent = {}`，下游 `generate_sql` 节点用 `SELECT 1` 兜底。用户看到"1"这个结果但完全不知道出了什么问题。后端 #6 在降级分支里多推一条 warning：

   ```python
   # app/agent/nodes/generate_intent.py:187
   writer({
       "type": "warning",
       "step": step,
       "message": "未能理解查询意图，将返回占位结果。请换一种更具体的问法...",
   })
   ```

2. `app/agent/nodes/run_sql.py:313`（后端 #3 相关）：空结果校验失败时也推 warning；`run_sql.py:328-331` 结果集超过 5000 行被 LIMIT 截断时，同时推 warning + result 事件带 `truncated: true`。

**前端原本的问题**：

- `frontend/src/types/agent.ts` 改动前的 `AgentEvent` 是三元素 union：

  ```typescript
  export type AgentEvent = ProgressEvent | ResultEvent | ErrorEvent;
  ```

  没有 `WarningEvent` 类型，前端无法识别 `type: "warning"` 的事件。

- `App.tsx` 的 `onEvent` 是一串 `if (event.type === "...")` 判断，最后落到 `else`（当 error 处理）。

- 结果：后端推的 `{"type": "warning", ...}` 在前端被当成 error 处理，message 状态变成 `"error"`，原本的进度信息全丢了，UI 错乱——**后端想提示降级，前端却展示成"查询失败"**。

#### 原来是怎么样的

##### 类型定义

```typescript
// 改动前：frontend/src/types/agent.ts（节选）
export type ResultEvent = {
  type: "result";
  data: unknown;
  // ⚠️ 没有 truncated / max_rows 字段
};

export type ErrorEvent = {
  type: "error";
  message: string;
};

export type AgentEvent = ProgressEvent | ResultEvent | ErrorEvent;
// ⚠️ 没有 WarningEvent

export type ChatMessage = {
  id: string;
  role: "user" | "assistant";
  content: string;
  createdAt: number;
  status?: "streaming" | "done" | "error";
  steps?: StepState[];
  result?: unknown;
  // ⚠️ 没有 warnings 字段
  error?: string;
};
```

##### App.tsx 的 onEvent（改动前结构）

改动前的 `onEvent` 是这样一串判断（简化版，保留结构、省略细节）：

```typescript
const onEvent = (event: AgentEvent) => {
  setMessages((current) =>
    current.map((message) => {
      if (message.id !== assistantId) return message;

      if (event.type === "progress") {
        return { ...message, /* 更新 content / steps */ };
      }

      if (event.type === "result") {
        return { ...message, status: "done", /* 写 content / result */ };
      }

      // ⚠️ 改动前这里就是兜底（实际等同于 else）
      // warning 事件因为没有匹配分支，落到这里被当 error
      return {
        ...message,
        status: "error",
        content: mapBackendMessage(event.message).title,
        error: mapBackendMessage(event.message).detail,
      };
    }),
  );
};
```

**问题**：TypeScript 的 discriminated union 在这里发挥了反作用——`event.type` 被收窄到 `"result"` 后，剩下的分支里 `event` 的类型是 `ErrorEvent`，TS 认为 `event.message` 一定存在。但实际上后端推了 `WarningEvent`，运行时 `event.type === "warning"`，三个 `if` 都不匹配，落到最后的 `return` 把它当 error 处理。

这就是 discriminated union 在后端协议演进时的一个典型坑：**前端类型一旦落后于后端，"未识别的事件类型"会被静默地错误归类**。

**实际表现**（改动前的 bug 现场）：

1. 用户问一个 LLM 解析不出意图的怪问题（例如"今天天气怎么样"）。
2. 后端 `generate_intent` 失败，降级为 `intent = {}`，下游跑 `SELECT 1` 返回 `[{"1": 1}]`。
3. 后端推 warning："未能理解查询意图..."
4. 前端收到 warning 事件，但因为类型里没有 WarningEvent，落到 error 分支：
   - `message.status` 从 `"streaming"` 变成 `"error"`。
   - `message.content` 被改成 `mapBackendMessage("未能理解查询意图...").title`（一般是"查询失败"）。
   - `message.error` 写入友好文案。
5. 紧接着后端推 progress（"生成 SQL" running），但 message 已经是 error 状态，UI 显示矛盾。
6. 最后后端推 result（`SELECT 1` 结果），但 message.status 是 error，ResultTable 仍然可能渲染（取决于条件分支），整体 UI 错乱。

用户视角看到的就是："问了个怪问题，前端先显示查询失败，又冒出一个表格写着 1，不知所云"。后端做了降级容错的努力，全被前端这一层误判抹掉了。

#### 怎么做（类型扩展 + 累积渲染）

设计四层：

1. **类型层**：加 `WarningEvent`，并把它加入 `AgentEvent` union。这样 `onEvent` 里访问 `event.message` 时 TS 能精确收窄。
2. **state 层**：`ChatMessage` 加 `warnings?: string[]`（数组而不是单个 string，因为一次查询可能产生多个 warning——例如既有"意图解析失败"又有"结果截断"）。
3. **渲染层**：`MessageBubble` 用 `message.warnings?.map(...)` 渲染多个黄色提示条。
4. **去重**：同一个 message 内部，对同一句 warning 文本去重，避免后端重复推导致 UI 重复。

#### 代码改动（含完整注释）

##### types/agent.ts 扩展

```typescript
// 改动后：frontend/src/types/agent.ts:21-38

export type ErrorEvent = {
  type: "error";
  message: string;
};

// 2026-07-20 (#6)：warning 事件（如意图解析失败、结果截断）
// 与 error 区别：warning 不终止流程，只是提示用户结果可能不完整
export type WarningEvent = {
  type: "warning";   // ⭐ 字面量类型，让 AgentEvent union 可辨识
  message: string;   // 必填：给用户看的人话文案
  step?: string;     // 可选：产生 warning 的节点名（如"生成查询意图"），便于排查
};

// 把 WarningEvent 加入 union 后，TS 在 onEvent 里收窄 event.type 时
// 能正确识别 warning 分支，不再误归到 error
export type AgentEvent =
  | ProgressEvent
  | ResultEvent
  | ErrorEvent
  | WarningEvent;   // ⭐ 新增成员
```

```typescript
// 改动后：frontend/src/types/agent.ts:46-57

export type ChatMessage = {
  id: string;
  role: "user" | "assistant";
  content: string;
  createdAt: number;
  status?: "streaming" | "done" | "error";
  steps?: StepState[];
  result?: unknown;
  truncated?: boolean;  // 2026-07-20 (#3)：结果集是否被截断（详见 #23-C）
  warnings?: string[];  // 2026-07-20 (#6)：累积的 warning 提示（数组，一次查询可能多条）
  error?: string;
};
```

**为什么 `warnings` 是 `string[]` 而不是 `string`**：

- 一次查询可能既触发 `generate_intent` 的"意图解析失败"warning，又触发 `run_sql` 的"结果截断"warning，两条文案都要展示。
- 后端流式推送，前端按顺序累积，所以用数组而不是单个字符串。
- `?` 可选：大部分 message 不会有 warning，不必默认空数组占内存。

##### App.tsx onEvent 新增 warning case

```tsx
// 改动后：frontend/src/App.tsx:172-181
if (event.type === "warning") {
  // 2026-07-20 (#6)：warning 累加到 message.warnings，不终止流程
  // ⭐ 为什么不设置 status: "error"：warning 不是错误，链路可能正常返回结果
  //    （例如结果截断的 warning 同时会跟一个正常的 result 事件）
  const existing = message.warnings ?? [];
  // ⭐ 为什么 ?? [] 而不是直接用 message.warnings：
  //    warnings 是可选字段，第一次收到 warning 时它是 undefined，
  //    [].concat 会报错。空数组兜底让首条 warning 也能正常追加。
  return {
    ...message,
    warnings: existing.includes(event.message)
      ? existing   // ⭐ 去重：同一句 warning 不重复追加
                   // 后端可能因为 retry / 重发机制推两次相同文案
      : [...existing, event.message],  // ⭐ 不可变更新：用 spread 浅拷贝再追加
                                       // 不能用 existing.push()，会改原数组导致 React 不重渲染
  };
}
```

**几个关键决策的注释**：

1. **`existing.includes(event.message)` 去重**：用全字符串比较。这里假设后端同一 warning 的文案是确定性的（同一个 `writer({"type": "warning", "message": "..."})` 调用推两次，message 字段完全一致）。如果后端把时间戳或动态值拼进 message，去重会失效，但当前后端实现是固定文案，所以这个去重策略够用。

2. **为什么不更新 `content` 字段**：`content` 是气泡的主体文本（"正在执行：xxx" / summarizeResult 等），warning 是**附加信息**而不是**主体内容**。把 warning 放进 content 会让"正在执行：生成 SQL"被覆盖掉，破坏正在进行的进度展示。

3. **为什么不修改 `status` 字段**：warning 不改变流程状态。message 可能同时是 `status: "done"`（流程正常结束）+ 有 warnings（期间有过降级）。把 status 改成 error 会触发 `MessageBubble` 的错误分支渲染。

4. **`map` 闭包返回新对象**：React 状态更新必须返回新引用，浅比较才会感知变化。`{...message, warnings: ...}` 是浅拷贝并替换 warnings 字段，引用变化触发重渲染。

##### MessageBubble.tsx 渲染

```tsx
// 改动后：frontend/src/components/MessageBubble.tsx:128-136

{/* 2026-07-20 (#6): warning 提示条（黄色），如意图解析失败 */}
{message.warnings?.map((w, i) => (
  <div
    key={i}
    className="mt-3 border border-amber-400/40 bg-amber-50 px-3 py-2 text-sm text-amber-700 dark:bg-amber-900/20 dark:text-amber-300"
  >
    ⚠️ {w}
  </div>
))}
```

**Tailwind class 逐项解释**：

- `mt-3`（margin-top: 0.75rem）：与上方的 content 文本拉开 12px 间距，避免视觉粘连。
- `border border-amber-400/40`：1px 边框，amber-400 是 Tailwind 内置的橙黄色（`#fbbf24`），`/40` 是 40% 透明度。**为什么用 `/40` 而不是纯色**：边框是辅助元素，透明度低一些不抢眼，主体靠背景色突出。
- `bg-amber-50`：极浅黄背景（`#fffbeb`），类似 macOS 的通知背景。
- `px-3 py-2`：内边距 12px / 8px，给文字呼吸感。
- `text-sm`：14px 字号，比 content 略小（content 是 15px），表明是辅助信息。
- `text-amber-700`：深琥珀色文字（`#b45309`），在浅黄背景上有足够对比度（WCAG AA 通过）。
- `dark:bg-amber-900/20 dark:text-amber-300`：暗色模式覆盖。`amber-900/20` = 深棕黄 20% 透明度叠在暗色背景上，`amber-300` 是亮琥珀文字。**为什么需要 dark: 覆盖**：amber-50 在深色背景下会刺眼，必须切换到深色背景 + 亮色文字。

**`?.map` 的妙用**：

```tsx
{message.warnings?.map((w, i) => ...)}
```

`message.warnings` 可能是 `undefined`（大部分 message 没有 warning）。可选链 `?.` 让 undefined 直接短路返回 undefined，React 把 undefined 渲染成空，不会报错。等价于：

```tsx
{message.warnings && message.warnings.length > 0
  ? message.warnings.map((w, i) => ...)
  : null}
```

但更简洁。

**`key={i}` 的取舍**：用数组索引作 key 在大多数场景是反模式（数组顺序变化会导致 React 错乱复用 DOM）。但这里 warnings 是**只追加、不重排、不删除**的数组，索引作 key 安全。如果未来要支持"撤回某条 warning"，需要换成稳定 id。

#### UX 决策

**1. 为什么 warnings 用黄色（amber）而不是红色？**

| 颜色 | 语义 | 在本项目的用法 |
|------|------|------|
| 红（apple-red） | 流程终止、不可恢复 | `MessageBubble.tsx:123` 的 `message.error` 块 |
| 黄（amber） | 流程继续、结果可能不完整 | warning 块、truncated 块 |

warning **不终止流程**——`generate_intent` 失败后链路继续走 `SELECT 1` 兜底，用户能拿到结果（虽然没意义）。error 是 `run_sql` 抛异常、链路终止。两者语义不同，颜色必须区分。

**2. 为什么不弹 toast（Toast 系统已存在）？**

`Toast.tsx` 已经实现了 warning 类型的右下角通知。但本次改动没用它，原因：

- warning 跟**具体某条 message** 绑定（"这次问数的意图解析失败了"），toast 是全局浮层，没有上下文。
- 用户滚动消息列表回头看时，toast 已经消失了（默认 3s 自动消失），无法回溯。
- 多个 warning 堆叠在 toast 里会变成"通知轰炸"。

所以选择**气泡内联渲染**，跟 message 绑定持久存在。

**3. 为什么允许累积多个 warning？**

一次查询的链路是 `extract_keywords → generate_intent → recall_* → generate_sql → run_sql`，多个节点都可能推 warning：

- `generate_intent` 失败 → "未能理解查询意图"
- `run_sql` 空结果 → "返回 0 行数据"
- `run_sql` 截断 → "结果集过大"

这些 warning 互不冲突，都应该展示给用户。累积式渲染让用户看到完整的"链路降级过程"。

#### 验证方式

手动验证（无自动化测试）：

1. 启动后端，故意让 `generate_intent` 失败（例如临时改 prompt 让 LLM 输出无法解析的格式）。
2. 前端发起一次查询，应该看到：
   - 气泡里出现黄色提示条："⚠️ 未能理解查询意图，将返回占位结果..."
   - 同时气泡底部仍然出现 `SELECT 1` 的结果表格（1 行 1 列）。
   - 气泡 status 是 `done`，不是 `error`。
3. 切换暗色模式，提示条应该自动变成深棕黄背景 + 亮琥珀文字。

#### 与 React.memo 的交互

`MessageBubble` 用 `memo(MessageBubbleImpl)` 包裹（`MessageBubble.tsx:179`），浅比较 props 决定是否重渲染。本次新增的 `warnings` 和 `truncated` 字段会影响 memo 的判断：

```typescript
// React.memo 的浅比较伪代码
function shallowEqual(prevProps, nextProps) {
  const prevKeys = Object.keys(prevProps);
  const nextKeys = Object.keys(nextProps);
  if (prevKeys.length !== nextKeys.length) return false;
  for (const key of prevKeys) {
    if (prevProps[key] !== nextProps[key]) return false;
  }
  return true;
}
```

`message` 是 props 之一。当 App.tsx 调用 `setMessages` 更新某个 message 的 warnings 时：

- `{...message, warnings: [...]}` 创建**新的 message 对象**，引用变化。
- `messages.map(m => m.id === targetId ? newMessage : m)` 创建**新的 messages 数组**。
- App.tsx 把新 messages 传给 MessageBubble，浅比较 `prev.message !== next.message` → 不等 → 重渲染。

这正是我们想要的：**只有发生 warning 累加的那条 message 重渲染，其他 message 浅比较相等（`m === m`，引用没变）→ 跳过重渲染**。

如果改成原地修改（`existing.push(...)`）：

- message 引用没变，浅比较认为 props 相等，**memo 跳过重渲染**。
- React 看不到状态变化，UI 不更新，warning 不显示。
- 这就是为什么"不可变更新"在 React 里是硬性要求，不是风格偏好。

`App.tsx:254-262` 还有 useRef + useCallback 模式：

```typescript
const startQueryRef = useRef(startQuery);
startQueryRef.current = startQuery;
const handleUseExample = useCallback((example: string) => setDraft(example), []);
```

这是为了让传给子组件的 callback 引用稳定，配合 memo。本次没改动这部分，但理解 #6 的渲染链路时需要知道：**warning 的渲染性能依赖整个 App.tsx 的 memo 友好结构**。

#### 子改造 A：ResultTable 客户端分页

##### 为什么（大数据集卡顿）

代码里有这一行：


- 在渲染过程中调用 `setPage` 会触发 React 警告（"Cannot update a component while rendering a different component"）。

#### 子改造 B：SSE 断连提示

##### 为什么（连接中断用户无感）

改动前 `agentApi.ts` 的 `streamQuery` 是这样读流：

```tsx
// 改动前：frontend/src/lib/agentApi.ts（简化）
const reader = response.body.getReader();
while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  // 解析 chunk，调 onEvent
}
```

问题：

- 网络抖动、后端崩溃、代理超时等情况下，`reader.read()` 返回 `done: true` 直接退出循环。
- 流结束时如果没有收到 `result` 或 `error` 事件，前端**完全不知道出问题了**。
- 用户看到气泡停在"正在执行：xxx"状态，一直转圈，直到外层 `try/catch` 超时（实际上 `fetch` 流式响应没有默认超时，可能无限卡住）。
- 已收到的部分 progress 信息不会被清理，但也不会推进。

##### 原来是怎么样的

```tsx
// 改动前：frontend/src/lib/agentApi.ts
const reader = response.body.getReader();
const decoder = new TextDecoder("utf-8");
let buffer = "";

while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  // ... 解析 chunk，调 options.onEvent(event)
}

// 流末尾的半截 buffer
buffer += decoder.decode();
const tail = parseSseChunk(buffer);
if (tail) {
  options.onEvent(tail);
}

// ⚠️ 流结束就 return 了，没有任何"是否正常结束"的检查
```

##### 怎么做（跟踪 terminated 状态）

设计：

1. 用一个布尔变量 `terminated` 跟踪是否收到过 `result` 或 `error` 事件（这两种是后端明确的"链路结束"信号）。
2. 在主循环和尾部 tail 处理两个地方，遇到 `result`/`error` 就把 `terminated` 置为 `true`。
3. 流结束时检查 `terminated`，如果是 `false` 就**主动推一个 warning 事件**给上层。
4. 上层 `App.tsx` 的 `onEvent` 按 #6 的机制渲染"连接中断"黄色提示条。

##### 代码改动（含完整注释）

```tsx
// 改动后：frontend/src/lib/agentApi.ts:73-114

const reader = response.body.getReader();
const decoder = new TextDecoder("utf-8");
let buffer = "";
// 2026-07-20 (#23)：跟踪是否收到过 result/error 终止事件
// 流意外断开且没终止事件 → 推一个 warning 让前端 UI 提示用户
let terminated = false;
// ⭐ 为什么用局部变量而不是 React state：
//    streamQuery 是一个纯 async 函数，跟 React 渲染无关
//    它的状态不需要触发重渲染，只需要在函数内部跟踪即可

while (true) {
  const { value, done } = await reader.read();
  if (done) break;

  buffer += decoder.decode(value, { stream: true });
  const chunks = buffer.split(/\n\n/);
  buffer = chunks.pop() ?? "";
  // ⭐ chunks.pop() 取出最后一段半截 buffer 留给下次拼接
  //    SSE 协议用 \n\n 分隔事件，但 TCP 分包可能把一个事件切成两段

  for (const chunk of chunks) {
    const event = parseSseChunk(chunk);
    if (event) {
      // ⭐ 检测到终止事件就标记
      if (event.type === "result" || event.type === "error") {
        terminated = true;
      }
      options.onEvent(event);
    }
  }
}

// 流结束时 decoder 里可能还残留最后几个字节
buffer += decoder.decode();
const tail = parseSseChunk(buffer);
if (tail) {
  if (tail.type === "result" || tail.type === "error") {
    terminated = true;
  }
  options.onEvent(tail);
}

// 流结束但没拿到 result/error → 连接中断，提示用户
if (!terminated) {
  options.onEvent({
    type: "warning",
    message: "连接中断，未能拿到完整结果。请重新发起查询。",
  });
}
// ⭐ 为什么不 throw new Error：
//    用户可能已经收到部分进度（progress 事件已渲染），
//    throw 会被 App.tsx 的 catch 当成 ApiError/JS Error 处理，
//    把整个 message 状态改成 error，覆盖掉已有的进度展示。
//    推 warning 是"附加提示"，保留已有信息，让用户自己判断要不要重试。
```

**几个决策的注释**：

1. **为什么 `result` 和 `error` 都算终止**：
   - `result` 是后端 `run_sql.py:335` `writer(result_event)`，表示"查询成功，结果已发"。
   - `error` 是后端 `run_sql.py` 或 `query_service.py` 推的异常事件，表示"查询失败，已告知原因"。
   - 两种都是后端**明确的"链路结束"信号**——之后不会再有事件。
   - `warning` **不算终止**：因为 warning 之后可能还跟一个正常的 result（例如"结果截断"warning 后紧跟带 truncated 的 result）。
   - `progress` **不算终止**：节点还在跑。

2. **为什么不用 `throw new Error`**：
   - 用户在断连前可能已经收到若干 progress 事件，气泡里已经渲染了进度条。
   - throw 会让 `App.tsx` 走 catch 分支，`message.status` 被改成 `"error"`，覆盖已有的 progress 展示。
   - 推 warning 是**叠加**，原本的进度信息保留，用户看到"进度走到一半 + 黄色连接中断提示"，体验更好。

3. **为什么是 `terminated` 局部变量而不是 ref**：
   - `streamQuery` 是普通 async 函数，不依赖 React 渲染周期。
   - 局部变量在函数作用域内就够用，函数返回后自动 GC。
   - 用 ref 反而多一层间接，没有收益。

注意 `ChatMessage` 没加 `max_rows` 字段——`max_rows` 是后端协议信息，但 UI 文案里写死了 "5000 行"（见 `MessageBubble.tsx:141`），所以前端不需要把它带到 state。如果未来要支持动态上限，可以再补。

###### App.tsx onEvent 处理

```tsx
// 改动后：frontend/src/App.tsx:161-170

  // 2026-07-20 (#3)：结果截断时把 truncated 标记带到 message
  return {
    ...message,
    status: "done",
    content: summarizeResult(event.data),
    result: event.data,
    truncated: event.truncated === true ? true : undefined,
    // ⭐ 为什么写成 event.truncated === true ? true : undefined：
    //    1. event.truncated 是可选字段，可能是 true / false / undefined
    //    2. 用 === true 严格匹配，避免 truthy 误判（比如后端误传字符串 "true"）
    //    3. 取 true 或 undefined（不取 false）：
    //       - false 和 undefined 在 message.truncated 的语义上等价（"没截断"）
    //       - 但 undefined 在 React 渲染时 {message.truncated && ...} 直接短路，
    //         而 false 还要走一遍 &&（虽然结果一样）
    //       - 同时让 ChatMessage 的 truncated 字段保持稀疏（只有真截断才有值）
  };
}
```

**严格比较 `=== true` 而不是 `=== true` 的省略形式**：

如果写成 `truncated: event.truncated`，那么：

- `event.truncated === true` → `message.truncated = true` ✅
- `event.truncated === undefined` → `message.truncated = undefined` ✅
- `event.truncated === false` → `message.truncated = false` ⚠️（语义和 undefined 等价，但值不同）

虽然结果都对（`false && X` 和 `undefined && X` 都不渲染），但保持 `truncated` 字段**只在真截断时存在**，让 React.memo 的浅比较更稳定（false 和 undefined 在浅比较时不算相等，可能触发不必要的重渲染）。

## 整体 SSE 事件协议（改动后）

改动后，`/api/query` 的 SSE 流支持 4 种事件类型。完整协议如下：

| 事件类型 | 触发节点 | 字段 | 前端处理 |
|---|---|---|---|
| `progress` | 每个节点的 running / success / error 状态变化 | `step: string`、`status: "running"\|"success"\|"error"` | `App.tsx:153` 更新 `message.content` + `upsertStep` 累积到 `message.steps`；`StepRail` 渲染步骤条 |
| `result` | `run_sql` 成功（`run_sql.py:335`） | `data: unknown`、`truncated?: boolean`、`max_rows?: number` | `App.tsx:161` 设置 `status: "done"`，把 `data` 存到 `message.result`，`truncated` 存到 `message.truncated`；`ResultTable` 渲染表格 + 分页 |
| `warning` | `generate_intent` 降级（后端 #6）、空结果（`run_sql.py:313`）、结果截断（`run_sql.py:328`）、前端断连兜底（`agentApi.ts:109`） | `message: string`、`step?: string` | `App.tsx:172` 累加到 `message.warnings`（去重）；`MessageBubble:129` 渲染黄色提示条 |
| `error` | `run_sql` 安全拦截（`run_sql.py:272`）、DB 异常、链路异常 | `message: string` | `App.tsx:183` 设置 `status: "error"`；`MessageBubble:122` 渲染红色错误条 + toast |

**事件流时序示例**（一次正常查询）：

```
客户端 POST /api/query
  ↓
后端流式推送：
  data: {"type":"progress","step":"抽取关键词","status":"running"}
  data: {"type":"progress","step":"抽取关键词","status":"success"}
  data: {"type":"progress","step":"生成查询意图","status":"running"}
  data: {"type":"progress","step":"生成查询意图","status":"success"}
  ...（recall / generate_sql / run_sql）
  data: {"type":"warning","message":"结果集过大，仅展示前 5000 行..."}  ← 截断 warning
  data: {"type":"result","data":[...],"truncated":true,"max_rows":5000}
  ↓
流结束（result 已收到 → terminated=true → 不推断连 warning）
```

**事件流时序示例**（意图解析失败降级）：

```
后端推送：
  data: {"type":"progress","step":"生成查询意图","status":"running"}
  data: {"type":"warning","step":"生成查询意图","message":"未能理解查询意图..."}
  data: {"type":"progress","step":"生成查询意图","status":"success"}  ← 注意：降级也算 success
  ...（后续节点用空 intent 继续跑，最终 SELECT 1）
  data: {"type":"result","data":[{"1":1}]}
  ↓
流结束
```

**事件流时序示例**（连接中断）：

```
后端推送到一半（例如刚发完"生成 SQL" progress）：
  data: {"type":"progress","step":"生成 SQL","status":"running"}
  ↓
网络断开，reader.read() 返回 done=true
  ↓
前端 terminated=false（没收到 result/error）
  ↓
前端主动推：options.onEvent({type:"warning", message:"连接中断..."})
  ↓
App.tsx 把 warning 加到 message.warnings，气泡显示"连接中断"黄色提示
```

**SSE chunk 在网络层的实际格式**：

后端推送的每个事件在 HTTP 响应体里长这样（文本协议）：

```
data: {"type":"progress","step":"抽取关键词","status":"running"}

data: {"type":"progress","step":"抽取关键词","status":"success"}

data: {"type":"result","data":[{...},{...}]}
```

每个事件以 `data: ` 开头，事件之间用空行（`\n\n`）分隔。`agentApi.ts:85` 的 `buffer.split(/\n\n/)` 就是按空行切块，每个块再 `parseSseChunk` 解析。

TCP 分包可能导致一个完整事件被切成两个 chunk 到达：

```
第一次 reader.read() 返回：
  "data: {\"type\":\"progress\",\"step\":\"抽取关键词"

第二次 reader.read() 返回：
  "\",\"status\":\"running\"}\n\n"
```

`agentApi.ts:86` 的 `chunks.pop()` 把最后一段（不完整的）留在 buffer 里等下次拼接，避免解析半个 JSON 失败。如果直接 `JSON.parse(buffer)` 会抛异常，事件丢失。

### 常见坑

**坑 1：用 `existing.push(event.message)` 而不是 `[...existing, event.message]`**：

```tsx
// ❌ 错误写法
const existing = message.warnings ?? [];
existing.push(event.message);  // 直接改原数组
return { ...message, warnings: existing };

// 问题：虽然 warnings 字段被改了，但因为还是同一个数组引用，
// React 的浅比较认为没变化（prev.warnings === next.warnings），
// MessageBubble 不会重渲染，新的 warning 不显示。
```

```tsx
// ✅ 正确写法（本次改动）
const existing = message.warnings ?? [];
return {
  ...message,
  warnings: [...existing, event.message],  // 创建新数组
};
```

React 状态更新的铁律：**永远返回新引用，不要原地修改**。这跟 Vue 的响应式机制不同（Vue 用 Proxy 能感知 push），React 是纯函数式思路。

**坑 2：useEffect 依赖写 `[rows]` 导致无限循环**：

```tsx
// ❌ 错误写法
const rows = normalizeRows(data);  // 每次渲染都返回新数组
useEffect(() => {
  setPage(0);
}, [rows]);  // ⚠️ rows 每次渲染都是新引用 → useEffect 每次都跑 → setPage(0) → 重渲染 → 又跑...

// 结果：组件疯狂重渲染，浏览器卡死，控制台报警 "Maximum update depth exceeded"
```

```tsx
// ✅ 正确写法（本次改动）
const rows = normalizeRows(data);
useEffect(() => {
  setPage(0);
}, [data]);  // data 是 prop，引用稳定，只在父组件真的换了 data 时触发
```

记住原则：**useEffect 依赖里不要放 "派生计算出来的引用类型"**（数组、对象、函数），只放原始值或稳定的引用。

**坑 3：在渲染过程中调用 setPage 修正越界**：

```tsx
// ❌ 错误写法
const safePage = Math.min(page, totalPages - 1);
if (safePage !== page) {
  setPage(safePage);  // ⚠️ 渲染中 setState，React 报警告
}
```

```tsx
// ✅ 正确写法（本次改动）
const safePage = Math.min(page, totalPages - 1);
// 不写回 state，只在当前渲染里用 safePage
const pagedRows = rows.slice(safePage * PAGE_SIZE, safePage * PAGE_SIZE + PAGE_SIZE);
```

派生值（能从已有 state + props 算出来的值）不要再存一份到 state，每次渲染时算即可。

**坑 4：discriminated union 漏写分支让 TS 漏检**：

```typescript
// ❌ 危险写法（改动前的 onEvent）
if (event.type === "progress") return ...;
if (event.type === "result") return ...;
return /* 当 error 处理 */;
```

这种结构 TS 不会警告"你漏了 warning 分支"——因为最后的 `return` 没有条件，TS 认为是"兜底"，能处理所有剩下的类型。如果新加 `WarningEvent` 进 union 但忘了在这里加 `if (event.type === "warning")`，warning 会落到兜底分支被当 error 处理。

```typescript
// ✅ 更安全写法（用 exhaustive check）
switch (event.type) {
  case "progress": return ...;
  case "result": return ...;
  case "error": return ...;
  case "warning": return ...;
  default:
    // ⭐ never 类型检查：如果 union 加了新成员但没加 case，
    //    event 在 default 分支里类型是 never，TS 报错
    const _exhaustive: never = event;
    return _exhaustive;
}
```

本次改动用的是 if 链而不是 switch，所以没有这个保护。这是个轻量权衡——if 链读起来更接近自然语言，代价是漏分支不会被 TS 抓到。如果未来事件类型继续增加（比如加 `info` / `debug`），建议改成 switch + exhaustive check。

1. **分页是纯客户端，数据已经全量到前端**：

   - `App.tsx:177` `existing.includes(event.message)` 假设后端 warning 文案是确定性的。

5. **truncated 提示是双保险但可能视觉重复**：

   - 如果未来觉得视觉冗余，可以在 `App.tsx` 的 result 分支里做去重（truncated=true 时从 warnings 里过滤掉相似文案）。

6. **`max_rows` 没带到前端 state**：
   - `ChatMessage` 只有 `truncated?: boolean`，没有 `max_rows`。
   - `MessageBubble.tsx:141` 的 "5000" 是硬编码。
   - 后端调整上限时前端要同步改文案。未来可以加 `max_rows` 字段动态渲染。
