## 一、背景

Grill-me 审查发现两个 AI 层痛点：

| 刀 | 痛点 | 影响 |
|---|---|---|
| Knife 3 | 召回阶段串行循环，3-5s 延迟 | 用户体感慢 |
| Knife 4 | 全部 prompt 为 Zero-shot，无示例 | SQL 生成准确率低，JOIN 策略靠 LLM 猜 |

本笔记记录改造方案与实现细节。

## 二、召回并行化（asyncio.gather）

### 2.1 问题定位

三个召回节点（`recall_column.py` / `recall_value.py` / `recall_metric.py`）的原始逻辑：

```python
# 原始：串行循环
for keyword in keywords:
    embedding = await embedding_client.aembed_query(keyword)
    results = await repository.search(embedding)
    # ... 累加结果
```

N 个关键词串行执行，每个关键词包含网络 IO（embedding 调用 + Qdrant/ES 检索），总耗时 = N × T。

### 2.2 改造方案

用 `asyncio.gather` 将 N 个关键词的检索任务并行化：

```python
async def _search_one_keyword(keyword: str) -> list[ColumnInfo]:
    """单个关键词的完整检索流程"""
    embedding = await embedding_client.aembed_query(keyword)
    return await column_qdrant_repository.search(embedding)

# 并行执行 + 故障容忍
tasks = [_search_one_keyword(kw) for kw in keywords]
all_results = await asyncio.gather(*tasks, return_exceptions=True)

for result_item in all_results:
    if isinstance(result_item, Exception):
        logger.warning(f"[recall_column] 关键词检索失败（跳过）: {result_item}")
        continue
    for column_info in result_item:
        if column_info.id not in column_info_map:
            column_info_map[column_info.id] = column_info
```

### 2.3 关键设计决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 并行方式 | `asyncio.gather` | IO 密集型任务，asyncio 天然适合 |
| 故障容忍 | `return_exceptions=True` | 单个关键词失败不应中断整个召回 |
| 去重 | `id not in map` | 并行结果需按 column_id 去重，与原逻辑一致 |
| 内部函数 | `_search_one_keyword` | 封装 embedding + search 两步为原子任务 |

### 2.4 预期效果

- N 个关键词的召回耗时从 N×T 降为 ~T（理论值）
- 实际受 Qdrant/ES 并发连接数限制，但量级提升明显
- 单个关键词失败不会导致整个召回为空（降级而非失败）

### 2.5 三节点统一模式

三个召回节点改造方式完全一致，只差在调用的 repository 不同：

| 节点 | 检索方式 | Repository |
|---|---|---|
| `recall_column.py` | Qdrant 向量检索 | `column_qdrant_repository` |
| `recall_value.py` | ES 全文检索 | `value_es_repository` |
| `recall_metric.py` | Qdrant 向量检索 | `metric_qdrant_repository` |

## 三、Prompt Few-shot 改造

### 3.1 改造原则

1. **示例必须使用项目真实 schema**：dim_region / dim_customer / dim_product / dim_date / fact_order + 指标 GMV/AOV
2. **示例覆盖递进难度**：从简单到复杂，引导 LLM 学习不同的 SQL 模式
3. **示例统一用一个场景**：关键词扩展示例统一用"华南地区黄金会员在今年Q1的GMV"，让三层召回的输出形成互补
4. **不破坏原有规则**：few-shot 放在规则之后、`{query}` 占位符之前

### 3.2 generate_sql.prompt — 3 个 few-shot

| 示例 | 覆盖模式 | 关键 SQL 元素 |
|---|---|---|
| 示例一 | 单表聚合 + 时间过滤 | `SUM` + `JOIN dim_date` + `WHERE year/month` |
| 示例二 | 多表 JOIN + 分组聚合 | `JOIN dim_product` + `GROUP BY category` + `ORDER BY DESC` |
| 示例三 | 多维筛选 + 指标口径 | 3 表 JOIN + `region_name` + `member_level` + GMV 口径 |

每个示例都包含：
- 用户查询（自然语言）
- SQL 输出（纯文本，无 Markdown 包裹）
- AS 别名使用中文业务名称（符合 prompt 规则第 7 条）

### 3.3 correct_sql.prompt — 1 个修正示例

场景：`category` 字段在 `fact_order` 中不存在，属于 `dim_product` 表

```
待纠正 SQL: SELECT category, SUM(order_quantity) FROM fact_order GROUP BY category
错误信息: Unknown column 'category' in 'field list'
纠正后的 SQL: SELECT dim_product.category AS 商品品类, SUM(fact_order.order_quantity) AS 总销量 
              FROM fact_order JOIN dim_product ON fact_order.product_id = dim_product.product_id 
              GROUP BY dim_product.category
```

附修正说明，引导 LLM 理解"最小修改"原则：只补 JOIN，不改语义。

### 3.4 3 个关键词扩展 prompt — HR → 电商

| Prompt | 原示例（HR） | 新示例（电商） |
|---|---|---|
| `extend_keywords_for_column_recall` | 员工身份类型/在职状态/转正状态... | 会员等级/地区/大区/年份/季度/成交金额/销售额/订单金额 |
| `extend_keywords_for_value_recall` | 在职/实习生/转正/最近三个月 | 华南/黄金/2025/Q1/GMV |
| `extend_keywords_for_metric_recall` | 转正人数/转正率/转正情况... | GMV/成交总额/订单总额/销售额/成交金额 |

为什么三个 prompt 用同一个问句？
- 三层召回分别抽取：字段名、取值、指标名
- 同一句话拆出三层关键词，输出互补，覆盖完整召回链路
