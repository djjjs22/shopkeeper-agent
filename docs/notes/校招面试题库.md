## 零、面试核心故事（必背）

> 每个故事 30 秒讲完，面试官追问时再展开细节。
> 原则：先讲结论（一句话），再讲过程（两三句），最后讲收获（一句话）。

### 故事1：「一句话 → SQL」

**主动讲**：项目做了什么

> 我做了一个系统，用户用大白话问问题，系统自动查数据库返回结果。
> 比如用户问"华东卖了多少货"，系统就去查表、算数字，返回答案。

**面试官追问："怎么把大白话变成查表语句？"**

> 分三步。第一步，从用户的话里提关键词——"华东"是地名，"卖了多少"是金额。
> 第二步，去数据库里找——哪个表里存了地名？哪个字段存了金额？这一步我用了三种找法同时进行：一种找意思相近的（比如用户说销售额，系统知道对应的是金额字段），一种找名字一样的（华东直接匹配到地区表），还有一种找业务规则（比如什么叫GMV）。
> 第三步，把找到的表名和字段名，跟用户的问题一起，交给AI生成查表语句。

**追问升级："具体写了什么代码？"**

> 三种找法对应三块代码。第一种意思相近的，用的是向量搜索——先把字段名转成一串数字，然后比较哪两个数字串最像。第二种名字一样的，用的是关键词搜索，直接去数据库里找有没有叫"华东"的值。第三种业务规则，是提前在配置文件里写好的——比如GMV这个指标定义成"订单金额的求和"。
>
> 三块代码各自独立，同时执行，结果全部拿到后再合并。这样不会因为一个慢就拖累另外两个。

### 故事2：「包升级崩了怎么办」

**主动讲**：遇到的棘手问题

> 有一次我把项目依赖的一个包升级了，结果直接报错，整个文字转数字的功能崩了。
> 原因是这个包升级后不再支持我自己部署的服务了，它现在只认它自己的平台。我写了一段替代代码，用自己的方式去连我的服务，绕过了这个限制。
> 这件事让我明白，用别人的代码时，最好在关键地方包一层自己的东西。

**面试官追问："什么叫包一层自己的东西？"**

> 就是我不直接拿别人的代码去连我的服务，而是自己写一个中间层。这样将来别人的代码再怎么变，我只需要改中间层这一小块，项目其他地方完全不受影响。就像给水管装了个活接头——换水管的时候，不用把墙砸了。

**追问升级："你怎么排查出是升级导致的？"**

> 我先看报错信息——它说参数不对，应该是仓库名但收到了网址。然后我去查了这个包的更新记录，发现新版本确实改了规则。确认原因后，我立刻想到先回滚到旧版本让服务恢复，再慢慢想修复方案。实际上回滚就是一行命令把版本号改回去。

### 故事3：「AI 写的语句有坑，怎么防」

**主动讲**：怎么保证质量

> AI 生成的SQL不靠谱。有时候它会写删除数据的危险语句，有时候会在外面包一层多余的符号导致数据库看不懂。我加了检查——执行前先拦一遍，有危险词就直接拒绝。

**面试官追问："怎么拦？"**

> 两层。第一层是语法检查——先让数据库预演一下，只看语句能不能跑通，不真的执行。跑不通就让AI改。第二层是安全检查——真正执行前，再看一遍有没有删除、修改这类词。有就直接告诉用户"这个不允许"。

**追问升级："具体代码怎么写的？"**

> 安全检查我写了大概几十行。核心就三件事：第一，查语句里有没有"删除""修改"这些危险词，有就挡住。第二，查是不是只允许"查询"操作——不是查询的直接拒绝。第三，查有没有被人注入恶意代码的痕迹。为了防止正常语句被误判——比如字段名叫"已删除的记录"里面有"删除"二字——我还加了处理：先把语句里引号包着的内容去掉，再检查剩下的。

**追问升级："你怎么验证这个检查是有效的？"**

> 我写了20个测试用例。正常的查询语句跑一遍确认不会误拦，故意的删除语句、修改语句也跑一遍确认一定被拦。测试的时候还真发现了一个问题——一句正常的查询"选择叫DROP的标签"被误判成了删除操作。原因是我的检查没区分引号里的内容和真正的命令字。修复后就全通过了。

## 二、多路召回与检索

### Q4 ⭐⭐ 三路召回分别是什么？为什么不能只用一路？

**问题**: 项目里用了哪三路召回？各自走什么存储、解决什么问题？

**参考答案**:

| 路数 | 存储 | 技术 | 解决的问题 | 举例 |
|------|------|------|-----------|------|
| 字段召回 | Qdrant | 向量语义搜索 | 用户说"销售额"→匹配到 `order_amount` | 抽象概念匹配 |
| 指标召回 | Qdrant | 向量语义搜索(不同集合) | "GMV"→匹配到 `SUM(order_amount)` | 业务口径匹配 |
| 取值召回 | Elasticsearch | IK 中文全文搜索 | "华东"→精确找到字段值 | 枚举值精确匹配 |

**为什么不能只用一路**：
- 只用向量检索 → 枚举值"华东""华南"语义太近，会乱匹配
- 只用全文检索 → "销售额"这种抽象概念在 ES 里搜不到，ES 只存了具体取值
- 字段和指标分两个 Qdrant 集合 → 避免"销售额"既命中字段又命中指标时混在一起

---

### Q5 ⭐⭐ Qdrant 里存了什么？向量是怎么生成的？

**参考答案**:
- Qdrant 存的是字段名+描述+别名拼接后的向量（1024 维）
- 向量由 TEI 服务（text-embeddings-inference）加载 bge-large-zh-v1.5 模型生成
- 构建时：`字段信息文本 → POST /embed → 1024维向量 → upsert 到 Qdrant`
- 查询时：`用户问题 → POST /embed → 向量 → Qdrant.query_points() → 最相似的字段`

---

### Q6 ⭐⭐⭐ 向量召回的 score_threshold 设多少？为什么？

**参考答案**: 0.6（余弦相似度）。低于 0.6 的结果与用户问题相关性太低，纳入上下文反而干扰 LLM 判断。

## 三、SQL 生成与执行

### Q9 ⭐⭐ 从用户问题到 SQL 执行的完整流程是怎样的？

**参考答案**:

```
用户问题
  → 关键词提取（Jieba 分词）
  → 三路并行召回（字段/取值/指标）
  → 合并召回结果 + 按 ID 补齐元数据
  → LLM 过滤相关表和指标
  → 补充日期/数据库环境等额外上下文
  → LLM 生成 SQL（Prompt 含表结构、字段、指标口径）
  → EXPLAIN 语法校验
  → 失败 → LLM 修正 SQL + 错误信息 → 再校验
  → 通过 → SQL 安全防火墙检查 → MySQL 执行 → 流式返回结果
```

---

### Q10 ⭐⭐ LLM 生成 SQL 时的 Prompt 里包含什么？

**参考答案**: 用 YAML 结构化传入：
- `table_infos`：召回的表名、字段名、类型、描述
- `metric_infos`：业务指标定义（如 GMV=SUM(order_amount)）
- `date_info`：当前时间信息（帮助 LLM 判断"上个月"是几月）
- `db_info`：数据库方言和版本（MySQL 8.0）

---

### Q11 ⭐⭐⭐ 你加了什么安全措施？为什么？

**参考答案**:
- 新增 `SQLSafetyValidator` 类，三层防护：
  1. 关键字黑名单（正则 `\b` 词边界匹配，拦截 DROP/DELETE/UPDATE 等 13 个关键字）
  2. SELECT 白名单（`.startswith("SELECT")`，只允许查询）
  3. SQL 注入检测（UNION SELECT / OR '1'='1' / -- 注释截断）
- 测试中发现的 Bug：`SELECT 'DROP' AS label` 被误杀，修复为"先移除字符串字面量再匹配"
- 20 个单元测试覆盖所有场景，ruff 代码规范检查通过

---

### Q12 ⭐⭐ EXPLAIN 校验能发现什么问题？不能发现什么？

**参考答案**:
- ✅ 能发现：SQL 语法错误、表不存在、字段名拼错
- ❌ 不能发现：字段放错表（`dim_product.date_id`）、JOIN 条件错误、笛卡尔积、业务逻辑错误

## 四、数据仓库与元数据

### Q15 ⭐⭐ 项目的数仓模型是怎样的？

**参考答案**:
- 星型模型：1 张事实表 `fact_order` + 4 张维度表
- `dim_region`（地区）、`dim_customer`（客户）、`dim_product`（商品）、`dim_date`（时间）
- 事实表存外键 + 度量值（order_quantity、order_amount）

---

### Q16 ⭐⭐ 元数据知识库是怎么构建的？

**参考答案**:
- 从 `conf/meta_config.yaml` 读取表/字段/指标定义
- 对每个字段生成向量 → 写入 Qdrant
- 对每个字段的真实取值 → 写入 Elasticsearch
- 结构化元数据（表结构、类型、主外键）→ 写入 MySQL meta 库
- 执行命令：`uv run python -m app.scripts.build_meta_knowledge -c conf/meta_config.yaml`

## 五、数据开发与工程化

### Q21 ⭐⭐ 项目用什么管理 Python 依赖？和 pip 有什么区别？

**参考答案**:
- 使用 `uv`（Rust 写的超快 Python 包管理器）
- `uv sync` = `pip install -r requirements.txt`，但快 10-100 倍
- `uv add --dev pytest` = 添加开发依赖
- 优势：有锁文件（uv.lock）、虚拟环境自动管理

---

### Q22 ⭐⭐ 你是怎么写单元测试的？用了什么框架？

**参考答案**:
- 框架：pytest（Python 最流行的测试框架）
- 配置：`conftest.py` 设置 `sys.path`
- 正向测试：用 `assert` 验证正确结果
- 负向测试：用 `pytest.raises(ValueError, match="...")` 验证异常
- 运行：`uv run pytest tests/ -v`
- 规范：pytest 命名规则 `test_*.py` + 类名 `Test*` + 方法名 `test_*`

---

### Q23 ⭐⭐ 代码规范检查用什么？

**参考答案**:
- `ruff` — Python 代码检查和自动修复工具
- `ruff check app/ tests/` 检查问题
- `ruff check app/ tests/ --fix` 自动修复
- 常见问题：import 顺序（I001）、未使用的变量（F841）、未使用的导入（F401）
