|--------|----------------------|----------|

| `\s*` / `\s+` | 零个或多个 / 一个或多个空白字符。`SLEEP\s*\(` 同时覆盖 `SLEEP(`、`SLEEP (`、`SLEEP\n(` | `sql_safety.py:157-158, 172` |
| `re.sub(r"/\*.*?\*/", " ", s)` | 用空格替换块注释。**不用空串**替换是为了保留单词边界（否则 `UNION/**/SELECT` 会变成 `UNIONSELECT`，反而绕过 `\bUNION\b\s+SELECT`） | `sql_safety.py:274` |

| `(rows, truncated) = await ...` | Python 多返回值解包。函数返回一个 tuple，调用方用两个变量分别接 | `run_sql.py:298` |
| `set(a) \| set(b)` / `set(a) - set(b)` | 集合的并集 / 差集。表名白名单合并 CTE 名 / 找出"引用了但不在白名单"的表 | `sql_safety.py:356, 358` |

### #1 错误信息脱敏

#### 为什么（信息泄露风险）

数据库异常原文是**最容易被忽略的泄露面**。一条 MySQL 异常可能长这样：

```
(pymysql.err.ProgrammingError) (1146, "Table 'dw.fact_orde' doesn't exist")
[SQL: SELECT region_id, SUM(order_amount) FROM fact_orde GROUP BY region_id]
```

这一条异常信息里同时泄露了：

1. **真实库名** `dw`（攻击者知道了数据仓叫 `dw`，下一步可以直接构造 `dw.dim_customer`）
2. **错误猜测的表名** `fact_orde`（反过来告诉攻击者：正确表名很可能叫 `fact_order`）
3. **列名** `region_id` / `order_amount`（暴露字段语义，等于把度量定义送给对方）
4. **SQL 上下文**（GROUP BY 维度、聚合方式，暴露数据仓的指标口径）

更严重的是这种泄露是**可被攻击者主动触发**的：用户在前端故意构造一个问错的 query，让 LLM 生成一条"故意写错表名"的 SQL，数据库回一句 `Table 'xxx' doesn't exist`，攻击者就能逐字符探测 schema。

原代码的写法：

```python
except Exception as e:
    logger.error(f"[SQL安全] 执行失败: {e}")
    writer({"type": "error", "message": str(e)})  # ⚠️ 直接透传数据库异常原文
    raise
```

`str(e)` 把整个异常对象（含上面的全部细节）原样推到 SSE 流里，前端直接渲染给用户看。攻击者根本不用注入，正常 query 就能拿信息。

#### 怎么做（分类映射 + 服务端完整日志）

策略分三层：

1. **服务端 `logger.error` / `logger.exception` 记完整异常**（含 SQL、表名、堆栈），用于内部排查
2. **给前端的 message 走分类映射表**，把已知的异常特征映射成不暴露细节的中文文案
3. **前端再兜一层**：万一后端漏改了某个出口（push 了原始异常），前端的 `mapBackendMessage` 也不再把原文透出来，统一改成"已记录日志，请换一种问法"

为什么用"分类映射"而不是"统一一句话"？

- **用户体验**：用户看到"查询的表不存在，请确认表名或换一种问法"比看到"查询执行失败"更知道下一步该做什么
- **不泄露**：友好文案里没有表名、列名、SQL 上下文
- **顺序敏感**：更具体的特征（如 `unknown column`）放在更宽泛的特征（如 `doesn't exist`）前面，避免被错误归类

**`run_sql.py` 新增的映射表和分类函数（`app/agent/nodes/run_sql.py:113-156`）**：

```python
# ─────────────────────────────────────────────────────────────────────
# 错误信息脱敏（2026-07-20 #1 安全加固）
# ─────────────────────────────────────────────────────────────────────
# 原 run_sql 直接 `writer({"type": "error", "message": str(e)})` 会把 MySQL /
# SQLAlchemy 异常原文（含表名、列名、SQL 片段、连接信息）泄露给前端。
# 改成分类映射，给前端友好文案，完整异常服务端 logger 已记。

# 后端错误特征 → 友好文案 的映射表
# 顺序敏感：更具体的特征放在前面
_BACKEND_ERROR_MAP: list[tuple[str, str]] = [
    # 表不存在（具体的几种 MySQL 报错文本，逐一映射）
    ("table", "查询的表不存在，请确认表名或换一种问法"),
    # ↑ 匹配 "Table 'xxx' doesn't exist" 之类
    ("unknown column", "查询的字段名有误，请换一种问法描述"),
    # ↑ 匹配 "Unknown column 'xxx' in 'field list'"
    ("doesn't exist", "查询的对象不存在，请换一种问法"),
    # ↑ 兜底各种 doesn't exist（视图、库等）

    # 语法错误
    ("syntax", "SQL 语法错误，已记录，请重试或换一种问法"),
    ("you have an error in your sql syntax", "SQL 语法错误，已记录，请重试"),
    # ↑ MySQL 经典报错："You have an error in your SQL syntax"

    # 字段引用歧义（JOIN 多表时列名重复）
    ("ambiguous", "查询字段有歧义，请更精确地描述"),

    # 连接 / 超时
    ("connection", "数据库连接异常，请稍后重试"),
    ("timeout", "查询超时，请缩小查询范围或换更精确的条件"),

    # 权限
    ("access denied", "无权限执行此查询"),

    # 内存（兜底 #3 的 OOM 场景）
    ("out of memory", "查询结果过大，请加更精确的筛选条件"),
]


def _safe_error_message(exc: Exception) -> str:
    """把数据库异常分类成对前端安全的简短文案

    Args:
        exc: 数据库执行抛出的异常

    Returns:
        前端可展示的中文文案（不包含 SQL / 表名 / 列名等敏感信息）
    """
    # str(exc) 把异常对象转字符串，.lower() 小写化便于子串匹配（大小写不敏感）
    msg = str(exc).lower()
    # 遍历映射表，第一个命中的特征就返回（所以顺序敏感：具体在前）
    for pattern, friendly in _BACKEND_ERROR_MAP:
        if pattern in msg:  # 子串匹配，不用正则，更快也更可控
            return friendly
    # 兜底：完全不匹配任何已知特征时，不透传原文，给通用文案
    return "查询执行失败，已记录日志，请换一种问法或缩小查询范围"
```

**`query_service.py` 也加了一份映射表（`app/services/query_service.py:34-50`）**：

```python
# ─────────────────────────────────────────────────────────────────────
# 错误脱敏（2026-07-20 #1）：给前端的友好文案，避免泄露 SQL/表结构
# ─────────────────────────────────────────────────────────────────────
# 注意：服务层的映射表和 run_sql 不同——服务层更上层，异常类型不一样。
# 这里聚焦在"链路异常"（连接、限流、内存），而 run_sql 聚焦在"SQL 异常"。
_SERVICE_ERROR_MAP: list[tuple[str, str]] = [
    ("timeout", "请求超时，请缩小查询范围或换更精确的条件"),
    ("connection", "服务连接异常，请稍后重试"),
    ("rate", "请求过于频繁，请稍后再试"),  # ↑ 限流（rate_limit 中间件触发）
    ("memory", "查询结果过大，请加更精确的筛选条件"),
]


def _friendly_error(exc: Exception) -> str:
    """服务层异常 → 前端友好文案（不透传 str(exc) 原文）"""
    msg = str(exc).lower()
    for pattern, friendly in _SERVICE_ERROR_MAP:
        if pattern in msg:
            return friendly
    return "处理失败，已记录日志，请换一种问法或稍后重试"
```

`query_service.py` 在两个出口（`query` 和 `query_multi_agent`）的修改（`app/services/query_service.py:117-119, 193-195`）：

```python
        except Exception as e:
            # 2026-07-20（#1 脱敏）：服务端记完整异常，给前端友好文案
            # ↑ logger.exception 比 logger.error 多打一条完整 traceback
            logger.exception(f"[query_multi_agent] 链路异常: {e}")
            error = {"type": "error", "message": _friendly_error(e)}
            yield f"data: {json.dumps(error, ensure_ascii=False, default=str)}\n\n"
```

注意 `logger.exception` vs `logger.error` 的区别：
- `logger.error("xxx")` 只记录一行字符串
- `logger.exception("xxx")` 除了记字符串，还会自动附上当前异常的完整 traceback（堆栈）

链路层用 `exception` 是因为这类异常往往不是 SQL 本身的问题（可能是 ES 连不上、Redis 挂了、LLM 超时），需要堆栈才能定位。

#### 前端配合改动

`frontend/src/lib/errorMessages.ts:170-190` 的 `mapBackendMessage`：

```typescript
/**
 * 把后端 SSE error 事件的 message 转友好错误
 * 优先关键字匹配，否则用通用兜底（不再透传原文，防敏感信息泄露）
 *
 * 2026-07-20 (#1 脱敏)：后端已经在 query_service / run_sql 统一脱敏，
 * 这里再兜一层：万一后端遗漏，前端也不会把 SQL / 表名 / 列名显示给用户。
 */
export function mapBackendMessage(message: string): FriendlyError {
  for (const { pattern, error } of BACKEND_KEYWORD_MAP) {
    if (pattern.test(message)) {
      return error;
    }
  }
  // 没匹配到关键字 → 通用兜底（不透传原文，避免敏感信息泄露）
  // ↑ 改动前是 detail: message || "..."，会把后端原文显示出来
  return {
    title: "查询失败",
    detail: "已记录日志，请换一种问法或稍后重试。",
    toastType: "error",
    retryable: true,
  };
}
```

为什么前端也要再兜一层？因为：

- 后端可能有多个出口（`run_sql` 节点出口、`query_service` 两个出口、未来可能新增的节点出口）
- 万一某个出口忘记走 `_safe_error_message`，前端就成了"最后一道关"
- 双层防护：后端漏一个，前端兜底；前端漏了，至少后端已经脱敏过

#### 验证方式

手动验证（在测试环境跑）：

1. 构造一个会触发"表不存在"的 query，比如问"查询 fact_orde 表"（故意少一个字母）
2. 看前端 MessageBubble 显示的应该是 `查询的表不存在，请确认表名或换一种问法`
3. 看后端日志（`logs/` 下）应该是完整的 `(pymysql.err.ProgrammingError) (1146, "Table 'dw.fact_orde' doesn't exist")`
4. 在浏览器 DevTools 的 Network 面板看 SSE 流，`error` 事件的 `message` 字段应该是脱敏后的文案，**不应该出现 `fact_orde` 字样**

边界 case 验证：

- 触发一个未在映射表里的异常（比如手动 raise 一个 `RuntimeError("unexpected")`），确认前端显示的是通用兜底文案 `查询执行失败，已记录日志，请换一种问法或缩小查询范围`，而不是 `unexpected`

### #2 SQL 防火墙补 4 类盲区

#### 为什么（攻击面分析）

初版三层防护（关键字 / SELECT 白名单 / 注入特征）在 2026-06-29 上线后，跑了一段时间发现还有 5 类攻击能直接穿过去。下面每类配一个**真实可执行的攻击 SQL**（基于本项目白名单里的 `dim_region` / `fact_order` 等表构造）。

**盲区 1：时间盲注 + 文件读写**

时间盲注的攻击原理：当数据库不直接回显数据（比如当前查询被包在子查询里、或者权限很受限）时，攻击者用"条件 + SLEEP"来通过响应时间反推数据。

```sql
-- 攻击示例 1：时间盲注，逐字符猜当前 mysql 用户
SELECT * FROM dim_region
WHERE region_id = 1
  AND IF(ASCII(SUBSTRING((SELECT user FROM mysql.user LIMIT 1), 1, 1)) > 100,
         SLEEP(5), 0)
-- 如果当前用户名首字符的 ASCII > 100，响应会卡 5 秒；否则秒回。
-- 攻击者用二分法几次就能确定一个字符，循环就能拿到整个用户名字符串。
```

```sql
-- 攻击示例 2：把整张 fact_order 写到服务器可读文件，再通过其他途径（如 LOAD_FILE）读回
SELECT * FROM fact_order INTO OUTFILE '/tmp/leak.csv'
-- 数据直接落地成文件。如果 MySQL 服务账户和 Web 服务账户有共享目录，
-- 攻击者接下来用 LOAD_FILE('/tmp/leak.csv') 或直接 HTTP 访问就能拿走全部数据。
```

```sql
-- 攻击示例 3：读服务器敏感文件
SELECT LOAD_FILE('/etc/passwd') FROM dim_region
-- 直接把 Linux 的 /etc/passwd 内容当作查询结果返回。
```

原 `FORBIDDEN_KEYWORDS` 里只有 `LOAD`、`IMPORT` 这种粗粒度关键字，`SLEEP`、`BENCHMARK`、`OUTFILE`、`DUMPFILE`、`LOAD_FILE` 一个都没拦。

**盲区 2：系统表 / 系统库**

MySQL 自带一批元数据表，能拿到整个数据库的结构、用户、权限、性能统计。

```sql
-- 攻击示例：直接拿所有表名（等于把 schema 拱手送人）
SELECT table_name FROM information_schema.tables
WHERE table_schema = 'dw'
-- 返回 dw 库下所有表名：dim_region / dim_customer / dim_product / dim_date / fact_order
-- 攻击者拿到表名清单后，下一步直接 SELECT * FROM information_schema.columns
-- 就能拿到每张表的所有列名和类型，整个数据仓的设计完全暴露。
```

```sql
-- 攻击示例：拿数据库账户列表（暴力破解的前置情报）
SELECT user, host, authentication_string FROM mysql.user
-- mysql.user 表存着所有数据库账户和密码哈希。
-- 如果 MySQL 服务被错误配置允许该应用账户读 mysql 库，密码哈希就会被泄露。
```

其他系统库同理：`performance_schema.threads`（看其他连接在执行什么 SQL）、`sys.schema_table_statistics`（看每张表的访问频率，反推哪些表重要）。

原代码完全没有针对系统表的拦截规则。

**盲区 3：漏拦的写操作关键字**

`CALL` / `HANDLER` / `DO` / `SET` 这几个 MySQL 语句在原 `FORBIDDEN_KEYWORDS` 里没有：

```sql
-- 攻击示例 1：CALL 调用存储过程（存储过程里可能藏着写操作或 DDL）
CALL p_archive_old_orders()
-- 如果数据库里有一个 p_archive_old_orders 存储过程，
-- 它内部可能是 DELETE FROM fact_order WHERE order_date < '2024-01-01'。
-- CALL 直接绕过了关键字层，把数据删了。
```

```sql
-- 攻击示例 2：HANDLER 直接读表（绕过权限检查）
HANDLER dim_customer OPEN;
HANDLER dim_customer READ FIRST;
-- HANDLER 是 MySQL 的底层存储接口，能绕过部分 SELECT 权限检查。
-- 攻击者可以借此读到理论上无权访问的行。
```

```sql
-- 攻击示例 3：DO 执行表达式（包括 SLEEP）
DO SLEEP(30)
-- DO 单独执行一个表达式，不返回结果。
-- 攻击者用 DO SLEEP(30) 做时间盲注，比 SELECT ... SLEEP 更隐蔽。
```

```sql
-- 攻击示例 4：SET 修改 session 变量（影响后续查询行为）
SET sql_mode = ''
-- 改了 sql_mode 后，后续查询的严格性会变化，可能绕过某些约束。
-- 或者 SET @@autocommit = 0 配合后续操作制造事务副作用。
```

**盲区 4：UNION SELECT 被块注释分割绕过**

这是 4 类盲区里最阴险的一个。原注入检测里有这条正则：

```python
r"UNION\s+SELECT",  # 期望拦住 "UNION SELECT"
```

`\s+` 只匹配空白字符（空格、tab、换行）。但 MySQL 把 `/**/` 当成空白的等价物，所以：

```sql
-- 攻击示例：UNION/**/SELECT 用块注释绕过
SELECT region_id, region_name FROM dim_region WHERE region_id = 1
UNION/**/SELECT user, authentication_string FROM mysql.user
-- 原正则 r"UNION\s+SELECT" 匹配不上（因为 UNION 和 SELECT 之间是 /**/，不是空白）
-- 数据库执行时把 /**/ 丢掉，等价于 UNION SELECT，直接把 mysql.user 的内容合并出来
```

更恶心的变体：

```sql
-- 多个块注释分割
SELECT * FROM dim_region WHERE id=1 UNION/**/ /**/SELECT 1,2,3
-- 块注释里塞空格
SELECT * FROM dim_region WHERE id=1 UNION/* */SELECT 1,2,3
-- 大小写混合
SELECT * FROM dim_region WHERE id=1 UnIoN/**/SeLeCt 1,2,3
```

原代码不仅没有剥块注释的逻辑，而且 `UNION\s+SELECT` 这条正则也没有 `re.IGNORECASE`（虽然 `INJECTION_PATTERNS` 的循环用了 IGNORECASE，但 `\s+` 根本不匹配 `/**/`，flag 也救不了）。

**盲区 5：校验顺序不统一**

原来的 `validate()` 校验顺序是：空值 → 移除字面量 → 危险关键字 → SELECT/WITH 开头 → 注入检测。问题：

- 没有剥块注释，导致关键字检测、注入检测都能被 `/**/` 绕过
- 没有"表名白名单"这一层（虽然 `ALLOWED_TABLES` 类变量定义了 5 张表，但函数体里压根没用，等于形同虚设 —— 详见 CLAUDE.md 2026-07-20 那条"默认参数若只在 docstring 承诺、函数体没实现 = 埋坑"的笔记）

需要统一成新的顺序：**空值 → 移除字面量 → 剥块注释 → 危险关键字 → SELECT/WITH 开头 → 注入检测 → 表名白名单**。

#### 怎么做（5 类补丁 + 顺序统一）

逐条对应 5 类盲区：

**修盲区 1 + 修盲区 3**：补 `FORBIDDEN_KEYWORDS`，加 7 个写操作 / 危险操作关键字。

**修盲区 1**：再补 `INJECTION_PATTERNS`，加 `SLEEP` / `BENCHMARK` / `INTO OUTFILE` / `INTO DUMPFILE` / `LOAD_FILE` 这几个**作为函数调用 / 子句**的形态（关键字层只能拦"出现这个词"，注入层能拦"SLEEP 后面紧跟左括号"这种函数调用形态，更精确）。

**修盲区 2**：补 `INJECTION_PATTERNS`，加 `information_schema` / `mysql.` / `performance_schema` / `sys.` 这 4 条系统库匹配。

**修盲区 4**：在 `validate()` 里增加"剥块注释"这一步，用 `re.sub(r"/\*.*?\*/", " ", sql_no_strings, flags=re.DOTALL)` 把块注释替换成空格，再用剥完的版本做关键字和注入匹配。

**修盲区 5**：把校验顺序统一成"剥块注释 → 关键字 → SELECT/WITH → 注入 → 表名白名单"，并且把"表名白名单"这一层真正实现（`_extract_tables` + `_extract_cte_names`）。

新的校验顺序（完整流程）：

```
原始 SQL
  │
  ▼
[1] 空值检查（not sql.strip() → 拦截）
  │
  ▼
[2] sql_upper = sql.upper()  ← 全部大写化
  │
  ▼
[3] sql_no_strings = 移除单引号字面量  ← 防 SELECT 'DROP' AS label 被误杀
  │
  ▼
[4] sql_no_comments = 剥块注释（DOTALL，空格替换）← 防 UNION/**/SELECT 绕过 ★新增
  │
  ▼
[5] FORBIDDEN_KEYWORDS 匹配（在 sql_no_comments 上）← 现在能拦 UNION/**/CALL
  │
  ▼
[6] SELECT / WITH 开头校验
  │
  ▼
[7] INJECTION_PATTERNS 匹配（在原始 sql 上，IGNORECASE）
  │   ← 现在 SLEEP / information_schema / INTO OUTFILE 都能拦
  │
  ▼
[8] 表名白名单校验（在 sql_no_comments 上）  ← ★新增（之前形同虚设）
  │   - 抽 WITH 后的 CTE 名（视作合法临时表）
  │   - 抽 FROM / JOIN 后的表名
  │   - 任何一个不在 (白名单 ∪ CTE 名) 集合里就拦截
  │
  ▼
返回干净 SQL
```

为什么注入检测（步骤 7）必须**优先于**表名白名单（步骤 8）？因为：

- 如果白名单在前，`SELECT * FROM fact_order; DROP TABLE users JOIN xxx` 会被报成"非白名单表 xxx"
- 注入检测在前，能识别出 `; DROP` 是堆叠注入，给出更准确的"SQL 注入"分类
- 错误分类不准会让运维误判（看到"非白名单表"以为是 LLM 写错表名，实际是被注入了）

为什么 `OUTFILE` / `DUMPFILE` / `LOAD_FILE` 既加到 `FORBIDDEN_KEYWORDS` 又加到 `INJECTION_PATTERNS`？两层冗余：

- 关键字层只要"出现这个词"就拦，简单粗暴，覆盖广
- 注入层用更精确的正则（如 `\bINTO\s+(OUTFILE|DUMPFILE)\b`），能识别"作为子句"的形态，便于错误归类

**新增注入特征（`app/core/sql_safety.py:153-177`）**：

```python
INJECTION_PATTERNS: List[str] = [
    # ... 原有的 ' OR '1'='1 / UNION SELECT / ; DROP / -- / /* ... */ 等 ...

    # 2026-07-20 新增（#2 安全加固）：

    # 时间盲注：SLEEP() / BENCHMARK() 让数据库卡住，攻击者通过响应时间反推数据
    # 匹配示例: SELECT IF(1=1, SLEEP(30), 0)

    # 匹配示例: SELECT * FROM information_schema.tables
    #          SELECT * FROM mysql.user
    #          SELECT * FROM performance_schema.threads
    #          SELECT * FROM sys.schema_table_statistics
    r"\binformation_schema\b",
    #    ↑ 整个单词匹配 information_schema
    r"\bmysql\s*\.",

    r"\bperformance_schema\b",
    r"\bsys\s*\.",
    #    ↑ 同 mysql. 的写法，只拦 sys. 子句不拦 sys 单词

    # 文件读写（虽然 OUTFILE/DUMPFILE/LOAD_FILE 已在 FORBIDDEN_KEYWORDS，
    # 但 LLM 可能写成 INTO OUTFILE 形式，独立正则兜底）
    r"\bINTO\s+(OUTFILE|DUMPFILE)\b",
    #    ↑ 匹配 INTO OUTFILE / INTO DUMPFILE 两种形态
    #    ↑ (OUTFILE|DUMPFILE) 是正则的"或"分组
    r"\bLOAD_FILE\s*\(",
    #    ↑ 匹配 LOAD_FILE('/etc/passwd') 这种函数调用形态

    # 不在白名单的系统函数（信息泄露/资源消耗）
    r"\bLOAD_FILE\s*\(",
    # ↑ 这条和上面重复了，是 diff 里的一个冗余（不影响功能，因为列表匹配只要命中一条就拦；
    #   但属于可清理的小瑕疵，后续可以删掉）

```

为什么系统库的匹配用 `\bmysql\s*\.` 而不是 `\bmysql\b`？因为：

- 只想拦"`mysql` 库的表引用"（`mysql.user`）
- 不想拦"`mysql` 这个字段名"（如果某张业务表里有个字段叫 `mysql_version`，`\bmysql\b` 会误杀）
- `mysql.` 这个组合（单词 + 点）才表明这是 schema.table 引用

**剥块注释（`app/core/sql_safety.py:267-274`）**：

```python
# 2026-07-20 新增（#2 安全加固）：剥块注释后再做关键字/注入匹配
# 防止 `UNION/**/SELECT`、`DROP/**/TABLE` 这种用注释分割绕过检测。
# 用空格替换（不是空串），保留关键字边界。
# 注意：剥注释后 sql_no_strings 里就不再有块注释，所以 INJECTION_PATTERNS 里
# 的 r"/\*.*\*/" 不会再匹配（这正好对——剥掉之后还匹配说明 SQL 本来就没块注释，
# 是正常情况；INJECTION_PATTERNS 那条规则用于"在剥之前"已经拦住可疑 SQL，
# 现在策略改成"剥之后用更严的关键字/注入规则"，更彻底）。
sql_no_comments = re.sub(r"/\*.*?\*/", " ", sql_no_strings, flags=re.DOTALL)
```

正则 `r"/\*.*?\*/"` 详解：

- `/\*` 匹配 `/*`（`\` 是转义，因为 `*` 在正则里有特殊含义）
- `.*?` 非贪婪匹配任意字符。**非贪婪**很关键：
  - 输入 `/* a */ b /* c */`
  - 贪婪 `.*` 会匹配 `a */ b /* c`（把中间的 `b` 也吞掉）
  - 非贪婪 `.*?` 会分别匹配 `a` 和 `c`，保留中间的 `b`
- `\*/` 匹配 `*/`
- `flags=re.DOTALL` 让 `.` 也匹配换行符。**没有这个 flag 的话，跨行块注释剥不掉**：

```sql
SELECT * FROM dim_region /*
  这是个跨行注释
*/ WHERE id = 1 UNION SELECT password FROM mysql.user
```

不加 DOTALL 时 `.*` 默认不匹配 `\n`，整段跨行注释会逃过剥除。

为什么用空格 `" "` 替换而不是空串 `""`？

```python
# 假设输入是 "UNION/**/SELECT"
re.sub(r"/\*.*?\*/", "", "UNION/**/SELECT")  # → "UNIONSELECT"
re.sub(r"/\*.*?\*/", " ", "UNION/**/SELECT")  # → "UNION SELECT"
```

如果替换成空串，`UNION/**/SELECT` 会变成 `UNIONSELECT`，反而绕过 `\bUNION\b\s+SELECT` 正则（因为 `\b` 词边界失效）。用空格替换保留单词边界，让后续正则能正常命中。

**新增表名白名单校验（`app/core/sql_safety.py:331-363`）**：

```python
# ═══════════════════════════════════════════════════════════
# 第三层半：表名白名单校验（2026-07-20 修复：之前形同虚设）
# ═══════════════════════════════════════════════════════════
# 从 FROM / JOIN 后提取表名（已移除字符串字面量，防止 'FROM' 之类字面量误命中），
# 大小写不敏感比对白名单；任何一个不在白名单即拦截。
#
# CTE 名（WITH temp AS (...)）作为临时表被 FROM 引用是合法的，
# 抽取后动态并入白名单，避免误杀。
#
# 优先级（2026-07-20 #9 动态加载）：
#   1. 调用方显式传 allowed_tables
#   2. 启动时从 meta_db 加载的 _DYNAMIC_ALLOWED_TABLES（如有）
#   3. 类变量 ALLOWED_TABLES 硬编码兜底
# 传空列表 [] 则跳过校验（用于单元测试或非生产环境临时关闭白名单）。
if allowed_tables is not None:
    # 调用方显式传白名单 → 用调用方的（测试场景常用）
    base_whitelist = [t.upper() for t in allowed_tables]
elif cls._DYNAMIC_ALLOWED_TABLES:
    # 启动时从 meta_db 加载过动态白名单 → 用动态的（生产环境）
    base_whitelist = list(cls._DYNAMIC_ALLOWED_TABLES)
else:
    # 都没有 → 兜底用类变量硬编码的 5 张表
    base_whitelist = [t.upper() for t in cls.ALLOWED_TABLES]
effective_whitelist = base_whitelist
if effective_whitelist:
    # ↑ 空列表 [] 进不来这里，等于跳过校验（测试场景用）
    cte_names = cls._extract_cte_names(sql_no_comments)
    referenced_tables = cls._extract_tables(sql_no_comments)
    # CTE 名视作"当前 SQL 内合法引用"，并入白名单
    # ↑ 比如 WITH temp AS (...) SELECT * FROM temp 里 temp 是合法的
    allowed_set = set(effective_whitelist) | cte_names
    # set.difference 保留"引用了但不在白名单"的表名
    # ↑ referenced_tables - allowed_set = 越界的表名
    disallowed = sorted(referenced_tables - allowed_set)
    if disallowed:
        raise ValueError(
            f"SQL 安全拦截：检测到非白名单表 {disallowed}。"
            f"只允许查询 {cls.ALLOWED_TABLES}。"
        )
```

**配套的两个提取方法（`app/core/sql_safety.py:372-424`）**：

```python
@classmethod
def _extract_tables(cls, sql_no_strings_upper: str) -> set[str]:
    """从 SQL 的 FROM / JOIN 子句中提取被引用的表名

    供 validate() 的白名单校验使用。输入必须是已大写化、已移除字符串字面量的
    SQL（防 'SELECT FROM xxx' 这类字面量误命中）。

    支持的形态（详见 TABLE_REFERENCE_PATTERN 注释）：
      - FROM dim_region
      - JOIN fact_order
      - FROM dw.dim_region （schema.table，取 table 部分）
      - FROM `dim_region` （反引号包裹，被忽略）
      - FROM dim_region dr （别名不被提取）

    不支持逗号分隔的多表 FROM（当前 SQL 模板不生成这种形态，遇不到）。

    Args:
        sql_no_strings_upper: 已处理过的 SQL（大写 + 字面量移除）

    Returns:
        表名集合（全部大写）。匹配不到返回空集。
    """
    tables: set[str] = set()
    # re.finditer 找出所有匹配，逐个遍历
    for match in re.finditer(cls.TABLE_REFERENCE_PATTERN, sql_no_strings_upper):
        # name2 存在说明是 schema.table 形态，取真正的表名 name2；否则取 name
        # ↑ 比如 dw.dim_region，name="dw", name2="dim_region"，要的是 dim_region
        table_name = match.group("name2") or match.group("name")
        if table_name:
            tables.add(table_name.upper())  # 统一大写便于比对
    return tables

@classmethod
def _extract_cte_names(cls, sql_no_strings_upper: str) -> set[str]:
    """从 SQL 的 WITH 子句中提取 CTE 名

    供 validate() 的白名单校验使用。CTE 是当前 SQL 自己定义的临时表，
    被 FROM 引用时不应触发"非白名单"拦截。

    匹配形态（详见 CTE_NAME_PATTERN 注释）：
      - WITH temp AS (...)                → {TEMP}
      - WITH a AS (...), b AS (...)       → {A, B}

    Args:
        sql_no_strings_upper: 已处理过的 SQL（大写 + 字面量移除）

    Returns:
        CTE 名集合（全部大写）。无 WITH 子句返回空集。
    """
    names: set[str] = set()
    for match in re.finditer(cls.CTE_NAME_PATTERN, sql_no_strings_upper):
        cte_name = match.group("cte")
        if cte_name:
            names.add(cte_name.upper())
    return names
```

配套的两个正则常量（`app/core/sql_safety.py:179-209`），注释里详细说明了每个分组的作用：

```python
# 表名引用提取正则
# 支持的形态：
#   FROM dim_region               → name="dim_region"
#   JOIN fact_order               → name="fact_order"
#   FROM dw.dim_region            → name="dw", name2="dim_region"（取 name2）
#   FROM `dim_region`             → name="dim_region"（反引号被忽略）
#   FROM dim_region dr            → name="dim_region"（别名 dr 不被提取）
TABLE_REFERENCE_PATTERN: str = (
    r"(?:\bFROM|\bJOIN)\s+`?(?P<name>[a-zA-Z_]\w*)`?"
    r"(?:\s*\.\s*`?(?P<name2>[a-zA-Z_]\w*)`?)?"
)
# 解析：
#   (?:\bFROM|\bJOIN)  → 非捕获组，匹配 FROM 或 JOIN（带词边界）
#   \s+                → 至少一个空白
#   `?                 → 可选的反引号
#   (?P<name>...)      → 命名捕获组 name，匹配表名
#   [a-zA-Z_]\w*       → 标识符规则：字母或下划线开头，后接字母数字下划线
#   (?:\s*\.\s*`?(?P<name2>...)`?)?  → 可选的 .schema 形态，捕获到 name2

# CTE 名抽取正则
# 形态覆盖：
#   WITH temp AS (...)                → 命中 temp
#   WITH a AS (...), b AS (...)       → 命中 a + b
# 关键约束：要求 name 后紧跟 `AS (`（左括号必须紧跟），避免误匹配
# SELECT 子句里的 `col AS alias`（alias 后面不会是左括号）
CTE_NAME_PATTERN: str = (
    r"(?:\bWITH\s+|,\s*)`?(?P<cte>[a-zA-Z_]\w*)`?\s+AS\s*\("
)
# 解析：
#   (?:\bWITH\s+|,\s*)  → 非捕获组：WITH 后跟空白，或者逗号后跟空白
#                         ↑ 这两种形态覆盖 "WITH temp AS (" 和 ", b AS (" 两种
#   `?                  → 可选反引号
#   (?P<cte>...)        → 命名捕获组 cte，匹配 CTE 名
#   \s+AS\s*\(          → 必须紧跟 "AS ("（AS 关键字后跟左括号）
```

为什么 CTE_NAME_PATTERN 要求 `AS (` 紧跟？

- 合法 CTE：`WITH temp AS (SELECT ...)` —— `AS` 后面紧跟 `(`
- 误匹配风险：`SELECT salary AS bonus FROM ...` —— `AS` 后面是 `bonus`，不是 `(`
- 加上 `\s*\(` 约束后，第二种不会被误识别成 CTE

```python
@classmethod
def set_dynamic_allowed_tables(cls, table_names: list[str]) -> None:
    """启动时从 meta_db 加载表名 → 注入到 SQL 校验器

    让"加表"流程零代码改动：在 meta_config.yaml 加表 + 跑 build_meta_knowledge，

    cls._DYNAMIC_ALLOWED_TABLES = {
        t.lower() for t in table_names if t and isinstance(t, str)
    }
    # ↑ 注意转小写：和 _extract_tables 返回的大写集合做并集时要注意大小写统一
    # ↑ ⚠️ 实际这里有个小坑：动态加载的是小写，但 _extract_tables 返回的是大写，
    #      比对时 set 是大小写敏感的。仔细看 validate() 里的代码：
    #        base_whitelist = list(cls._DYNAMIC_ALLOWED_TABLES)  ← 小写
    #        referenced_tables = cls._extract_tables(...)        ← 大写
    #   好在 validate() 调用时 SQL 已经 upper() 过了，白名单也要统一 upper。
    #   这里 set_dynamic 时转 lower 是为了和 ALLOWED_TABLES（小写原文）保持一致，
    #   真正比对前在 validate() 里会再处理。属于"约定：白名单存原文小写，
    #   比对时统一转大写"的风格。

@classmethod
def get_effective_allowed_tables(cls) -> list[str]:
    """返回当前生效的表名白名单（动态优先，否则兜底）"""
    if cls._DYNAMIC_ALLOWED_TABLES:
        return sorted(cls._DYNAMIC_ALLOWED_TABLES)
    return list(cls.ALLOWED_TABLES)
```

### #3 LIMIT 兜底 + truncated 字段

#### 为什么（OOM 风险）

LLM 生成的 SQL 有两类典型失误会导致结果集爆炸：

1. **漏 LIMIT**：LLM 生成 `SELECT * FROM fact_order`（没加 LIMIT），而 `fact_order` 是订单事实表，生产环境动辄千万行
2. **WHERE 条件写错**：本意是"查华东区昨天的订单"，LLM 写成 `WHERE region_id IS NOT NULL`（条件过宽）

无论哪种，原代码都会触发灾难：

```python
# 改动前：dw_mysql_repository.run
result = await self.session.execute(text(sql))
return [dict(row) for row in result.mappings().fetchall()]
#                                   ^^^^^^^^^^ 全量物化
```

`fetchall()` 把整个结果集一次性拉到 Python 内存，再 `[dict(row) for row in ...]` 转成列表。如果结果集是 1000 万行，每行平均 200 字节，就是 **2GB 内存**——单请求就能把进程 OOM 掉。

更危险的场景：**multi-agent 并行**（见 query_service.py 的 `query_multi_agent` 路径）。如果 supervisor 一次拆出 3 个子任务，每个子任务都跑一条漏 LIMIT 的 SQL，3 个 2GB 同时分配，进程必死。

为什么不在 SQL 层加 `SELECT * FROM ({sql}) LIMIT 5000`？因为**会破坏聚合语义**：

```sql
-- LLM 原本生成的 SQL
SELECT region_id, SUM(order_amount) AS gmv
FROM fact_order
GROUP BY region_id
-- 期望：每个 region 一行聚合结果，可能就 30 行

-- 如果外面包一层 LIMIT
SELECT * FROM (
  SELECT region_id, SUM(order_amount) AS gmv
  FROM fact_order
  GROUP BY region_id
) t LIMIT 5000
-- 这层包装对聚合查询无害，因为聚合后只有 30 行
-- 但是！如果原 SQL 本身是 "SELECT * FROM fact_order"（无聚合），
-- 包一层 LIMIT 5000 会悄悄截断到 5000 行，用户看不到完整数据，且不知情
```

更深的问题：包子查询会让原 SQL 里的 CTE、变量、hint 全部失效（MySQL 不支持在子查询里引用外层 CTE）。所以**不能改 SQL，只能在 fetch 层兜底**。

#### 原来是怎么样的

`app/repositories/mysql/dw/dw_mysql_repository.py` 改动前（diff 1729-1739 行）：

#### 怎么做（fetchmany 兜底，不改 SQL 语义）

核心思路：**fetch 层多取一行用来判断是否截断**。

- `fetchmany(N + 1)`：尝试取 N+1 行
- 如果实际拿到 ≤ N 行：结果集完整，`truncated = False`
- 如果实际拿到 N+1 行：结果集肯定超过 N 行，`truncated = True`，截掉多余的那一行

为什么"多取一行"能判断截断？因为：

- 如果实际数据有 5000 行，`fetchmany(5001)` 返回 5000 行（拿不到第 5001 行），`len(rows) == 5000`，不大于 5000，`truncated = False` ✅
- 如果实际数据有 5001 行，`fetchmany(5001)` 返回 5001 行，`len(rows) == 5001 > 5000`，`truncated = True`，然后 `rows[:5000]` 截掉最后那行 ✅
- 如果实际数据有 1000 万行，`fetchmany(5001)` 也只返回 5001 行（数据库游标不会把全部数据拉过来），内存占用恒定 ✅

关键是 `fetchmany` 的语义：**它不会把整个结果集物化到内存**，只是从数据库游标里取出前 N 行。即使结果集有 1 亿行，`fetchmany(5001)` 也只用 5001 行的内存。

#### 代码改动（含完整注释）

**repository 层（`app/repositories/mysql/dw/dw_mysql_repository.py:18-21, 69-83`）**：

```python
# 2026-07-20 新增（#3 LIMIT 兜底）：单次查询返回行数硬上限
# LLM 生成的 SQL 若漏 LIMIT 或 WHERE 写错，可能 SELECT * FROM fact_order
# 把千万行全部物化到内存，单请求就能 OOM 进程。fetchmany 兜底防灾难。
_MAX_RESULT_ROWS = 5000
# ↑ 模块级常量。命名全大写 + 下划线，表示"常量"。
# ↑ 5000 行的依据：前端表格组件渲染超过 5000 行会卡顿；
#   且单行平均 200 字节 × 5000 = 1MB 内存，安全。


class DWMySQLRepository:
    """负责查询数仓真实表结构和字段样例值"""

    # ... 其他方法 ...

    async def run(self, sql: str) -> list[dict]:
        """执行最终 SQL，并把 SQLAlchemy 行对象转换成前端更易消费的字典列表

        2026-07-20（#3 LIMIT 兜底）：用 fetchmany(_MAX_RESULT_ROWS + 1) 取行，
        超过 _MAX_RESULT_ROWS 视为截断（结果集过大）。这一层只防 OOM，
        不主动改 SQL 加 LIMIT（避免破坏聚合语义）。
        返回结果带 truncated 标记由调用方消费（见 run_sql.py）。
        """
        # 注意：返回类型注解还是 list[dict]，但实际返回 (list[dict], bool) 元组
        # ↑ 这是 docstring 和类型注解的小瑕疵（应该改成 tuple[list[dict], bool]），
        #   不影响运行，但后续可清理。

        # text(sql) 把字符串包装成 SQLAlchemy 的 TextClause 对象
        # 即使是原生 SQL 也建议用 text()，能享受 SQLAlchemy 的参数绑定等安全特性
        result = await self.session.execute(text(sql))

        # 多取一行用来判断是否截断
        # fetchmany(n) 返回最多 n 行（如果结果集不足 n 行，返回实际行数）
        # mappings() 让每一行变成 RowMapping 对象，可以用 row["列名"] 取值
        rows = result.mappings().fetchmany(_MAX_RESULT_ROWS + 1)

        # 判断是否截断：如果拿到的行数 > _MAX_RESULT_ROWS，说明结果集肯定还有更多
        truncated = len(rows) > _MAX_RESULT_ROWS

        if truncated:
            # 截掉多取的那一行，只保留前 _MAX_RESULT_ROWS 行
            rows = rows[:_MAX_RESULT_ROWS]
            # ↑ 切片操作：rows[:N] 取前 N 个元素（不包括第 N 个）

        # 把 RowMapping 转成普通 dict，再和 truncated 一起作为元组返回
        # list comprehension（列表推导式）：[dict(row) for row in rows]
        return [dict(row) for row in rows], truncated
```

```python
# 把查询结果推送给前端
# type="result" → 表示这是最终结果
# data=result → 实际的数据内容
# truncated=True → 结果超过 _MAX_RESULT_ROWS 被截断（2026-07-20 #3）
result_event: dict = {"type": "result", "data": result}
if truncated:
    # 截断时在 result 事件上附加 truncated + max_rows 字段
    result_event["truncated"] = True
    result_event["max_rows"] = 5000
    # ↑ 前端拿到 max_rows 后可以显示"仅展示前 5000 行"

    # 同时发一条 warning 让前端 UI 明显提示
    # ↑ 这是"双通道通知"：result 事件带结构化字段（给程序消费），
    #   warning 事件给文案（给用户直接看）。前端两处都会渲染。
    writer({
        "type": "warning",
        "message": "结果集过大，仅展示前 5000 行。请加更精确的筛选条件或聚合维度。",
    })
    logger.warning(
        f"[SQL安全] 结果截断：返回 {len(result)} 行（实际更多）"
    )
writer(result_event)
```

为什么用"双通道"通知（result 事件带字段 + 单独发 warning 事件）？

- **result 事件的结构化字段**（`truncated: true`, `max_rows: 5000`）：供前端做更精细的 UI 处理（比如表格上方显示固定 banner、给"导出 CSV"按钮加 disabled）
- **warning 事件的文案**：保证即使前端没适配 `truncated` 字段，至少 warning 渲染了用户也能看到提示
- **冗余但安全**：前端漏接其中一个，另一个还能兜底

## 改动之间的关联

#1 / #2 / #3 是**纵深防御（defense in depth）**的三个层次，互为补充，任一层失效都不会导致灾难：

```
                        用户问题 → LLM 生成 SQL
                                  │
                                  ▼
              ┌──────────────────────────────────────┐
              │  #2 入口防火墙（SQL 校验）            │
              │  - 危险关键字：DROP/CALL/SET/OUTFILE  │
              │  - 注入特征：UNION SELECT/SLEEP/系统表 │
              │  - 表名白名单：只允许业务表            │
              │  - 剥块注释：防 UNION/**/SELECT 绕过  │
              └──────────────────────────────────────┘
                                  │ 通过
                                  ▼
              ┌──────────────────────────────────────┐
              │  #3 执行兜底（fetchmany）              │
              │  - 单查询最多 5000 行                  │
              │  - truncated 标记                      │
              │  - 防 OOM                              │
              └──────────────────────────────────────┘
                                  │ 执行
                                  ▼
              ┌──────────────────────────────────────┐
              │  #1 错误脱敏（分类映射）               │
              │  - 数据库异常 → 友好文案               │
              │  - 完整异常进服务端 logger             │
              │  - 防 schema 泄露                     │
              └──────────────────────────────────────┘
                                  │
                                  ▼
                              前端展示
```

三层各自独立的失败模式：

| 层 | 失效场景 | 后果 | 其他层兜底 |
|----|----------|------|------------|
| #2 入口防火墙 | LLM 生成的新攻击模式没被规则覆盖（比如新出现的函数） | 危险 SQL 漏到数据库 | #3 不会让攻击者用 `SELECT * FROM huge_table` 拖垮进程；#1 不会让数据库报错时泄露 schema |
| #3 执行兜底 | 上限设得过高（比如改成 100 万行） | 仍然可能 OOM | #2 在 SQL 层先拦掉"没 WHERE 的大表查询"（虽然当前规则没专门拦这个，但白名单已经限定了表） |
| #1 错误脱敏 | 映射表漏掉某种新异常类型 | 给前端兜底文案（不泄露） | 前端 `mapBackendMessage` 再兜一层 |

一个完整的攻击场景演示三层如何配合：

**场景**：攻击者在前端输入"查询所有订单的密码字段"

1. LLM 被诱导生成 `SELECT password FROM dim_customer`（假设 dim_customer 有 password 字段）
2. **#2 入口防火墙**：
   - 关键字检查：没有危险关键字，通过
   - 注入检查：没有 UNION/SLEEP 等，通过
   - 表名白名单：`dim_customer` 在白名单里，通过
   - **结果**：SQL 通过校验（这是设计上的——白名单表允许查任何字段，字段级权限不在 SQL 防火墙范围内）
3. **#3 执行兜底**：执行成功，返回 1000 行（没超 5000），`truncated = False`
4. **#1 错误脱敏**：没异常，不触发

这个场景说明：**SQL 防火墙不能解决所有安全问题**（比如字段级权限控制需要专门的 RBAC 层），但至少能保证：
- 不会 OOM（#3）
- 不会因报错泄露 schema（#1）
- 不会被注入攻击（#2 的 5 类盲区都补上了）

另一个场景演示 #1 和 #2 配合：

**场景**：攻击者输入"查询 fact_orde 表"（故意少一个字母，试探表名）

1. LLM 老老实实生成 `SELECT * FROM fact_orde`
2. **#2 入口防火墙**：
   - 表名白名单校验：`fact_orde` 不在白名单（白名单是 `fact_order`），拦截
   - 错误信息：`SQL 安全拦截：检测到非白名单表 ['FACT_ORDE']`
3. **#1 错误脱敏**：安全拦截的 ValueError 走的是内层 except，文案是"SQL 安全拦截：检测到非白名单表 ..."，这个文案已经不泄露任何额外信息（攻击者只知道"这个表名不被允许"，不知道正确表名是 `fact_order`）

这个场景里 #2 拦住了，#1 不需要触发。但如果 #2 漏了（比如未来加了一张新表，白名单更新慢了一拍），SQL 进了数据库触发 `Table 'dw.fact_orde' doesn't exist`，#1 会把异常脱敏成"查询的表不存在，请确认表名或换一种问法"——依然不泄露正确表名。

## 附录：常见疑问

**Q1：为什么 `FORBIDDEN_KEYWORDS` 里既加 `OUTFILE` 又在 `INJECTION_PATTERNS` 里加 `INTO OUTFILE`？是不是冗余？**

A：是冗余，但是**故意的双重防护**。关键字层只要 SQL 里出现 `OUTFILE` 这个词就拦（粗粒度，覆盖广）；注入层用 `\bINTO\s+(OUTFILE|DUMPFILE)\b` 精确匹配"作为子句"的形态（细粒度，便于错误归类）。安全功能遵循"一层漏了另一层兜底"原则，宁可冗余不可漏拦。

**Q2：剥块注释后，原 `INJECTION_PATTERNS` 里的 `r"/\*.*\*/"` 不就匹配不到了吗？是不是该删掉？**

A：理论上是的，但保留它有一个好处——**如果未来有人改 `validate()` 把剥注释那行删了或改错，这条规则还能兜底**。属于"防御性保留"。另外这条原正则没加 `re.DOTALL`，跨行注释本来就匹配不到，留着也无害。

**Q3：`_DYNAMIC_ALLOWED_TABLES` 在 `set_dynamic_allowed_tables` 里转小写，但 `_extract_tables` 返回大写，比对时不会有大小写不一致的 bug 吗？**

A：仔细看 `validate()` 的代码（`sql_safety.py:345-360`）：
```python
if allowed_tables is not None:
    base_whitelist = [t.upper() for t in allowed_tables]  # ← 显式传的转大写
elif cls._DYNAMIC_ALLOWED_TABLES:
    base_whitelist = list(cls._DYNAMIC_ALLOWED_TABLES)    # ← 动态的是小写！
else:
    base_whitelist = [t.upper() for t in cls.ALLOWED_TABLES]  # ← 兜底转大写
```
中间分支 `base_whitelist = list(cls._DYNAMIC_ALLOWED_TABLES)` 没有转大写，而 `_extract_tables` 返回的是大写集合。所以**动态白名单这条路径上确实存在大小写不一致的潜在 bug**——如果动态加载的表名是大写形态（如 `DIM_REGION`），存进 `_DYNAMIC_ALLOWED_TABLES` 时被 `t.lower()` 转成了小写（`dim_region`），比对时 `DIM_REGION`（来自 _extract_tables）和 `dim_region`（来自动态白名单）会因为大小写不同而被判为"非白名单"，误杀合法查询。

这是一个值得后续修的小瑕疵（要么 `set_dynamic` 时不转小写，要么比对前统一 `upper()`）。当前测试能过是因为测试里用的是 `dim_region`（小写）→ 转大写 `DIM_REGION` → 比对 `DIM_REGION`，刚好走的是 `else` 兜底分支（没走动态分支）。属于"测试覆盖不到的隐藏路径"，CLAUDE.md 2026-07-20 那条"既有测试 monkeypatch 失效是信号"的笔记可以参考。

**Q4：`run_sql.py` 内层 except（SQL 安全校验失败）为什么不脱敏？**

A：因为安全拦截的 ValueError 是我们自己 `raise` 的，文案是我们设计的（如"SQL 安全拦截：检测到危险关键字 'DROP'"），不包含数据库内部信息，本身就是"对用户友好且不泄露"的。再脱敏反而会让错误信息变模糊。**只对"数据库回的异常"脱敏，不对"我们自己 raise 的错误"脱敏**——这是脱敏的边界。

**Q5：5000 行这个上限是怎么定的？**

A：三个考量：
- **前端渲染**：表格组件渲染超过 5000 行会明显卡顿（浏览器 DOM 节点过多）
- **内存占用**：单行平均 200 字节 × 5000 = 1MB，单请求内存可控
- **SSE 传输**：5000 行的 JSON 大约几百 KB 到几 MB，HTTP 流式传输不会卡

如果未来业务真的需要看更多行，正确做法是引导用户用 `GROUP BY` / 聚合 / 分页（`LIMIT 5000 OFFSET ...`），而不是无上限地查全表。

**Q6：multi-agent 并行时，每个子任务都能拿 5000 行吗？3 个子任务就是 15000 行？**

A：是的。每个 `run_sql` 节点独立调 `dw_mysql_repository.run`，独立 `fetchmany(5001)`。3 个子任务并行就是 3 × 5000 = 15000 行的内存峰值。这是可接受的（3MB 左右）。但如果 supervisor 拆出 10+ 个子任务，就需要在编排层再加一个"全局结果集预算"的限制（比如所有子任务的 rows 总和不超过 20000），这是后续 #11 多 agent 资源治理的话题。
